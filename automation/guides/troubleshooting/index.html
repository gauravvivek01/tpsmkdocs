    1. Troubleshooting

    Due to the number of tests running on a nightly build, it may be necessary to rerun test cases using the npm run testcase <category> <reference> 1 <cleanup> command to check the terminal output

    1.1 Test not Passing

    "Test not Passing" is a case where the nodejs framework was able to copy the configuration files to the selected devices and run the test case script using the sshCommand class, but the test case is consistently failing.

    Checks

    • Did the test report the failure? If not proceed to 1.3 Test not Reporting
    • If the receiver is rebooted, and the test is repeated, does it pass? If so it might be a problem with a previous test not terminating correctly

    1.1.1 Test Configuration Errors

    If the logs for the test case list a test configuration error it means the connection failed to launch for a known reasion. An example error message is Test Configuration Error: RX is set to private and we are trying to connect to a shared session. The test configuration error message is generated by the check_error_codes function in connection_test_common.sh. There are 2 known problems that might cause this.

    1. If we are running a dual test case, the test case script should have a variable near the beginning called INDEX. This variable specifies which target we are choosing to connect to from the JSON config file (INDEX=0 means target 1, INDEX=1 means target 2 etc.). If the second test case script is using the wrong index, the test case might try t launch 2 identical sessions to the same TX which might cause a test configuration error. In most cases, test_case_a.sh should have INDEX=0, test_case_b.sh should have INDEX=1 and so on.
    2. This can also happen if a connection is left open from a previous test, or hasn't been terminated correctly. The cleanup scripts should take care of breaking the connections after the tests are finished, unless the test is run in debug mode with the cleanup option set to false.

    1.2 Test not Running

    Checks

    • Did the script and config copy over to the device?
    • Could the test find the requested devices from the soak?

    1.2.1 Failed to find Device

    Failed to find Device means the AutomationFramework was unable to find a device for a particular test case. The Framework will report to the database what device it couldn't retreive.

    This can happen due to a problem with the labtestrig.json file, or simply due to there being no available matching device.

    In rare cases this can be caused by the framework not resetting the soak after it completes a test. See the following code snippet for resetting the soak file. 

    soak.updateDeviceListInUsetatus(0, false, true);

    1.2.1.1 Failed to Find TX

    If the device that couldn't be found was a TX, ask yourself if the test requires 2 different TX's or can the same result be obtained using 1 TX. To change the test to use one TX, go to the test case target config files and change "_exclusive" to "Off". If the test case does need 2 seperate TX's you might need to look into setting up a new TX in the lab, or changing the device type of the second TX to a similar but different device type.

    1.2.1.2 Failed to Find RX

    Since dual test cases require 2 seperate RXs, there's not much we can do except change the device type to one that's available, or source another RX from elsewhere. Check the labtestrig.json file isup to date with the latest version.

    1.2.2 Failed to transfer files

    Usually this is a problem with the device itself. You might see a message saying PutFile: failed to connect to... followed by the IP Address of the device. If this message shows up, it's usually after a test which causes a device to reboot. If an unplanned leave test reboots a receiver and finishes quickly after, the framework might move onto the next test case faster than it takes the first device to come back online. When this happens the script and the config can't copy over to the device, resulting in a failure. This failure won't report to the server as it's not expected. There are 2 ways to combat this...

    Info

    We can avoid this by changing the order of test cases in unplanned_leave or unmanage in such a way that a test case requiring a certain device type doesn't follow another test using the same device type. If we need to follow a test with another of the same device type, we should add a sleep to the previous test case script, to delay long enough for the device to reboot.

    e.g A test case using an SE RX shouldn't follow another test using an SE RX.

    1.3 Test not Reporting

    • Does the test case script have a sendResults function?
    • Check the docker machine logs with the command docker-compose logs --tail=500 testcaserest in the AutomationNodejs directory. Was the POST request seen by the server?
    • Are the "_testCaseResultsServer" values in the tstconfig files matching the IP Address of the docker machine?
    • SSH into the receiver and try to parse the JSON config file using jq . <json config file name>

    1.3.1 Test Configuration Invalid JSON

    Occasionally if we add new values to the JSON config file, the syntax might get changed and our jq application can't parse the text. Usually this is self-explanatory from the terminal output which usually reads parse error: Expected seperator between values at.... This failure cannot report to the server as it cannot read the results server IP from the JSON config file, nor any of the other relevant information.

    Check the test_case__a.json on the first receiver using jq . test_case_<ref>_a.json. The test configuration files are generated through testCase.js and testAssetConfiguration.js in a function called generateConnectionTestFile
