    1.2 Terms and Definitions

    Term Definition
    Boxilla Blackbox's Appliance Management Software
    Decoder See Receiver
    Encoder See Transmitter
    Multiviewer A multiviewer is similar to a video wall. But while video walls require a separate TV display for each video source, multiviewers only require one display surface. Multiviewers are able to showcase full, split, quad, and custom configurations on the same pixel surface.
    PCoIP PC over IP (PCoIP) is a remote display protocol that Teradici developed for delivering remote desktops and applications to endpoints. PCoIP works by rendering client desktops on a network or cloud server. The desktop's pixels are then compressed, encrypted and then transmitted to the client device. The client then decrypts and renders the desktop view for the user.
    RFX RemoteFX Codec Extension developed by Microsoft which uses a lossy image codec to encode screen images with efficient and effective compression. This is the lossy scheme used in Blackbox products.
    Receiver A Blackbox appliance that receives and displays a video stream. A receiver is also referred to as a client or decoder.
    Source The term Source is commonly used to describe the PC connected to a transmitter appliance.
    Shared Mode A generic term used in Blackbox to describe schemes that allow more than one receiver view the video source from a transmitter. See Municast and Multicast.
    Transmitter A Blackbox appliance that captures and transmits a video from a source PC. A transmitter is also referred to as a server or encoder.
    Video Source A source of input for a Transmitter, normally the DVI or Display Port output of a PC.
    Video Wall A video wall is a special multi-monitor setup that consists of multiple computer monitors, video projectors, or television sets tiled together contiguously or overlapped in order to form one large screen. Typical display technologies include LCD panels, Direct View LED arrays, blended projection screens, Laser Phosphor Displays, and rear projection cubes.

    1.4 Role of the OSD in the overall SDKVM Architecture

    The OSD provides the main user interface for the SDKVM. The SDKVM is a multi-user system and each user can provision and manage a range of connections and workspaces.

    The OSD communicates with the workspaceserver through a REST API and MQTT messages to launch workspaces and connections. The status of these connections and workspaces are returned to the SDKVM either as MQTT messages or through polling the workspaceserver's REST API.

    These specification focuses on the fully managed version of the product.

    1.5 Authentication of User Login

    In a SDKVM system managed by Boxilla a user login must be authenticated by Boxilla. The process described below only applies to a managed system.

    From the above diagram,

    • The OSD checks if it is in managed mode.
      • If the SDKVM is not manged the OSD uses the existing mechanism to verify a user against the credentials in user.json
      • If the SDKVM is managed then the OSD will follow the Boxilla Authentication process outlined below:

    1.5.1 Boxilla Authentication Process for Managed Systems

    For this process the OSD will use a utility called the STATS_CLI which is located at /usr/local/bin/stats_cli. This utility allows various commands to be sent to the Boxilla Management System which manages the SDKVM. If a user is successfully authenticated then the SDKVM sysytem will update the user details locally by writing/updating the file systems in the JSON files. The JSON files include users.json, connections.json, and workspaces JSON files in the user directory structure.

    1.5.1.1 Login Process Flow for Managed Systems

    • The User enters the login credentials on the OSD login screen.
    • The OSD will kill any existing instances of "/usr/local/bin/stats_cli". This is required because any existing instances could be as a result of previously failed login attempts/

    killall /usr/local/bin/stats_cli
    - OSD then passes the user credentials (username and password) to the Boxilla management system using the STATS_CLI application program which is available in /usr/local/bin/stats_restapi_cli. 

    /usr/local/bin/stats_cli –login username password
    • The OSD should then start a 30 second timer (this should be a configurable setting in /home/snuc/SDKVM/users/system_level_preferences/system_preference.json and should default to 30 seconds) so that it can timeout if no response is received from the Boxilla Management System within that period of time.
    • The OSD will display a progress or busy indicator to indicate that the authentication process has started.
    • The STATS_CLI program launched by the OSD will return an authentication message back to OSD through MQTT interface (see details below).
    • If the authentication process ends in success then the OSD will:
      • Stop the 30 second timer.
      • Disable the progress or busy indicator.
      • Move to the OSD main screen.
      • Write a syslog entry indicating that the defined user has successfully logged in.
    • If the authentication process ends in failure then the OSD will:
      • Stop the 30 second timer
      • Disable the progress or busy indicator.
      • Display an error message the user on the OSD. "Authentication Failed. Please try again."
      • Write a syslog entry indicating that the reason for the failure, the user name and any additional details returned in the MQTT message.
    • If the authentication process ends in failure due to a time out then the OSD will:
      • Disable the progress or busy indicator.
      • Display an error message to the user on the OSD. "Authentication Failed due to timeout. Please try again."
      • Write a syslog entry indicating that the authentication failed due to a lack of response from Boxilla.
      • Send an alert indicating that a login timeout has occurred: 
         /usr/local/bin/stats_cli –timeout username
    1.5.1.1.1 MQTT messages required for Authentication

    The OSD should subscribe to the following MQTT endpoints:

    MQTT Topic:  topic  blackbox/sdkvm/deskvue/status/osd_sdkvm/webservice/authentication

    This endpoint will return a message indicating success or failure. The login_status provides the primary indication of success or failure.

    {"login_type" : "local", "login_status" : "success", "user_name" : " ", "details" : "Boxilla return details" }

    MQTT Topic:  topic  blackbox/sdkvm/deskvue/status/osd_sdkvm/webservice/error

    This endpoint will provide error details for a malformed request to the stats_cli.

    {"error_details" : "unknown command", "command_requested" : "" , "parameters_passed" : ""
}

    1.5.1.2 Logout Process Flow for Managed Systems

    • A user selects the logout option on the OSD.
    • The OSD will send a logout notification to Boxilla using the STATS_CLI application program which is available in /usr/local/bin/stats_restapi_cli. 
    /usr/local/bin/stats_cli –logout username
    • The OSD should then start a 30 second timer (this should be a configurable setting in /home/snuc/SDKVM/users/system_level_preferences/system_preference.json and should default to 30 seconds) so that it can timeout if no response is received from the Boxilla Management System within that period of time.
    • The OSD will display a progress or busy indicator to indicate that the logout process has started.
    • The STATS_CLI program launched by the OSD will return an authentication message back to OSD through MQTT interface.
    • If the logout process ends in success then the OSD will:
      • Stop the 30 second timer.
      • Disable the progress or busy indicator.
      • Move to the OSD login screen.
      • Write a syslog entry indicating that the defined user has successfully logged out.
    • If the logout process ends in failure then the OSD will:
      • Stop the 30 second timer
      • Disable the progress or busy indicator.
      • Display an error message the user on the OSD. "Logout Failed. Please try again."
      • Write a syslog entry indicating that the reason for the failure, the user name and any additional details returned in the MQTT message.
    • If the logout process ends in failure due to a time out then the OSD will:
      • Disable the progress or busy indicator.
      • Display an error message to the user on the OSD. "Logout Failed due to timeout. Please try again."
      • Write a syslog entry indicating that the authentication failed due to a lack of response from Boxilla.
      • Move to the OSD login screen. ( I assume we would not want the failure to communicate with Boxilla from preventing us from logging out.)
      • Send an alert indicating that a login timeout has occurred: 
        /usr/local/bin/stats_cli –logout –timeout username
    1.5.1.2.1 MQTT messages required for Logout

    The OSD should subscribe to the following MQTT endpoints:

    MQTT Topic:  topic  blackbox/sdkvm/deskvue/status/osd_sdkvm/webservice/logout

    This endpoint will return a message on success or failure. This message can be used as confirmation that the command was processed.

    {"logout" : "local",  "details" : "Boxilla return details" }

    This endpoint will provide error details for a malformed request to the stats_cli.

    MQTT Topic:  topic  blackbox/sdkvm/deskvue/status/osd_sdkvm/webservice/error 

    {"error_details" : "unknown command", "command_requested" : "" , "parameters_passed" : ""
}

    1.6 Changes to OSD Tab Design for a Fully Managed System

    Note: These only apply to fully managed systems.

    1.6.1 Networking

    • In a fully managed system the network tab should be read only for all user types. All network configuration is done through Boxilla, so the Network tab simply shows the assigned networking details in a read-only manner.
    • This applies to all users. It is not possible for any user (including admin types) edit or change network settings on a managed system.

    1.6.2 Display Settings

    • In a fully managed system Boxilla will not be able to manage the display monitor settings, so an admin user will still have the ability to edit and change monitor settings, but local and power users will have no access to these settings.

    1.6.3 Global Settings

    • In a fully managed system Boxilla will not be able to manage the global settings, so an admin user will still have the ability to edit and change all of the settings on this page, but local and power users will have no access to the volume setting only.

    1.6.4 Templates

    • This tab should be hidden on a fully managed system because all templates are set and created on the Boxilla management system.

    1.6.5 Connections

    • In a fully managed system users (admin, local and power) cannot modify, edit or add connections; they can only launch them to check that they are functional.
    • The options below outlined in black should be removed for all users in a fully managed system.
    • It is not possible for any user (including admin types) edit or change connection settings on a managed system.
    • The option to add new connections should be removed for all user types in managed mode.
    • The connections screen should be visible admin and power users, but it should be hidden for local users.

    1.6.6 Workspaces

    • In a fully managed system users (admin, local and power) cannot modify, edit or add workspaces; they can only launch them, so the options outlined in black should be removed.
    • The option to add new workspaces should be removed for all user types in managed mode.

    1.6.7 Users

    • In a fully managed system users are centrally managed by Boxilla. It should not be possible to edit or change any user details so this page should be read only.
    • This applies to all users. It is not possible for any user (including admin types) edit or change user settings on a managed system.
    • For local and power users the users page should be hidden.

    1.6.8 System Settings

    • In a fully managed system the Admin user should only have access to the items enclosed in the black frame (Factory Restore, System Diagnostics, OSD Operations), so the other options should be removed or hidden.
    • A power user and a local user should only have access to the Systme information (Software version) panel.

    1.11 OSD changes for Boxilla based Upgrade

    Support for progress status and exit codes have been added in Deskvue software upgrade. These changes are require integration with OSD to provide the user with correct progress/status information, meaningful failure reasons in the event of a failure, and successful messages in the case when the upgrade proceeds without error.

    1.11.1 Exit codes:

    Error Identifier Error Number Error Description
    UPDATE_SUCCESS 0 Firmware upgrade successful
    ERR_ARGUMENT 1 Error in passing arguments to utility
    ERR_NOMEM 2 Not enough memory in unit or error while any memory related operation.
    ERR_BACKUP 3 Error while taking backup of original firmware snapshot
    ERR_RESTORE 4 Error while restoring original firmware snapshot
    ERR_HEADER 5 Error while parsing image metadata ERR_SEGMENT
    ERR_CHECKSUM 7 Error while matching checksum after update
    ERR_UPDATE 8 Error while placing components in firmware partitions.
    ERR_SAME_VERSION 9 All the components in firmware image are duplicate.
    ERR_ENCRYPT 10 Failed to decrypt self-extracting package
    ERR_SIG_VERIFY 11 Failed to verify the signature of firmware package
    ERR_INTERNAL 12 Internal error while updating the package

    1.11.2 Progress/status bar

    The current status of an upgrade can be read from the file "/home/snuc/SDKVM/info/upgrade_progress.txt"

    The file has the following format:

    PROGRESS:<INT>
EXIT:<0>

    The integer number in the PROGRESS tag signifies the percentage of upgrade progress. The exit code signifies the result of upgrade. A non-zero exit code any time during the upgrade signifies the upgrade process has failed. A zero exit code with 100 progress value signifies successful upgrade.

    The OSD should read the progress value and update a progress bar of the OSD. Once the progress has reached 100 then it should post a message indicating either pass or failure. If it is a failure simply provide the message : "Upgrade Failed: Error Code."

    These messages should appear for both a local upgrade or Boxilla upgrade.

    If the upgrade is a Boxilla upgrade then the OSD should present a dialog indicating that an upgrade is in progress and show the progress of the update.

    1.11.3 MQTT messages required for Boxilla Upgrade Notification

    The OSD should subscribe to the following MQTT endpoints:

    MQTT Topic:  topic  blackbox/sdkvm/deskvue/status/osd_sdkvm/upgrade

    This endpoint will return a message at the start and end of a remote upgrade process

    {"status" : "<started/finished>",  "progress" : <0 - 100>  , "exit_code" : <0/n> }

    This facility can be tested with the following command.

    mosquitto_pub -h 127.0.0.1 -t "blackbox/sdkvm/deskvue/status/osd_sdkvm/upgrade" -m "{\"status\":\"started\",\"progress\": 50 , \"exit_code\": 0       }"

    This can be verified with mosquitto_sub:

    mosquitto_sub -h 127.0.0.1 -v -t blackbox/sdkvm/deskvue/status/osd_sdkvm/upgrade

    The actual final status and progress should be read from the files detailed above.

    Questions about this can be added to https://bboxjira.atlassian.net/browse/BUG-5737

    1.12 OSD changes for handling Admin Users

    In version 1.0.0 of the product admin users were able to access all connections and workspaces on the system. In version 1.1.0 of the product the SDKVM can be fully managed by Boxilla. Connections and workspaces are allocated by Boxilla to users, so an admin user cannot access all connections and users by default. They can only access connections allocated to them by Boxilla. This means that the admin users will have populated access control lists in the user.json file. For unmanaged systems, the behaviour remains as is currently implemented. An admin user can access all connections and workspaces on the system.

    1.13 Check for Maximum Resolution Limits when Changing Display Resolutions

    Reference: https://bboxjira.atlassian.net/browse/BUG-5379?filter=10855

    If a user is attempting to set a resolution that causes the max canvass size of 16,384 x 16,384 to be exceeded then the OSD should:

    • Display an error message to the user indicating: “Maximum canvass size of 16,384 x 16,384 across all displays will be exceeded with this change. Change rejected.”
    • The OSD should not change the resolutions of the attached monitors.

    this last point is under consideration, do not implement for now

    • If a new monitor is inserted via a hot-plug then a “safe” resolution should be selected for this monitor to stay within canvas size (the current monitor management already deals with this scenario) and the OSD should display a message to the user indicating that this was done. The message should be: “Resolution nXn selected to stay within maximum canvass constraints on display “1/2/3 or 4” instead of preferred resolution of mXm”

    1.14 Handling Workspaces with Monitors that do not exist

    Reference: https://bboxjira.atlassian.net/browse/BUG-5771

    • If a workspace contains a monitor that is unavailable that monitor is currently labelled Null, ideally the OSD should identify this as Display not attached.
    • When editing a workspace in the OSD it shows the Detected Displays in the order that the workspace was created not how they are currently positioned. The Detect Displays area should reflect the current positioning of the monitors to prevent confusion to the user.

    1.15 Delete Monitor Configuration Option on the System Settings Screen

    Reference: https://bboxjira.atlassian.net/browse/BUG-5785

    The OSD should have an option to delete Monitor Settings. This should be available on the System Settings page under "Restore to Factory Defaults". On;y an Admin user should have access to this option. When the button is clicked it should present a confirmation dialog to the user with the following message:

    • This action will reset the monitor positioning and resolutions to default. Are you sure you want to continue?

    If the user clicks "Continue" then the OSD should call:

    monitor-management.py script called --reset
    This will remove the existing Monitor Configuration causing a new one to be generated.

    1.16 Changing Display Monitor Settings requires a Relaunch of any Active Workspace

    If we are in an active connection and we change monitor arrangement of resolution then we need to relaunch any current workspace.

    The workspace can be relaunched by calling:

    /home/snuc/KeyBindings/Scripts/relaunch_workspace.sh

    1.17 Deleting a Connection that is in use by Existing Workspaces

    This logic only applies to Unmanaged systems. Boxilla will manage the case for managed systems.

    If an admin user is deleting a connection, the system should:

    • Check if connection is used in any workspaces.
    • If it is, OSD should inform the user that the connection is used in a number of workspaces.
    • It should provide an option to display the associated Workspaces.
    • It should offer the user the option to delete the connection from all workspaces it is used and then to delete the connection, or to cancel operation.
    • Where a connection removed from a Workspace - the tile is left “blank” i.e. no connection in that workspace.

    1.18 Using UIDs for Templates instead of Names

    In early versions of the SDKVM we used Template Names in the workspace definition. This will be problematic when we move to user defined templates.

    It is proposed that we move to UIDs. This has implications for the existing SDKVMs that are using names in their workspace definitions. To deal with this change we propose moving the workspace_data_model_version_number to "2".

    All new workspaces should use UIDs as shown in the table below. If the OSD encounters an older version of a workspace then it will need to translate the name to a UID.

    Name
    Full-screen tile on 1-display 00000001
    Quad-tiles on 1-display 00000002
    Four tiles - PIP Right 00000003
    Quad-Tiles - PIP left 00000004
    Eight Tiles on 1-display 00000005
    Seven Tiles on 2-displays 00000006
    One Tile Across all Displays 00000007
