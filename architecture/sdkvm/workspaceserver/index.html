    1. Workspace Server

    1.1 Overview

    1.2 Connection and Workspace APIs

    1.2.1 Connection Check

    The OSD Connection screen allows an Administrative user create connections in an unmanaged system. A user can check a connection by selecting the 'check' button.

    It is proposed that this end-point will also be used for Control Bridge.

    The check connection function will launch a connection by sending a POST with a connection object to:

    127.0.0.1:5000/connection

    see data model for the format of the json formatted connection object.

    1.2.2 Connection Check Status

    The status for a connection can be retrieved via a GET Request from the end point:

    127.0.0.1:5000/connection
    It will return the status for the connection as active or inactive. 

    {
    "query": "inactive"
}

    1.2.3 Workspace Launch

    A workspace can be launched by posting a workspace json object to 

    127.0.0.1:5000/workspace

    see data model for the format of the workspace object.

    1.2.4 Workspace Termination

    A user can terminate a workspace by selecting the 'terminate' button.

    The workspace terminate function will terminate the active connection by sending a DELETE to: Only one workspace is active at anyone time.

    127.0.0.1:5000/workspace

    1.2.5 Workspace Status

    The status for a workspace can be retrieved via a GET Request from the end point:

    127.0.0.1:5000/workspace
    It will return the status for the workspace as active or inactive. The workspace is active if any one connection within the workspace is still active.

    {
    "query": "inactive"
}

    1.3 Generate SDKVM User Configuration from Boxilla Data

    To generate the User Configuration and Workspace and Connection directory structures and files run the following command:

     ~/workspaceserver/process_json_data_file.py -d <path to data file>
    for example:

     ~/workspaceserver/process_json_data_file.py -d ~/SDKVM/boxilla/single_user.json

 will produce the following in ~/SDKVM/users
├── 1
│   ├── connections
│   │   └── 1644.json
│   └── workspaces
│       └── 1.json
├── test_users.json

Where 1 is the user ID
and test_users.json is the users,json file (temporarily renamed to test_users.json while we develop the feature. Once we are happy with the functionality we will set this name to users.json)
    Note. The file should be invoked from ~/workspaceserver

    1.6 Running the Worskpace Server as a Service

    • Flask as a Systemd Service
    • without venv
    • As a user service

    1.6.1 Sample Services File

    [Unit]
Description=Workspace Server for Emerald Deskvue
Wants=mosquitto.service
After=network.target mosquitto.service
PartOf=graphical-session.target

[Service]

WorkingDirectory=/home/snuc/workspaceserver
ExecStart=/home/snuc/workspaceserver/app.py 
Restart=on-failure
PermissionsStartOnly=true
TimeoutSec=600
RuntimeDirectoryMode=755


[Install]

WantedBy=graphical-session.target

    1.6.2 Installation

    • mkdir -p /home/snuc/.config/systemd/user/
    • cp workspaceserver.service /home/snuc/.config/systemd/user/
    • systemctl --user enable workspaceserver

    1.6.3 Depenencies

    To generate a requirements.txt file run the following commands:

    • pip3 install pipreqs
    • python3 -m pipreqs.pipreqs .

    1.6.4 Starting Stopping

    • systemctl --user start workspaceserver
    • systemctl --user stop workspaceserver
    • systemctl --user status workspaceserver

    1.6.5 After Changing

    • systemctl --user daemon-reload

    1.6.6 Inspecting Logs

    • journalctl --user -u workspaceserver

    1.6.7 Service File Location

    /etc/systemd/system/workspaceserver.service

    1.6.8 Starting and Stopping

    • sudo systemctl daemon-reload - will reload the service
    • sudo systemctl start workspaceserver
    • sudo systemctl stop workspaceserver
    • sudo systemctl restart workspaceserver
    • sudo systemctl status workspaceserver
