    1.2.1.2 Audio

    To enable audio support for the KVM client

    /sound

    To enable analog audio the option is:

    /analog-audio

    (the option /sound should not be used in this case; though I can add a check with /analog-audio to disable the /sound option in this case).

    The script pulse_volume_control.sh needs to be placed/copied to /usr/bin/blackbox/ and the executable option set via chmod +x.

    The KVM can support sourcing audio from two audio sources on the TX. The analog audio is audio received on TX's audio port (3.5 mm). The HID audio is the audio the TX receives on the HID from the target PC. When making a connection we must specify the source on the TX by passing the required argument.

    1.2.2 Left Right Audio Channels

    It should probably mention that the data comes in as right-left samples from the TX (though might want to double check this). The default channel order/mapping for PCM is typically left-right; however, when testing this with the TX box the left and right channels were reversed; ie, something that should play on the right speaker would play on the left speaker and vis-versa. If I connect speakers to the target machine directly via the audio jack the channels play correctly on the left and right speakers.

    In the code I'm working on, I've reversed the channel order for the pulse audio stream so it outputs on the expected channel.

    1.2.3 Audio Formats for Shared Connections

    The audio data sent is just raw pcm data. Unlike the Video the Audio App engines on the appliance receivers does not parse/modify the data it receives. The audio packets are a fixed size of 512 bytes. Data going to/from the audio deserialiser/serialiser in the receiver and transmitter is in units of 64 bytes. To keep everything easily divisible the audio data will be structured as 512 bytes in each packet. Since the Audio data is written to DRAM, the header can be read for each received packet if it has not been written over by a new packet.

    The data is signed 48 Khz PCM 16 bit stereo so a sample is 32 bits wide.

    AVAE provides an option to turn on Audio header which provide a 64 byte header with timestamps and frame marker. A 512 byte packet is divided in 64 byte header and 448 byte payload (PCM 16 bit stereo) if the audio header functionality is turned ON. At present this functionlity is disabled, but we may decide to turn this on at some future point -that of course would require SDKVM changes.

    1.2.3.1 Audio ingress (MIC Path)

    In bi-directional audio, the audio data stream can be transmitted from the EMD transmitter to the SDKVM and vice versa. For simplicity's sake, we refer to the audio path from the SDKVM to the EMD transmitter as the 'MIC path'. The audio format remains consistent, whether it's for audio egress or audio ingress.

    • The EMD transmitter receives the audio data on port 9000
    • The audio packets have a fixed size of 512 bytes.
    • To maintain ease of handling, the audio data is structured in 512-byte packets.
    • It is signed 48 kHz PCM (Pulse Code Modulation) with a 16-bit stereo format, meaning each sample is 32 bits wide.

    1.2.3.2 Silence Supression

    The Egress side also support silence supression. After a certain silence threshold value egress side doesn't send the audio sample until it sees the genuine audio data. On the ingress side if the audio buffer is empty, it plays a silence sample on repeat to avoid distorted audio. Deskvue needs to implement a similar stratergy to play out silence.

    The audio buffer would simply drain and it will auto mute. The egress side still seems to stream out audio data even when audio channel is muted, ie, zero level samples (muted).

    1.2.4 Scripts for Confirming Audio

    1.2.4.1 Ingress

    source /opt/blackbox/test/debug_utilities.sh
#set -x

AAE_INGRESS_START_BUFFER_ADDR=0x6fff0000
#AAE_INGRESS_START_BUFFER_ADDR=0x3fff0000 for 2K SE/PE
#AAE_INGRESS_START_BUFFER_ADDR=0x1fff0000 for ZeroU
#AAE_INGRESS_START_BUFFER_ADDR=0x1fff0000 for Corrib

for i in $(seq 1 1 128)
do
        AUDIO_DATA=$(devmem $AAE_INGRESS_START_BUFFER_ADDR)
                UpperAndLower16 $AUDIO_DATA
                ConvertPCMToDecimal $UPPER_BYTE
                LEFT=$PCM_VALUE
                ConvertPCMToDecimal $LOWER_BYTE
                RIGHT=$PCM_VALUE
                printf "Left Channel: %04X %d %d Right Channel: %04X %d %d \n" $((UPPER_BYTE)) $(($UPPER_BYTE))  $((LEFT)) $((LOWER_BYTE)) $(($LOWER_BYTE)) $((RIGHT))
        AAE_INGRESS_START_BUFFER_ADDR=$(($AAE_INGRESS_START_BUFFER_ADDR+4))
done

    The functions UpperAndLower16 and ConvertPCMToDecimal can be found in Buildroot/buildroot-2017.02.4/board/blackbox/common/common/rootfs_overlay/opt/blackbox/test/debug_utilities.sh

    1.2.5 Extended Desktop Operation

    1.2.5.1 kvm connection (dual head) example:

    /window-sizing:1920x1080,1280x1024 /window-position:0x0,1920x0 /dual

    Specifies two comma delineated windows via smart sizing and two comma delineated positions for the respective windows. The "dual" switch is also required to enable extended desktop.

    1.2.5.2 kvm connection (single head) example:

    /window-sizing:1920x1080 /window-position:0x0

    Specifies one window via smart sizing and the position

    There wouldn’t be an extended desktop for a kvm connection (TX) since it is configured to present a single or dual set of monitors to windows at configured resolutions on the TX.

    1.2.5.3 RDP connection (extended desktop) example:

    /desktop-resolution: 3840x2160,3840x2160  /window-sizing:3840x2160,3840x2160 /window-position:0x0,3840x0

    Specifies the RDP connection should request a desktop width and height of 3840x1920 specified by /w and /h. The window size would typically match the /w and /h but that’s not necessary; it could be different, ie /window-sizing:1920x540 which would show the extended desktop scaled down to a window with the same ratio.

    1.2.6 Swap Heads or Flip

    /display-swap

    which flips the primary and secondary display to their respective windows.

    1.3.9 Range Control for Absolute Co-Ordinates for Mouse

    /abs-coords:(<x>,<y>)(<x>,<y>),[(<x>,<y>)(<x>,<y>)] - provides absolute coordinate mapping for primary and secondary monitors. These are done in coordinate pairs (x,y - upper left)(x,y - bottom right)

    Example of new params for a target running the hiseclabs driver on a dual head tx would be:

/abs-coords:(0,0)(16383,32767),(16384,0)(32767,32767) /show-local-mouse

    1.3.10 Show local Mouse Cursor

    /show-local-mouse - always show local mouse cursor

    1.3.11 Disable Hotbar/Overlay

    To remove the hotbar/overlay from the client

    -hotbar

    1.3.12 Hotbar Content Control

    /hotbar:feature1,feature2,feature3,...

    where the features are comma delimited and optional. The listed features will specify which ones should be enabled for the hotbar on the client. All other will be removed.

    For example:

    /hotbar:lock,aspect,resize,mute,volume
    where the options above control the following behaviours:

    • lock = lock/unlock input
    • aspect = stretch/maintain picture aspect ratio
    • resize = lock/unlock window resize
    • mute = mute/unmute audio
    • volume = volume control

    To enable all features: 

    /hotbar: all

    where all = enable all features

    Note

    • If the /hotbar option is not specified, the hotbar will have all features enabled.
    • If -hotbar is provided, hotbar will not be shown to the user.

    mosquitto_pub -h 127.0.0.1 -t "blackbox/sdkvm/deskvue/cmd/clients/unmute" -m {}

    The overlay can additionally be disabled setting the json payload:

    -m { "state": "disabled" }

    and enabled with

    1.19.1 Registering for Connection Status Events

    When a connection is terminated the client will send out a connection status event over MQTT. The OSD can subscribe to these events and terminate accordingly:

    MQTT Topic: 'blackbox/sdkvm/deskvue/workspace_server/event/connection

    {
    "boxilla":"id:08:00:27:c0:a1:9d",
    "client_uid":"800f998103",
    "connection_type":"transmitter",
    "connection_mode":"shared",
    "server_hostname":"192.168.1.161",
    "server_port":3389,
    "status":"disconnected",
    "reason":"",
    "error":"ERRBASE_SUCCESS",
    "time_stamp":"14:36:00 16-01-2023"
 }

    The "status" indicates the connection status and the OSD must use this status to reflect the correct text and color on the connection and workspace launch buttons.

    mosquitto_sub -h 127.0.0.1 -v -t blackbox/sdkvm/deskvue/workspace_server/event/connection

    1.20 Logging

    Add these environment variables for redirecting client logging to syslog

    export WLOG_APPENDER="syslog" 
export WLOG_PREFIX="pid=%pid:tid=%tid:fn=%fn -"

    Enabling debug logging on the client

    /log-level:debug

    1.21 Hot Key Handling

    • Currently we are using the 'Windows' hot key as our Controller Hoy Key. This poses certain issues, for example, the control sequence Windows + Up Arrow controls maximises the client window, but this key combination is also a popular hotkey sequence in Windows itself.
    • One option we have explored is to use the Ctrl key as an additional modifier key so when pressed in conjunction with combinations like Windows + Up Arrow would send the key sequence directly to windows.
    • Ultimately we want to move towards the industry standard control key sequences like Ctrl + Ctrl, Alt + ALt etc.
