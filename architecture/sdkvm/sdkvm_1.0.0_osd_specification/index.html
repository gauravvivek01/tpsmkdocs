    1.2 Terms and Definitions

    Term Definition
    Boxilla Blackbox's Appliance Management Software
    Decoder See Receiver
    Encoder See Transmitter
    Multiviewer A multiviewer is similar to a video wall. But while video walls require a separate TV display for each video source, multiviewers only require one display surface. Multiviewers are able to showcase full, split, quad, and custom configurations on the same pixel surface.
    PCoIP PC over IP (PCoIP) is a remote display protocol that Teradici developed for delivering remote desktops and applications to endpoints. PCoIP works by rendering client desktops on a network or cloud server. The desktop's pixels are then compressed, encrypted and then transmitted to the client device. The client then decrypts and renders the desktop view for the user.
    RFX RemoteFX Codec Extension developed by Microsoft which uses a lossy image codec to encode screen images with efficient and effective compression. This is the lossy scheme used in Blackbox products.
    Receiver A Blackbox appliance that receives and displays a video stream. A receiver is also referred to as a client or decoder.
    Source The term Source is commonly used to describe the PC connected to a transmitter appliance.
    Shared Mode A generic term used in Blackbox to describe schemes that allow more than one receiver view the video source from a transmitter. See Municast and Multicast.
    Transmitter A Blackbox appliance that captures and transmits a video from a source PC. A transmitter is also referred to as a server or encoder.
    Video Source A source of input for a Transmitter, normally the DVI or Display Port output of a PC.
    Video Wall A video wall is a special multi-monitor setup that consists of multiple computer monitors, video projectors, or television sets tiled together contiguously or overlapped in order to form one large screen. Typical display technologies include LCD panels, Direct View LED arrays, blended projection screens, Laser Phosphor Displays, and rear projection cubes.

    1.4 Role of the OSD in the overall SDKVM Architecture

    The OSD provides the main user interface for the SDKVM. The SDKVM is a multi-user system and each user can provision and manage a range of connections and workspaces.

    The OSD communicates with the workspaceserver through a REST API and MQTT messages to launch workspaces and connections. The status of these connections and workspaces are returned to the SDKVM either as MQTT messages or through polling the workspaceserver's REST API.

    1.8 EULA

    The first time the OSD launches on a new system it should present a window with the EULA to the user. The user must click accept before the OSD loads. On subsequent power ups the EULA is not presented, but if a factory restore takes place then the EULA should again be presented. This should also be viewable by clicking a "View End User License Agreement" button in the systems tab. Product Management will supply the text to be used in the EULA. The OSD can write a /home/snuc/info/eula_accepted.cfg file to indicate that the EULA has been accepted. The actual contents of the EULA can be read from : /home/snuc/info/eula_accepted.txt

    1.9 Help

    The OSD should display a help window with assigned hot keys when the user presses hotkey + h This can be a separate flutter application.

    • It should also have a single entry for hotkey + 0 - 9 for launching workspaces.
    • It should provide a summary of the other principle hot keys. | HotKey + r | Restore workspace to original layout | | HotKey + p | switch to the previous workspace | | HotKey + c | Move mouse pointer to the first screen (normally the OSD screen)| | HotKey + o | Maximise OSD in foreground | | HotKey + ESC | Minimise OSD in foreground | | HotKey + x | Close the active workspace | | HotKey + Up Arrow | Maximise active Window | | HotKey + Down Arrow | Minimise active Window | | HotKey + m | Mute all audio| | HotKey + u | Unmute all audio | | HotKey + b| Show hotbar on tiles |
    • The window should close when the user clicks outside it or when the user hits the escape key.
    • It should also be possible to bring this window up by hitting a '?' icon on the top bar
    • The OSD Helper window should centre itself on screen 1.
    • It should have a style consistent with the main OSD.
    • The window should not have a title bar.

    1.10 Logging

    The OSD must maintain a log of all user interaction in /var/log/syslog. All entries should be written through a stanard syslog interface.

    This should include the details of a user's interaction with the OSD, for example:

    • The OSD must maintain a log of all user interactions on the OSD screens in /home/snuc/SDKVM/logs/osd.log. All entries must have a date and time stamp.
    • If the user clicks a button then this action should be recorded.
    • If the user opens a dialog and selects a file, then this should be recorded.
    • Adding, deleting or editing users, connections or workspaces should all be recorded in the osd.log file.
    • If a user action causes a script to be launched then the path of the script should be recorded in the log with indications of success or failure of the launch.
    • If the user launches a workspace then the name of the workspace would be recorded, for example: "User [name] launched workspace ['name'] ['workspace ID'].
    • Indication on success or errors should be recorded, for example: "Workspace [name] [workspace ID] failed to launch: Error information [error details].

    1.11 Network Management

    The Administrator shall be able to:

        blackbox/sdkvm/deskvue/status/osd_sdkvm/workspace/1709ce654d/active

    In this example, "1709ce654d" is the workspace ID and "inactive" is the status.

    1.23 Inactivity Timers

    There are 3 inactivity timers that can be defined at a system level.

    1.23.1 Screen Saver

    This timer by default is disabled. The screen saver can be set from 1 to 999 minutes or never (i.e. displayed). When set, if there is no keyboard or mouse activity for the defined period – then the unit puts its attached monitors into low power mode. When keyboard or mouse activity is detected, then the monitors are brought back to normal operation. Setting the Screen Saver period can only be performed by an Administrator. (DPS Reference: UI0052)

    This will use the underlying Ubuntu power management, but the OSD will simply update the setting in /home/snuc/SDKVM/system_level_settings.json

        "screen_saver": "30",

    After the user changes this setting the OSD should update this file and send a POST message to:

    127.0.0.1:5000/settings
    This POST message will cause the SDKVM core to read the new settings.

    1.23.2 Workspace Inactivity timer

    This timer is by default is disabled. The Workspace Inactivity timer can be set from 1 to 60 minutes or never (i.e. never activated). When set, if no keyboard or mouse activity the workspace is closed (i.e. all connections closed) and OSD is displayed. (DPS Reference: UI0053a)

    The OSD will simply update the setting in /home/snuc/SDKVM/system_level_settings.json

        "workspace_inactivity_timer": "30",

    After the user changes this setting the OSD should update this file and send a POST message to:

    127.0.0.1:5000/settings
    This POST message will cause the SDKVM core to read the new settings.

    Only the Administrator can change this setting.

    1.23.3 OSD Inactivity timer

    This timer is by default is disabled. The OSD Inactivity timer can be set from 1 to 60 minutes or never (i.e. never activated). When set, if no keyboard or mouse activity and no active workspace (i.e. timer only starts when workspace is closed) – when timer period expires the user is logged out and log-in screen is presented to the user. (DPS Reference: UI0053)

    The OSD itself must maintain this timer and log the user out according to the setting.

    Only the Administrator can change this setting on an unmanaged system. On a managed system these settings are controlled by Boxilla.

    1.26 Managed and Unmanaged Mode

    The SDKVM system can operate in either managed or unmanaged mode. In managed mode the configuration (management) of the system is controlled by our Boxilla management platform. Boxilla is responsible for creating users, administering permissions, provisioning connections and creating workspaces.

    When a SDKVM is managed then the options to locally create Users and Connections and to configure device settings are disabled. All modifications must be done on Boxilla and downloaded to the SDKVM system. (DPS Reference: UI0134)

    In unmanaged mode a user with administrative access can create connections and assign them to workspaces. A user with administrative access should be able to allocate connections and workspaces to regular (local) users.

    • In unmanaged mode the SDKVM will support a maximum of 32-users and 32 connections.
    • All Administrators are assigned all Connections automatically in an unmanaged system.
    • An Administrator can unmanage the system (DPS Reference: UI0048b,UI0048c). Note: Only an Administrator can do this.

    1.26.1 Managing a SDKVM

    A SDKVM becomes managed through the process of discovery and management by Boxilla. When Boxilla has managed a SDKVM a two files will appear in:

        /home/snuc/SDKVM/boxilla
    These files are as follows:

    • The discovery file: /home/snuc/SDKVM/boxilla/discovery

    MGR=0:8c:10:1c:c6:d4
STATE=Discovered
DEBUG=OFF
    and the Boxilla Manager IP file: /home/snuc/SDKVM/boxilla/mgr_ip

    MGR_IP=10.211.129.3
    The OSD should use the presence of these files to determine if it is operating in managed or unmanaged mode. It should read the Boxilla mac address from the discovery file and use this as the Boxilla ID. The existing method of manually entering the Boxilla ID and serial number should be removed from the OSD. The OSD should indicate clearly if it is managed or unmanaged. Hovering on the Managed icon will provide additional information about the managing Boxilla entity including its IP address, Mac Address etc.

    Boxilla can choose to unmanage a device. In the process on unmanaging a device it will remove the users.json, discovery and mgr_ip files and also remove any existing user data structure and associated workspaces.

    If having been previously managed the OSD detects that these files and directories have been removed it should create a new empty users directory and a new users.json file.

    1.26.2 Administrator Permissions on a Managed System

    The Configuration option available to the Administrator on a managed unit are as follows: (DPS Reference: UI0135)

    • To over-ride the managed device setting by restoring to Factory defaults.
    • Set Keyboard locale on the system.
    • Set Transmitter configuration options for Transmitters that also reside within that managed domain.

    1.32.3 Registering for Connection Status Events

    When a connection is terminated the server will send out a workspace status event over MQTT (Port 1883). The OSD can subscribe to these events and terminate accordingly:

    MQTT Topic: topic blackbox/sdkvm/deskvue/status/osd_sdkvm/connection

    1.32.3.1 Example status for a specific connection

        blackbox/sdkvm/deskvue/status/osd_sdkvm/connection/1709ce654d/inactive
    In this example, "1709ce654d" is the Connection ID and "inactive" is the status.

    The OSD must listen for MQTT messages on the general topic blackbox/sdkvm/deskvue/status/osd_sdkvm/connection. When a message is received the OSD should determine the Connection ID, and terminate the any connect check associated with that ID if the status is set to inactive.

    Note: Multiple events may be received as the connections transition through a number of states; for example, when the connection is in the "connecting" or "connected" state it will send "active" status. When the connection is "terminating" or "terminated" it will send "inactive" events.

    This can be verified with mosquitto_sub:

    mosquitto_sub -h 127.0.0.1 -v -t blackbox/sdkvm/deskvue/status/osd_sdkvm

    For test purposes, a request to communicate a status change can be sent with mosquitto_pub

    mosquitto_pub -h 127.0.0.1 -t "blackbox/sdkvm/deskvue/status/osd_sdkvm/<connection_id>/inactive" -m " "
    where "" is the ID of the workspace you wish to terminate.

    1.32.3.2 Example status for all active connections

        blackbox/sdkvm/deskvue/status/osd_sdkvm/connection/inactive

    This is the general case where all active connections should be terminated. The OSD should iterate through any connections marked as active and reset their state to inactive.

    For test purposes, a request to communicate a status change can be sent with mosquitto_pub

    mosquitto_pub -h 127.0.0.1 -t "blackbox/sdkvm/deskvue/status/osd_sdkvm/connection/inactive" -m " "
