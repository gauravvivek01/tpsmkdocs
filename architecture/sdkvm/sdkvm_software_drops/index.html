    Software Drops For SDKVM

    SDKVM 1.0.0 EC/RC Drops

    EC1 - January 31st

    • System auto boots to the OSD
    • System can operate in unmanaged mode or managed mode (Managed mode is limited to making connections to managed appliances only, there is no Boxilla configuration)
    • A user with appropriate permissions can create connections and workspaces in an unmanaged device.
    • The OSD will display correct workspace status for active or inactive workspaces.
    • The SDKVM can be discovered by Boxilla.
    • The SDKVM can be managed by Boxilla. (but does not support Boxilla provisioned connections). It will copy the XML file (but not use it), place a Boxilla token on SDKVM, change IP address to a static IP.
    • Boxilla can reboot the SDKVM over REST
    • Boxilla can retrieve Product Details over REST
    • Boxilla can manually manage a SDKVM without having to depend on the discovery protocol.
    • The SDKVM can respond to a Boxilla Ping
    • The system will support multiple users in unmanaged mode and each of these users will be able to maintain a separate connection and workspace.
    • It will be possible to configure and persist network settings through the OSD allowing the user set a static or DHCP address. (Note: Boxilla does not support DHCP). Note: This is only possible in unmanaged mode.
    • Hotkeys are managed locally but there functionality will be limited to terminating connections. (Launching workspaces through a hot key will not be supported)
    • A user can launch workspaces based on pre-provisioned connections from the OSD workspace screen.
    • When a workspace is launched the OSD reflects the status of that launch. If the workspace is terminated by a hot key then the OSD will update to reflect that change in status.
    • Boxilla can retrieve information on a version compatibility check

    Not Supported

    • No audio support in this release.
    • Configuration of users and connection on the SDKVM through Boxilla is not supported.
    • It is not possible to launch Workspaces from hot keys

    EC2 - February 27th

    • Monitors designations, positions and resolutions can be managed through the OSD.
    • It will be possible to perform a basic upgrade of the system from a Thumb drive.
    • Boxilla can unmanage the SDKVM using the REST API and remove any Boxilla related settings.
    • Boxilla can change network settings on the SDKVM using the REST API.
    • The Boxilla Mac can be retrieved from the Discovery file and configured in the SDKVM system Settings File without the user having to enter this directly.
    • When a user logs out all active connections in a workspace are terminated.
    • It will be possible to configure and persist network settings through Boxilla.
    • Basic audio support on RDP, KVM and PCoIP connections.
    • Full connection failure and status messages on failed connections with appropriate messages.
    • It should be possible for the admin user to assign connections to other local (non-admin) users.
    • Access to the underlying operating system should not be possible.
    • Connection and workspace status should be reflected in the OSD and on the workspace launch buttons.

    EC3 - March 27th

    This is the first prototype for a version of the SDKVM suitable for demos. This will be the basis for RC1.

    • It will be possible to mute audio on a per tile level.
    • Full audio per tile control and integration
    • It will be possible to select the desired audio interfaces on the SNUC hardware through th OSD.
    • The SDKVM will provide informative and appropriate messaging for any connection failure or connection loss.
    • Complete System backup and restore to a thumb drive.
    • The System will support the retrieval of minimal diagnostic information and log files using a thumb drive.
    • Complete system lockdown available, but no bios or hard disk encryption.
    • Hotkeys are managed locally with the ability to configure locally for locally provisioned users. These mappings will allow users launch workspaces from mapped hot keys.
    • Workspace Restore functionality mapped to a hot key allows a user snap the workspace tiles to their original positions.
    • Connections launch and terminate reliably without freezing or dropout. (still an isue with freezing)
    • An Admin user can create workspaces and assign hot keys on a managed system.
    • It will be possible to perform a basic upgrade of the system from a Thumb drive.
    • It should be possible to set the keyboard locale from the OSD.
    • It should be possible to reboot the SDKVM from the OSD.
    • It should be possible to assign either DHCP or Static IP and have this persist through a reboot.
    • It should be possible to mute and unmute audio globally through a hot key (Windows +m, windows + u)
    • It should be possible to bring up the tile overlay controls for all tiles by pressing windows + b.

    RC1 - April 29th

    This is the first release for the SDKVM suitable for demos. This will be a preliminary proof of concept release. It can operate in managed or unmanaged mode, but the unmanaged mode is limited to Boxilla managing the device and facilitating connections to managed devices; however, Boxilla cannot configure the device or create workspaces and connections for the SDKVM.

    Likely Features:

    • It will be possible to mute audio on a per tile level.
    • Full audio per tile control and integration
    • It will be possible to select the desired audio interfaces on the SNUC hardware through th OSD.
    • The SDKVM will provide informative and appropriate messaging for any connection failure or connection loss.
    • Complete System backup and restore to a thumb drive.
    • The System will support the retrieval of minimal diagnostic information and log files using a thumb drive.
    • Complete system lockdown available, but no bios or hard disk encryption.
    • Hotkeys are managed locally with the ability to configure locally for locally provisioned users. These mappings will allow users launch workspaces from mapped hot keys.
    • Connections launch reliably from the OSD.
    • Connections launch reliably from the Hot Keys.
    • Connections can run on an extended soak without freezing or dropping.
    • The system can be discovered and managed by Boxilla, and it will also be possible to make connections to managed transmitters, but Boxilla cannot be used to create connections and workspaces or to otherwise configure the SDKVM.
    • An Admin user can create workspaces and assign hot keys on a managed system.
    • It will be possible to perform a basic upgrade of the system from a Thumb drive.
    • It should be possible to set the keyboard locale from the OSD login screen.
    • It should be possible to reboot the SDKVM from the OSD.
    • It should be possible to restore the system to factory defaults.
    • It should be possible to assign either DHCP or Static IP and have this persist through a reboot.
    • It should be possible to mute and unmute audio globally through a hot key (Windows +m, windows + u)
    • It should be possible to bring up the tile overlay controls for all tiles by pressing windows + b.

    Possible Features

    • An administrative user can create connections and workspaces and assign these to another standard user.
    • The system supports the configuration, hot swaping and reconfiguration of monitors.
    • Changes in the monitor configuration are propagated to existing workspaces.
    • Tile locations are calculated dynamically on launch by translating the virtual positioning system into actual screen co-ordinates.
    • Any additional support required for bios protection, system lock down and security.
    • Upgrade packages should be encrypted.

    Not supported in RC1

    • Leostream broker is not supported due to an inability to test and lack of a detailed formal specification or user scenarios.
    • The SDKKVM can be discovered and managed by Boxilla. It will also be possible to make connections to managed transmitters, but Boxilla cannot be used to create connections and workspaces or to otherwise configure the SDKVM.
    • The system does not support logins authenticated by Boxilla.
    • No Login and Logout events are sent to Boxilla.
    • The System will not support the retrieval of minimal diagnostic information and log files over Boxilla.
    • It will not be possible to upgrade the system from Boxilla.
    • Extended Desktop is not supported on any connection type.
    • Boxilla cannot retrieve statistics from the SDKVM.

    Limitations

    • The designated hotkey is the windows key and cannot be changed in this version.
    • Only horizontal placement of workspaces is supported in this release. Support for vertical placement will be supported in subsequent releases.
    • On the SDKVM we current cannot have a total pixel width or height over 16384 x 16384 this means than when using a 5K monitor the maximum resolutions that can be use is:
      • 1x Monitor at 5120x2160
      • 2x Monitors at 3840x2160
      • 1x Monitor at some resolution less than 3584 pixels wide.
    • DNS resolution of connection names does not work reliably.

    SDKVM 1.0.1

    • All relevant REST API interfaces are functional.
    • All short-listed bugs fixed
    • Full Boxilla integration :
      • Connections created in Boxilla can be provisioned on the SDKVM (assumes Boxilla does the necessary translation from XML into the json format for users and connections defined in the SDKVM data model). In the first iteration tile locations and widths for a template are normalised to a 1920x1080 screen - Data Model

    • The System will support the retrieval of minimal diagnostic information and log files over Boxilla.
    • It will be possible to upgrade the system from Boxilla or a Thumb drive.
    • The System will support the retrieval of complete diagnostic information and log files over Boxilla.
    • The System will support the retrieval of complete diagnostic information and log files using a thumb drive.
    • Dual network interfaces supported - active network interface detected on power-up and link change
    • Boxilla can force a user logout using the REST API.
    • Bios password protected
    • Extended Desktop Support
    • Unmanaged systems impose user, connection and workspace limits as defined in https://bboxjira.atlassian.net/browse/BUG-5139
    • The system will communicate changes in IP address to Boxilla over the Northbound Rest API
    • The system can be built from source for provisioning on a standard SNUC
    • Clonzilla image support for manufacturing
    • EC4 will be released in the form of an upgrade package.
    • Bios password protected
    • Secure boot
    • The system will implement a complete monitor management layer to manage monitor reconfiguration and hot-swap

    EC2 - Jule 22nd (provisional)

    • It will be possible to upgrade a TX from a SDKVM instance.

    • Variable size mouse cursor

    RC1 - August 30th (provisional)

    • All of the above with optimisation, profiling, critical bugs fixed and stabilisation of features.
    • Disk encryption to prevent mount and snoop
    • It will be possible to connect to Dual head transmitters and display each head in a different tile with provisioned audio.
    • Bonded connections

    • Beaconing for Mouse location
    • Licensing support

    • Production Grade Secure internal MQTT server for workspace server with TLS

    • Switch to previous workspace

    • Logged in users authentication through Boxilla and LDAP

    RC2 - September (provisional)

    • Statistics and Performance information for workspaces on the OSD
    • Statistics and Performance information for the OS Core on the OSD
    • Tile Pinning for fixed windows
    • Flexible Tile manipulation
    • User defined Templates
    • Dynamic Bandwidth Allocation
    • Zone support
    • Support for Organisational Units
    • Full Boxilla Integration - Boxilla understand and can manage the concept of workspaces and dependent connections, templates etc

    Workshop Agenda for SDKVM Scoping of 1.0.0

    Code Repo and Branching Strategy 9:30 10.00

    Discussion on the branching strategy and code management that we intend to use with the SDK This is a low level technical discussion of interest to SDKVM team only.

    Display Monitor Management 10:00 10:30

    Discussion on monitor management and its impact on the current design and data model. Micheal will discuss the proposed changes to accommodate monitor management and a move towards a virtual frame of reference to accommodate the requirements to create workspaces in Boxilla that are independent of monitor configuration. We are proposing a scheme for specifying workspaces and tiles that are independent of resolution. Getting this into the first version of the product is key as it will minimise future changes to the structure of the data model.

    This is a low level discussion of interest to SDKVM team and Boxilla team, but could have implications for customers use and expectations.

    Associated bugs for Monitor Management:

    • BUG-5086 OSD: Monitor detect function does not work reliably
    • BUG-4860 SDKVM/Openbox sometimes fails to detect all monitors
    • BUG-4595 OSD: Display screen Identify button doesn't work
    • BUG-5086 OSD: Monitor detect function does not work reliably
    • BUG-4860 SDKVM/Openbox sometimes fails to detect all monitors
    • BUG-5429 OSD: Display -Unable to update resolutions on multiple screens concurrently as unable to view confirmation prompt
    • BUG-5381 OSD: Display -If assigning primary status to screen on HDMI 1 (screen 1) then OSD incorrectly appears and stays on DP 2 (screen 2)
    • BUG-5368 OSD: Display -Black screen and partial screen after updating resolutions on two displays concurrently
    • BUG-5323 OSD: Display -Attempting to do a subsequent resolution change after the initial resolution change results in the selected display switching to the display which is the designated OSD display
    • BUG-4673 OSD: Recommending changes to Identify functionality in Display

    Assigning workspaces to other users. 10:30 11:00

    A requirement of the unmanaged and managed system is that an Admin user can create workspaces and assign then to other users. This has a lot of implications for the OSD and the data model. I will discuss a proposed solution for the data model, but we need product management input on how this will look in practise on the OSD. We need to give square root clear direction on how this should look and operate.

    This is a usability discussion of interest to SDKVM team and the Boxilla team, but could have implications for data model and customer experience.

    Questions

    • Can a power user create connections? - - No only admin can create connections or users.

    • Can a power user modify connections they have created? - No only admin can create connections or users.

    • Can an admin user allocate connections to other users, for example, can an admin user allocate connections to a power user so that the power user can subsequently use these connections in creating his own workspaces? - Yes - admin needs to be able to allocate connections to Power Users but in V1.0 only admin can create connections or users.

    • Can a connection created by a power user be allocated by an admin user to another admin user? Is this a copy of the connection, can the other admin user modify it independently of the power user that created it? - No only admin can create connections or users

    • Can a workspace created by a power user be allocated by an admin user to another user? Is this a copy of the workspace, can the other admin user modify it independently of the power user that created it? - Yes this should be possible, but only if the user has the access rights to connections in that Work-space

    • If an admin user allocates a workspace to another admin user. Is this a copy of the workspace, can the other admin user modify it independently of the original admin user that created it? - Yes all admin users can modify/update workspaces. Not sure there needs to be allocation of workspaces between admins . What can implement what is easier for V1.0.

    • Can an Admin user modify workspaces created by other admin users? Does the modification replace the existing workspace? - Yes all admin users can modify/update workspaces. Yes this replaces existing workspace for all.

    • Can a power user modify a connection allocated to it by the admin user? - Need more thinking on this as it can impact total available Workspaces in a Deskvue. For V1.0 we should do what is easiest even if means Power Users cannot create/modify workspaces.

    • Can a power user delete a connection allocated to it by an admin user? - No power users cannot delete connections allocated

    • Can an admin user delete a connection created by a power user? - Power-users do not have ability to create connections

    • If a connection is in an allocated workspace and the admin user (or power user) decides to delete the connection, should the deletion be prevented with a warning that the connection is in use, or should we delete the connection and remove it from any existing workspaces that feature it? -- Ideally we should prompt user with message saying in use in Workspaces (ideally listing workspaces where it used) user can decide to delete or cancel operation. It decided to delete connection it should be removed from all workspaces if features in. For V1.0 we may need to limit what we do .. time is more important than functionality
