    1.2 Connections

    • In a fully managed system users (admin, local and power) cannot modify, edit or add connections; they can only launch them to check that they are functional.
    • The options below outlined in black should be removed for all users in a fully managed system.
    • A power user can clone an existing workspace that has been assigned to them by the administrator.
    • It is not possible for any user (including admin types) edit or change connection settings on a managed system.
    • The option to add new connections should be removed for all user types in managed mode.
    • The connections screen should be visible admin and power users, but it should be hidden for local users.

    1.3 Proposal for Configuring Window Decorations

    Associated Jira ticket

    In the current SDKVM we create a window decoration bar (essential a title bar) with the following information

    • Connection Name
    • IP address
    • Close button

    For some applications like video walls it may be preferable to omit this window decoration.

    1.3.1 Longer Term Proposal For Window Decorations

    The longer term proposal for Window Decorations and Title Bars is as follows:

    • A workspace level setting will determine whether the title bar appears.
    • If Decorations are enabled then an additional drop down allows one to select what appears on the title bar.
      • Connection Name
      • IP address
    • The default settings for a workspace are:
      • Title bar enabled
      • Connection Name enabled
      • IP address disabled

    These settings are configured in the workspace definition as follows:

    {
  "workspace_name": "WS #5",
  "workspace_data_model_version_number": "1",
  "workspace_description": "",
  "workspace_id": "cb9f97c050",
  "workspace_created_by" : " A 10 digit unique ID for the creator or 'admin' if created by an admin user",
  "title_bars" : "True",
  "title_bar_connection_name" : "True",
  "title_bar_ipaddress" : "True",

    1.3.2 Proposal For Window Decorations for the IBC show

    For the IBC show it is proposed that we provide a decoration checkbox on the workspace edit screen. Enabling this checkbox would force decorations to be on for all tiles in the workspace. Disabling this would force decorations to be on for all tiles in the workspace.

    This would then use the existing "decorations" field in the tile section to enable or disable decorations.

    {
    ....
    "usbr": "False",
    "view_only": "False",
    "persistent": "False",
    "resizable": "True",
    "decorations": "True",
    "dual_head": "False"
}

    1.4 Proposal for Supporting force Login

    Note: This feature assumes managed mode of operation.

    1.4.1 Context

    It will be possible for a controlling entity (like Boxilla) to force the login of a user using the REST API on the SDKVM. The OSD will assist this login process in the following way.

    • The webservice component will send the above mqtt message to the OSD on receipt of Force Login from the REST API
        blackbox/sdkvm/deskvue/event/osd_sdkvm/force_login  -m {"username": "user"}
    • The OSD should subscribe to the topic: blackbox/sdkvm/deskvue/status/osd_sdkvm/force_login to receive this event.
    • The OSD has to validate the user name received in the above message against the users.json file.
    • The OSD does not have to authenticate this user and password through the Boxilla authentication process.
    • The OSD must raise an alert for FORCE LOGOUT for the previous user if another user is currently logged in (if no user is currently logged in then this is not required). The internal REST handler component will receive and process these alerts by adding them to the alerts queue.
    • If the user specified in the force login command is the same as the currently logged in user then the OSD must still perform a forced logout and forced login.
    • The OSD has to raise a FORCE LOGIN alert for the new user in the case of a succesful login. The internal REST handler component will receive and process these alerts by adding them to the alerts queue.
    • The OSD has to raise a failure alert for FORCE LOGIN for the new user in the case of a failed login.
    • The following format is used to send all of these alerts:
    "blackbox/sdkvm/deskvue/status/internal_rest_handler/alerts" -m {"alert_category": int,"alert_condition": int,"alert_component": int,
                                                            "alert_command": int, "alert_status":int, "alert_description":string,alert_context1": string,
                                                            ,"alert_context2":string,alert_context3": string,alert_context4": string}
    • Example Alert for Logout Success: (INFO, OPERATION, USER, FORCE_LOGOUT, SUCCESS, "OSD_LOGOUT", prev_user_name_as_string, "", "", "") where the fields like INFO and OPERATION are integer values that can be found in the tables below.
    • Example Alert for Login Success: (INFO, OPERATION, USER, FORCE_LOGIN, SUCCESS, "OSD_LOGIN", new_user_name_string, "", "", "") where the fields like INFO and OPERATION are integer values that can be found in the tables below.
    • Example Alert for Login Failed: (INFO, OPERATION, USER, FORCE_LOGIN, FAILED, "OSD_LOGOUT", prev_user_name_ch, "", "", "" ) where the fields like INFO and OPERATION are integer values that can be found in the tables below.

    The fields are populated as follows:

    alert_category

    Field Value
    CRITICAL 0
    WARNING 1
    INFO 2

    alert_condition

    Field Value
    METRIC 0
    TRAIT 1
    AVAILABILITY 2
    OPERATION 3
    SEVERITY 4
    MONITOR 5

    alert_status

    Field Value
    SUCCESS 0
    FAILED 1
    INPROGRESS 2
    CANCELLED 3
    TIMEOUT 4
    ONLINE 5

    alert_component

    Field Value
    USER 0
    APPLIANCE 1
    CONNECTION 2
    ETHERNET 3

    alert_command

    Field Value
    LOGIN 0
    AUTO_LOGIN 1
    MANUAL_LOGIN 2
    POST_CONNECT_LOGIN 3
    FORCE_LOGIN 4
    MANUAL_LOGOUT 5
    GUI_TIMEOUT_LOGOUT 6
    FORCE_LOGOUT 7
    REBOOT 8
    UPGRADE 9
    START 10
    DISCONNECT 11
    RDP_CONNECTION 12
    ETHERNET_LINK 13
    • The OSD also needs to update force login username and id in system_level_settings.json in the case of a succesful login.
    • The OSD must then go to the main window of the OSD using the login of the new user specified in the Force Login message.

    1.5 Proposal for Supporting force Logout

    It will be possible for a controlling entity (like Boxilla) to force the logout of a user using the REST API on the SDKVM. The OSD will assist this logout process in the following way.

    • The webservice will send an mqtt message in the format:
    blackbox/sdkvm/deskvue/event/osd_sdkvm/force_logout -m {}
    • The OSD should subscribe to the topic: blackbox/sdkvm/deskvue/status/osd_sdkvm/force_logout to receive this force_logout event.
    • The OSD must raise an alert for FORCE LOGOUT for the previous user if another user is currently logged in.
    • The OSD also needs to update force login username and id in system_level_settings.json in the case of a succesful logout these should be set to blank.
    "blackbox/sdkvm/deskvue/status/internal_rest_handler/alerts" -m {"alert_category": int,"alert_condition": int,"alert_component": int,
                                                            "alert_command": int, "alert_status":int, "alert_description":string,alert_context1": string,
                                                            ,"alert_context2":string,alert_context3": string,alert_context4": string}
    • Example Alert for Logout Success: (INFO, OPERATION, USER, FORCE_LOGOUT, SUCCESS, "OSD_LOGOUT", prev_user_name_as_string, "", "", "") where the fields like INFO and OPERATION are integer values that can be found in the tables above.
    • Example Alert for Logout Failed: (INFO, OPERATION, USER, FORCE_LOGOUT, FAILED, "OSD_LOGOUT", prev_user_name_ch, "", "", "" ) where the fields like INFO and OPERATION are integer values that can be found in the tables above.
    • The OSD must then go to the login screen.

    The fields are populated as shown in the tables above in the Force Login section.

    1.6 Force Workspace Proposal

    The Force Workspace feature allows a controlling entity like Boxilla or Control Bridge launch a Workspace by sending a command to the SDKVM's REST API. The OSD must assist this functionality in the following way:

    • To launch a workspace the Webservice will send an mqtt message: 
    blackbox/sdkvm/deskvue/cmd/osd_sdkvm/force_workspace -m {"username" : "user", "workspace_id" :  "id"}
    • The OSD should subscribe to the topic: blackbox/sdkvm/deskvue/cmd/osd_sdkvm/force_workspace.

    • There are three possible scenarios:

      • The specified user is already logged in:

        • In this case the OSD does not need to take any action regarding updating login details. It can simply launch the specified workspace.

      • There is no logged in user:

        • In this case the OSD must log the user in and update the system_level_settings file with the username and the associated user id. If the login is successfull it must also call the resrapi_cli (this will notify Boxilla that the specified user is logged into the SDKVM): " /usr/local/bin/restapi_cli -j user_login username &"
        • It can then launch the workspace:
        • In the case of failure to login the OSD must display an appropriate error message and should not launch the requested workspace and should not call the restapi_cli.

      • If the existing user and the specified user are different:

        • In this case the OSD must first logout the existing user and then attempt to login the specified user.
        • If the login is successfull it must update the system_level_settings file with the username and the associated user id ans also call the resrapi_cli (this will notify Boxilla that the specified user is logged into the SDKVM): " /usr/local/bin/restapi_cli -j user_login username &"
        • It can then launch the workspace:
        • In the case of failure to login the OSD must display an appropriate error message and should not launch the requested workspace and should not call the restapi_cli.

    1.7 Proposal for Extended Desktop Operation

    Associated Jira ticket

    The following is a proposal for dealing with extended desktop operation in the SDKVM:

    We add a connection level attribute called 'extended desktop' as a selectable radio button (shown below).

    When enabled this will make the option of a second head visible in the workspace configuration screen as shown below:

    The "dual_head" field of the associated connection will be set:

                    {
                    "connection_name": "Editing Machine",
                    "connection_id" : "A 10 digit unique ID",
                    "type": "PCOIP",
                    "target": "192.168.1.10",
                    "user_name": "BBOX",
                    "password": "cloud",
                    "domain": "",
                    "audio": "False",
                    "usbr": "False",
                    "view_only": "False",
                    "persistent": "False",
                    "dual_head": "True"
                }
    And the "dual_head" field of the associated tile will be set when the workspace is constructed.

          "tiles": [
        {
          "connection_id": "345ca583fb",
          "tile_name": "4K_192.168.1.169",
          "tile_location": "0,0",
          "tile_width": "25.0",
          "tile_height": "50.0",
          "target": "192.168.1.169",
          "type": "TX",
          "mode": "Shared",
          "audio": "False",
          "usbr": "False",
          "view_only": "False",
          "persistent": "False",
          "boxilla_id": "08:00:27:c0:a1:9d",
          "resizable": "True",
          "decorations": "True",
          "dual_head": "True",
          "target_head_id" : "1",
          "audio_type": "analog"
        },
    • A user can then select the tile associated with the second head.
    • The heads should operate independently of each other; for example:
      • It should be possible to select head 2 without selecting head 1
      • It should be possible to terminate one head of an active two head display without affecting the other head.
    • A connection with an extended desktop option can also be shared, so there can be multiple instances of head 1 and head 2 for a shared connection.
    • This adds significant complexity because it also means a dual head connection could also consist of just head 2. We could have a case where there are multiple shared instances of a dual head connection which has only head 2 active.
    • When the workspaceserver processes this template it combines any extended desktop selections into a single instance.

    The data model has been structured to allow each target head be associated with a single tile, so each target head of a connection can be placed on a different tile.

                "tiles": [
                {
                "connection_id": "345ca583fb",
                "tile_name": "4K_192.168.1.169",
                "tile_location": "0,0",
                "tile_width": "66.66",
                "tile_height": "1080",
                "target": "192.168.1.169",
                "type": "TX",
                "mode": "Shared",
                "audio": "False",
                "usbr": "False",
                "view_only": "False",
                "persistent": "False",
                "boxilla_id": "08:00:27:c0:a1:9d",
                "resizable": "True",
                "decorations": "True",
                "dual_head": "False",
                "target_head_id" : "1",
                "audio_type": "analog"
                },

    The preferred method is to specify a dual head connection with each head associated with a different tile. This approach gives us maximum flexibility; however, this approach adds complexity to the processing of workspace data and client launch command generation.

    1.7.1 Workspace Generation

    The OSD will present the user with dual head options for completing a template. If the user selects Head 1 of a dual head connection then the OSD will create a tile instance with the dual_head option set to "True". It will populate the location and size information consistent with its position in the chosen template. It will set "target_head_id" to 1.

    If the user selects Head 2 of a dual head connection then the OSD will create a tile instance with the dual_head option set to "True", and it will populate the location and size information consistent with its position in the chosen template. It will set "target_head_id" to 2.

    1.7.2 Workspace Processing

    The work of the workspace server component now becomes more complex. It must scan the workspace and when it finds a dual head connection it must defer the creation of the launch command until it has scanned the complete workspace to see if a second instance of that target exists. The algorithm uses three pieces of information to identify if a tile is part of a dual head connection:

    • "target": "192.168.1.169"
    • "dual_head": "True"
    • "target_head_id" : "1"

    We must remember that in the case of a shared connection there may be multiple instances of a single dual head target. This means that the workspace server must effectively pair tiles which refer to a single instance of a dual head connection.

    If it finds a first head instance and no second head then it simply displays that first head instance.

    If it finds a second head instance and no corresponding first head instance then it displays the second head instance only.

    1.7.3 Changes to Data Model for extended desktop

    The target head ID should be added to the tile definition

    "target_head_id" : "1"

    This can have one of two values:

    For the first head of a dual dead connection or for a single head connection.

    "target_head_id" : "1"

    For the second head of a dual dead connection.

    "target_head_id" : "2"

    1.8 GUI requirments for LACP/Bonding feature

    To enhance network reliability and performance, we propose binding Deskvue's dual Ethernet interfaces into a single logical link, enabling link redundancy and aggregation using the LACP (Link Aggregation Control Protocol) protocol.

    • Both Ethernet interface, “enp0s31f6” and “enp45s0” will be part of master bond “bond0”.
    • GUI should only refer to “bond0” for configuring network credentials such as IP address, subnet mask, gateway, DNS etc.
    • GUI should remove all restrictions for the “enp0s31f6” interface. It should not display an error notification when a network cable is plugged into the “enp0s31f6” interface.
    • An LACP toggle button is required to enable or disable the LACP functionality.
    • Similar to other network settings, if Deskvue is managed OSD should not let the user to enable/disable LACP button.
    • Upon toggling LACP button the GUI should call "/home/snuc/SDKVM/scripts/lacp_bonding.sh" script interface with following parameter.
      • LACP enable: --lacp (true or false)
      • Connection Name: --connection (static-ip or dhcp)

    Few examples:

    Turn ON LACP. Connection is “dhcp” /home/snuc/SDKVM/scripts/lacp_bonding.sh --lacp true –connection “dhcp”

    Turn ON LACP. Connection is “static-ip” /home/snuc/SDKVM/scripts/lacp_bonding.sh --lacp true –connection “static-ip”

    Turn OFF LACP. Connection is “static-ip” /home/snuc/SDKVM/scripts/lacp_bonding.sh --lacp false –connection “static-ip”

    Note: Argument is passed using ‘--‘ double hyphen.

    • lacp_bonding.sh script will take care of updating system_level_settings.json database.
    • When LACP is enabled, the script will switch to LACP bonding mode using following command.
    sudo nmcli con mod id <Connection Name> bond.options mode=802.3ad,miimon=100,lacp_rate=fast,xmit_hash_policy=layer2+3
    • When LACP is disabled, the script will switch to “Active-backup” bonding mode using following command.
    sudo nmcli con mod id <Connection Name> bond.options mode=active-backup,primary_reselect=2
    • In managed mode the LACP button is by default going to be disabled and user cannot modify its value.

    • In unmanaged mode the default value of LACP button is disabled

    • Connection name can be derived from “nmcli connection show –active”. I’d also like to inform that once the system is in bonding mode, you’ll get bond0-if1 and bond0-if2 active connection along with dhcp or static-ip. You just need to parse the static-ip or dhcp connection. 

      snuc@snuc-sdkvm:~$ nmcli connection show –active
NAME       UUID                                  TYPE      DEVICE
