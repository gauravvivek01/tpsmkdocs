    Leostream Brokered Connections

    Evaluation Infrastructure

    The test environment uses a number of virtual machines hosted on a Power Edge T330 ESXi server. Resource limitations (mainly free RAM) on the ESXi server forced us to use minimal Linux servers with small foot prints (512 MB RAM allocated individually).

    Configuration of Hardware

    Internal to the ESXi server:

    • 1 x Leostream Broker running on Linux
    • 1 x Active Directory Domain Controller (via Samba) running on Linux
    • 1 x LDAP Server running on Linux
    • 2 x Windows 10 Desktops (joined to the Active Directory Domain) running the Leostream Agent
    • 2 x Linux Desktops (1 x Fedora, 1 x open SUSE) running the Leostream Agent

    External to the ESXi server:

    • 2 x Windows 7 Desktops running the Leostream Connect software (doubled up from debug terminals)

    Launching Connections via Leostream Broker

    Users have two ways of authenticating:

    • Via defined authentication servers
    • Locally defined Leostream users

    Two authentication servers are currently defined in the test setup in the following order of precedence:

    • Active Directory Domain
    • OpenLDAP

    After successful authentication, the user may launch a connection to an available desktop from the desktops assigned to them from a defined pool.

    When launching connections the Leostream software automatically passes the same credentials which were supplied during authentication and thus if that user is not defined on the desktop been connected to then the user will normally be presented with an error message followed by a prompt to manually provide credentials.

    Truth Table For Desktop Authentication:

    AD Domain User openLDAP User AD Domain Desktop Non AD Domain Desktop
    Automatic Desktop Logon X X
    Manual Desktop Logon X X X

    There are two pools currently defined in the test setup with the policy defined to allocate one desktop per pool assigned to a user.

    Leostream Pools Summary:

    Pool Name Desktop 1 Desktop 2
    Windows Windows10EntNo1 Windows10EntNo2
    Linux Fedora36No1 OpenSuseLeapNo1

    Desktop AD Domain Summary:

    Desktop Name AD Domain Non AD Domain
    Windows10EntNo1 X
    Windows10EntNo2 X
    Fedora36No1 X
    OpenSuseLeapNo1 X

    Active Directory Domain Controller Configuration and Setup

    The Active Directory Domain Controller runs on a Linux distro provided by TurnKey Linux

    Useful Links

    • Domain Controller
    • Domain Controller Documentation

    Hardware Requirements

    • 1 VCPU, 8 GB of space and 512 MB of memory.

    Note: If changing network settings (e.g. post installation) be sure to rerun the domain provision inithook /usr/lib/inithooks/bin/domain-controller.py.

    The underlying software providing the domain services is Samba (Samba is actually capable of providing Active Directory Domain Services though work is in progress to develop support for the more advanced Active Directory Features e.g. Roaming Profiles).

    The main samba config file is located in /etc/samba/smb.conf.

    Though there is a web interface for managing the Samba users and the Active Directory configuration, but this is very limited.

    Another option is to open a local / remote (SSH is supported out of the box) terminal on the server and use the samba-tool CLI to affect the desired changes.

    Another option which is significantly more user-friendly is to remotely manage it via RSAT (use the Active Directory Computers and Users component) on a Windows PC on the domain. Later builds of Windows 10 have RSAT available and just require it to be enabled.

    When adding desktops to the domain use the Samba Administrator account that you specified during the initial installation and ensure that all desktops which are joining are using the Active Directory Domain Controller as their primary DNS.

    The default netbios name is DC1, and this should be used in any domain functions which may use the netbios name.

    The domain in the test setup is emerald.lan though this may also be referred to as the realm.

    From a configuration perspective one will primarily be assigning specific users to Groups which the Leostream Broker uses to determine what policy (can be loosely thought of as configuration) applies to a particular user.

    OpenLDAP Server Specific

    The OpenLDAP Server runs on a Linux distro provided by TurnKey Linux.

    Hardware Requirements

    - 1 VCPU, 6 GB of space and 512 MB of memory.

    The Installation is intuitive through a web interface provides functions for managing the OpenLDAP service.

    It does require a domain to manage the LDAP service (i.e. the directory structure for users); this has been specified as emerald.com.

    From a configuration perspective one will primarily be setting the OU attributes for specific users which the Leostream Broker uses to determine what policy (can be loosely thought of as configuration) applies to a particular user.

    Leostream Connect Configuration and Setup

    • When a user logs in the software will automatically launch the first connection assigned to the user from the Leostream Broker (this setting can be modified).
    • All connections are launched with the user credentials provided to the Leostream Broker when logging in the Leostream Connect software thus user account credentials should optimally be synchronised between the Leostream Broker and all Desktops running the Leostream Agent otherwise credentials will have to be input manually per connection launch (one disadvantage of using internal Leostream Broker users). In this case a clunky workaround is to manually define all user accounts to be the same on the Leostream Broker and the Desktops running the Leostream Agent but this is not ideal for larger setups both in the initial configuration and ongoing maintenance.
    • Centralised user control and management via a network domain is the only realistic solution for Leostream deployments with a significant number of users and desktops.

    Leostream Agent Specific

    • Assuming the Leostream Agent successfully installed and configured then the desktop will automatically register as a resource on the Leostream Broker.
    • Windows 10 installation and configuration was trivial.
    • Linux installation is more challenging. It has been installed on Fedora and openSUSE Leap, but Ubuntu or openSuse Tumbleweed require more research.

    Notes on Linux Installation

    • You must be logged in as root for installation to complete successfully i.e. no sudo.
    • The agent installer uses Java which requires the JDK to be installed on the desktop. Linux distros dont come with the official Oracle JDK but rather the open source version of OpenJDK.
    • OpenSUSE Leap does require insserv-compat to be installed.
    • Note: OpenSUSE uses a different desktop for RDP connections than its native one i.e. not KDE. You also require the kernel-devel packages to be installed (per the release notes). Be sure to install XRDP to allow RDP connections via the appropriate package manager because it does not come pre-installed on most distros.
    • For configuring the firewall its easiest to either use the desktop firewall UI to switch the network interface to the trusted zone or to disable the firewall entirely with the systemctl commands.
    • The agent uninstaller is created in /opt/leostreamagent/lib.

    Useful Linux commands

    Installing insserv-compat on openSUSE

    -sudo zypper refresh
-sudo zypper install insserv-compat

    If UnInstalling insserv-compat on openSUSE

    -sudo zypper remove insserv-compat

    Installing XRDP and opening the firewall port for RDP in openSUSE

    - sudo zypper -n in xrdp
-sudo systemctl enable xrdp
-sudo systemctl start xrdp
-sudo firewall-cmd --add-port=3389/tcp --permanent
-sudo firewall-cmd --reload

    Changing the root password:

    sudo passwd root

    Linux setting hostname:

    Hostnamectl set-hostname new-name

    An example of setting a static IP in a Debian based Linux Distro interfaces file:

    auto eth0
    iface eth0 inet static
        address 192.0.2.7/24
        gateway 192.0.2.254

    Controlling firewalld via systemctl:

    systemctl status firewalld
systemctl stop firewalld
systemctl disable firewalld
systemctl enable firewalld

    Installing kernel-devel on Fedora:

    dnf install kernel-devel

    Leostream Broker Specific

    Summary of assigning desktops to users

    Pools (Configuration menu) for Leostream agent hosts (Desktops) are defined and are assigned to Policies (Configuration menu) where individual Policies may contain multiple pools and Policies are used in Assignments (Configuration menu) for Authentication servers where users are assigned available Leostream agent hosts from a policy based on Groups for Active Directory users or OU Attributes for OpenLDAP users.

    Allocation of a desktop by the Leostream Broker

    When a user logs on to the Leostream Connect client and is allocated desktops from a Pool by the Leostream Broker then the desktops are considered Assigned and unavailable to be allocated to other users.

    This is the case even if the user is not actually logged on to the specific desktops i.e. its a possibility that a user merely logs on to the Leostream Connect client and doesnt connect to any of the desktops allocated to him by the Leostream Broker. These desktops will then be unavailable to other users until the user logs off the Leostream Connect client. I believe (needs to be investigated) that the Leostream Breaker has multiple configuration options available to mitigate desktop availability been constrained by idle users.

    Adding Active Directory Domain Controller as an authentication server

    Adding the Active Directory Domain Controller to Leostream (Setup, Authentication Servers) using the default Active Directory Administrator will not work. One must create a second admin user admin and use those credentials.

    Details

    • Name: TurnKey_Domain_Controller_103
    • Domain: emerald.lan
    • Hostname or IP address: 10.211.131.103
    • Encrypt connection to the authentication server using SSL (LDAPS): Enable
    • Login name or DN: CN=admin,CN=Users,DC=emerald,DC=lan
    • Password: user account admin password
    • Sub-tree:Starting point for user search: dc=emerald,dc=lan

    Note: if Active Directory is updated e.g. new groups that the Leostream Broker doesnt seem to automatically recognise the update and that you have to remove and then re-add the controller in Leostream for it to recognise the changes. Something to check with the Leostream contacts?

    Adding OpenLDAP as an authentication server

    • Nothing specific to note regarding OpenLDAP
    • Like Active Directory but adjust accordingly e.g. for Login name or DN: cn=admin,dc=emerald,dc=com (No need for CN=Users)
