    Boxilla Interface Specification for SDKVM 1.0.0

    As part of the development of the SDKVM and Opal we need to be able to interface with our Boxilla Management Platform in a similar manner to our existing appliances.

    In order to provide the necessary API calls we propose implementing the existing REST API on SDKVM and Opal by porting the C libraries associated with librestapi, webservice and discovery to the Ubuntu 20.04 platform.

    Some of the existing Boxilla API uses SSH. We do not propose implementing these SSH based APIs, but instead will replace them with REST API calls (and eventually in a later release by MQTT calls)

    REST API and Web Services

    The appliance REST API and web-service provides an existing appliance side south bound interface and a Boxilla north bound interface. This will be ported to the SDKVM so that Boxilla can re-use the existing messages structures. A complete analysis of existing appliance API calls and data flows is required so that we can implement a similar model.

    • The REST API must use the existing security configuration employed by the appliances. In particular, it must support:
      • HTTPS over TLS using the Boxilla self signed certificate.
      • This API needs to specify the content of the REST API Header.
      • This API needs to support Basic Authentication via Username & Password.

    • The REST API on the appliance has a dependency on the system libraries. These are not available on the SDKVM so an equivalent set of calls will have to be created.

    Replacing SSH Commands used by Boxilla

    Boxilla uses an SSH connection to the appliance for certain commands:

    1. Reboot A REST based reboot command exists - The API should call a script in /home/snuc/SDKVM/scripts/reboot.sh
    2. Restore/UnManage via restore script (no equivalent command exists) - The API should call a script in /home/snuc/SDKVM/scripts/unmanage.sh
    3. Get Product Details A REST based version is available using the version and system facts commands
    4. Software Upgrade (including transfer of the image file) (No equivalent REST API) - This needs a number of components:
    5. An ability to upload a file.
    6. A new API should to begin the upgrade process. It will call a script called in /home/snuc/SDKVM/scripts/upgrade.sh
    7. A new API that allows Boxilla determine the upgrade progress - The API should report the contents of a script in /home/snuc/SDKVM/info/upgarde_progress.txt. This will contain a figure between 0% and 100% to indicate progress or a text string indicating any errors that have occured in the upgrade process.
    8. Manage Appliance or Manually Add. To manually add an appliance of SDKVM Boxilla needs:
      • Architecture, uname -m,
      • Kernel Version, uname -r
      • PC Model, uname -n
    9. Set IP once the appliance has been managed (No available API to get IP address): - The API should call a script in /home/snuc/SDKVM/scripts/update_network.sh it should pass the following information:
      • Mode: Static or Auto, if manual it should pass the following
      • IP Address
      • Netmask
      • Gateway
      • DNS
    10. Get IP settings Endpoint. This will return the following information:
      • Mode: Static or Auto. In both cases it should return the the following
      • IP Address
      • Netmask
      • Gateway
      • DNS
    11. Boxilla currently retrieves the following information:
      • Architecture, uname -m,
      • Kernel Version, uname -r
      • PC Model, uname -n

    These are retrieved over SSH but these should be added to the version endpoint

    It is proposed that these commands be replaced with an equivalent REST API call.

    Missing End Points:

    The following end-points would be required:

    1. An endpoint to restore the Transmitter or Receiver. Execute the restore shell script on the device when Boxilla performs an Unmanage operation and return success or failure status. Currently Boxilla creates a new ssh connection and runs the shell script on behalf of device.
    2. An endpoint to clear out all the previous image files (.clu or .fit) to avoid possible failure with no space available error. Boxilla can call the REST API and check if available space is there in the device and then can start uploading the upgrade file to the device. Path: /home/snuc/SDKVM/upgrade
    3. An endpoint to upload the .clu or .fit file to the appliance. Which then gets stored into /tmp directory. at the appliance side, to accommodate the current flow followed in the Boxilla. On the SDKVM these files are stored in /home/snuc/SDKVM/upgrade
    4. An endpoint to get the md5sum of the uploaded file. Boxilla validates the file is correctly uploaded to the appliance by generating md5sum on the device machine itself and compare it the md5sum on the file local to that of Boxilla and compares it to be valid.
    5. An endpoint to run the Upgrade process on the new uploaded file.
    6. An endpoint to check if the upgrade has started.
    7. An endpoint to remove/clean the uploaded file once the upgrade process is completed. The path for upgrade files is /home/snuc/SDKVM/upgrade
    8. An endpoint to provide monitor details for a SDKVM. It would return a json object of the form:

    Porting Guide

    We wish to port this code from the appliance to the SDKVM and Opal projects, but the code should remain as common as possible at the REST API and webservice level because we do not want to maintain to different code bases.

    The Webservice and REST API depends on the System Library on the appliance. The system library provides access to low level functions on the appliance. These will not be present on the SDKVM or Opal so we will need the API calls to call the relevant low level functions depending on whether they are running on the appliance or the SDKVM.

    It might make sense to create a separate a system library for the SDKVM platform and Opal products. The layers above that in REST API should not contain any appliance specific code. They should only contain API code. All appliance calls or system calls should be moved out of that library.

    In the short term we are going to use the REST API for many functions that might in the longer run be more suitable for MQTT. We should consider writing any API calls so that they can easily migrate to MQTT in future releases.

    The discovery and webservice binaries will need to be upgraded at some future point by the snuc user which will not have sudo access, so binaries should be installed in /usr/local/bin ideally. This is where the systemd service should also run them from. They should also be statically linked to avoid the necessity for shared library access.

    Source Code Control

    We propose using GitHub for the common packages like RestAPI, Webservice and Discovery. These will exist in separate repositories which will eventually be integrated into a common SVN build system.

    Login and Authentication

    AD users

    AD user login
    • /api/ad/v1/user/login: BXA API to process AD user login request.
    • /api/ad/v1/user/auth_status: RX device polls this api to check status of AD Login request.
    • bxa-api/users/status: to notify user login (update user status, for successful login only); this is used to notify BXA that the RX has copied over the user.xml file.

    Static KVM Users

    Local user login via XML file. - User validation is local to the RX, user name & password exists in the xml file. - Post bxa-api/users/status to notify Boxilla of the login success (for successful login only)

    Log Off Notification

    • User log off, RX device uses the following API: send delete -> bxa-api/users/status to notify Boxilla of a user log-off

    Zones and Users

    - Basically Zones are being used to manage RX devices by boxilla in particular areas(zones)
- The zone value set and delete by Boxilla using restapi
- The zone value will be considered when OSD do user login manually or Auto as well as Boxilla do user login thorough REST API to get zone specific clouddata xml file from boxilla for user validattion if zone value set by Boxilla

    Managed and Non-Managed State

    The SDKVM will support both managed and non-managed modes of operation. Boxilla can discover (see discovery below) SDKVM devices and then manage them. The process of managing a device is similar to that used in managing an appliance.

    Un-managing a device

    To unmanage a device Boxilla will use a REST API call to invoke a restore-script on the SDKVM. This will be provided by the SDKVM Core Utilities.

    Discovery Protocol

    We need to port the existing appliance discovery protocol to the SDKVM. This has some integration to the appliance OS, for example, changing the IP address. This will need to be modified to accomodate the specific needs of the SDKVM.

    • Port the existing discovery protocol to run as a systemd service.
    • Document the various phases of discovery so that we can review.
    • Refactor any OS specific dependencies.
    • Transition Ubuntu OS from Network Manager to Systemd based network configuration.

    Configuring Network Settings on Ubuntu 20.04/22.04

    The discovery code must be modified on Ubuntu to configure the IP address using Network Manager.

    Allowing non-root user to run nmcli

    '''sh chmod u+s,a-w /usr/bin/nmcli 

    -[Setting a static ipaddress](https://linuxhint.com/setup_static_ip_address_ubuntu/)

We can set the IP using the nmcli interface.

```sh

nmcli con add type ethernet con-name 'static-ip' ifname enp86s0 ipv4.method manual ipv4.addresses 192.168.1.98/24 gw4 192.168.1.254
nmcli con up id 'static-ip' #check IP is setup
nmcli connection show # show available connections
nmcli connection delete static-ip # delete a connection
nmcli con mod static-ip ipv4.dns 182.168.1.254 # add dns
nmcli con up id 'static-ip' # activate the static connection
nmcli con add type ethernet con-name 'dhcp' ifname enp86s0 ipv4.method auto # DHCP example

    Audit Process

    • We need to understand the audit process from Boxilla because this will determine when the database will be pulled and when the SDKVM will transition to a managed state.
    • Document phases of the audit sequence for review.

    Upgrade

    • The SDKVM System should support upgrades by Boxilla.
    • SDKVM will use an upgrade file in the form of a debian package (.deb file). Boxilla will transfer this file to the SDKVM over a REST interface. This is a new REST interface which will need to be added to the REST API being ported from the appliance.
    • We need to create an upgrade script that will receive and process the debian package and report on progress. This will be provided as part of the SDKVM Core Utilities.

    XML to JSON Translation

    The SDKVM uses JSON files to manage users, connections and workspaces. It is proposed that Boxilla create these assets and down load then to the SDKVM.
