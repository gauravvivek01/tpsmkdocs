    SDKVM and Control Bridge

    References

    • Sturle Havsvik in the Scandinavian office controls the Control Bridge development in Europe.
    • IBC PRM
    • Jira Ticket Tracking Development
    • IBC is held in Amsterdam from 15 - 18 September 2023. These features are initially targetted for a demo at this show.

    Requirements

    • We will need to support control bridge operation on the SDKVM. This should be compatible with the existing appliance implementation.
    • The existing RX appliance can support certain control bridge operations via an unsecure REST API.
    • When we developed SDKVM initially this unsecure API was not supported. This unsecure API is enabled through Boxilla and can handle basic commands like launching and terminating connections. The unsecure API should only run when enabled by Boxilla. It should not run on bootup.
    • The existing C based implementation should be ported to the SDKVM.
    • The means of launching the actual connection and terminating on the SDKVM will be different. The connections can be launched by sending the necessary json connection object to the internal REST API server used by the workspace server. The unsecure RESTAPI will parse the control bridge data and form a suitable json object that can be sent to the workspaceserver API.
    • The necessary API details for the internal workspaceserver API are here - see Connection and Workspace APIs (note these may evolve for 1.2, but this document will always reflect the latest API)

    Proposed Architecture A

    • Port the existing unsecure REST API from the appliance.
    • Replace the existing connection launch and connection termination calls with those specified here - see Connection and Workspace APIs.
    • Data coming from Control Bridge should be parsed and translated into a JSON object that is compatible with the existing workspaceserver API.
    • This approach would require us to add a new API to handle workspaces (as opposed to connections)

    Alternative Proposal B

    • Control bridge can talk to Boxilla over https.
    • We use this existing scheme and allow Boxilla send the workspace launch and terminate commands to the SDKVm over the existing API.
    • This approach would require Boxilla to develop APIs (modelled on the connection API) that could launch and terminate a workspace.
    • This approach would require SDKVM to develop new secure APIs (modelled on the connection API) that could accept the launch and termination of a workspace.
    • Boxilla already has the capability to export its unsigned certificate so that other clients can control various functions through the northbound API.
    • Replace the existing connection launch and connection termination calls with those specified here - see Connection and Workspace APIs.

    Another Alternative Proposal C

    • Control bridge can talk to the secure REST API directly.
    • I am not aware if this has ever been done and if possible would require certificate installation etc or a capability to export the appliances unsigned certificate.
    • This approach would require SDKVM to develop new secure APIs (modelled on the connection API) that could accept the launch and termination of a workspace.
    • Replace the existing connection launch and connection termination calls with those specified here - see Connection and Workspace APIs.
    • This approach would not require Boxilla development.

    Implementation Details

    For Proposal A

    Force Login Requirements

    PUT User Login (Force Login)

    URL

    http://IP:7778/user/login

    Implementation Notes

    Logs a user in to Rx OSD

    Request Class JSON Structure
    {
            "user": "kvm_user",
            "force": 1,
            "ad": 1,
            "md5sum": "a7d794f4a687efd2094c589714e310e3"
}
    Request Class Formats
    Parameter Format
    user 32 byte char array max
    force 1/0
    ad 1/0
    md5sum 32 byte char array
    Response

    Http status code.

    Force Login Webservice Role

    {
            "user": "kvm_user",
            "force": 1,
            "ad": 1,
            "md5sum": "a7d794f4a687efd2094c589714e310e3"
}
    • Notes on value "user":
      • Force login will not work for username "Boxilla"
      • For force login the user should be an existing user in boxilla and be present in the in sdkvm users database"
    • Notes on "force" value:
      • 1, This will ill work irrrepective of login status.
      • 0, No user should be logged in, this means the OSD must be in a logged out state.
    • Notes on "ad" field ad = 0 for static user ad = 1 for an active directory user (currently SDKVM does not support ad users)
    • Notes on md5sum
      • If the checksums mismatch for database.json file then the SDKVM (webservice) will update the database.json, using the North Bound Rest Interface: 
        "https://%s/api/deskvue/v1/users"
    • The webservice will terminate any existing workspaces using http://localhost:5000/workspace restapi to get the workspaces status and then use the command below to terminate the connection: 
    /home/snuc/KeyBindings/Scripts/terminate_workspace/home/snuc/KeyBindings/Scripts/terminate_workspace.sh
    • The webservice will publish an mqtt message to the OSD after terminating the current workspace connections. This will take the following form:
        blackbox/sdkvm/deskvue/event/osd_sdkvm/force_login  -m {"username": "user"}

    Force Login OSD Role

    • The webservice component will send the above mqtt message to the OSD on receipt of Force Login from the REST API, so the OSD should subscribe to the topic: blackbox/sdkvm/deskvue/status/osd_sdkvm/force_login to receive this event.
    • The OSD has to validate the user name received in the above message against the users.json file.
    • The OSD must raise an alert for FORCE LOGOUT for the previous user if another user is currently logged in. The internal REST handler component will receive and process these alerts by adding them to the alerts queue.
    • The OSD has to raise a FORCE LOGIN alert for the new user in the case of a succesful login. The internal REST handler component will receive and process these alerts by adding them to the alerts queue.
    • The OSD has to raise a failure alert for FORCE LOGIN for the new user in the case of a failed login.

    • The following format is used to send all of these alerts: 

      "blackbox/sdkvm/deskvue/status/internal_rest_handler/alerts" -m {"alert_category": int,"alert_condition": int,"alert_component": int,
                                                            "alert_command": int, "alert_status":int, "alert_description":string,alert_context1": string,
                                                            ,"alert_context2":string,alert_context3": string,alert_context4": string}

    • Example Alert for Logout Success: (INFO, OPERATION, USER, FORCE_LOGOUT, SUCCESS, "OSD_LOGOUT", prev_user_name_as_string, "", "", "") where the fields like INFO and OPERATION are integer values that can be found in the tables below.

    • Example Alert for Login Success: (INFO, OPERATION, USER, FORCE_LOGIN, SUCCESS, "OSD_LOGIN", new_user_name_string, "", "", "") where the fields like INFO and OPERATION are integer values that can be found in the tables below.
    • Example Alert for Login Failed: (INFO, OPERATION, USER, FORCE_LOGIN, FAILED, "OSD_LOGOUT", prev_user_name_ch, "", "", "" ) where the fields like INFO and OPERATION are integer values that can be found in the tables below.

    The fields are populated as follows:

    alert_category

    Field Value
    CRITICAL 0
    WARNING 1
    INFO 2

    alert_condition

    Field Value
    METRIC 0
    TRAIT 1
    AVAILABILITY 2
    OPERATION 3
    SEVERITY 4
    MONITOR 5

    alert_status

    Field Value
    SUCCESS 0
    FAILED 1
    INPROGRESS 2
    CANCELLED 3
    TIMEOUT 4
    ONLINE 5

    alert_component

    Field Value
    USER 0
    APPLIANCE 1
    CONNECTION 2
    ETHERNET 3

    alert_command

    Field Value
    LOGIN 0
    AUTO_LOGIN 1
    MANUAL_LOGIN 2
    POST_CONNECT_LOGIN 3
    MANUAL_LOGOUT 4
    GUI_TIMEOUT_LOGOUT 5
    REBOOT 6
    UPGRADE 7
    START 8
    DISCONNECT 9
    RDP_CONNECTION 10
    ETHERNET_LINK 11
    • The OSD also needs to update force login username and id in system_level_settings.json in the case of a succesful login.
    • The OSD must then go to the main window of the OSD using the login of the new user specified in the Force Login message.

    • The OSD must then go to the login screen.

    Force Workspace Requirements

    Proposed Force Workspace REST API

    PUT Force Workspace Unsecure

    URL

    http://IP:7778/control/workspaces

    IMPLEMENTATION NOTES This node establishes a connection between the SDKVM and the Transmitters specified in the workspace. This object contains parameters used to log in a user and launch an associated connection. The user and associated connection must be configured prior calling this URL.

    Request Class JSON Structure
    {
    "action": "force_workspace",
    "workspace": "workspace-id",
    "user": "cb-user",
}

    REQUEST CLASS FORMATS

    Parameter Format
    action force_workspace
    workspace 32 byte char array
    user 32 byte char array

    Response

    Http status code.

    Force Workspace Webservice Role

    • The webservice receives a force workspace message on the REST API.
    • The Webservice validates the message by checking the user name against the user names in users.json.
    • The webservice also validates that the specified user has acess to the specified workspace and that the workspace ID is valid.
    • The webservice then send the following MQTT message to the OSD;

    blackbox/sdkvm/deskvue/cmd/osd_sdkvm/force_workspace -m {"username" : "user", "workspace_id" :  "id"}
    - The internal Rest Handler will supscribe to TBD message { "usename" : "user" } and send a post to https://%s/bxa-api/users/kvm/status on Boxilla.

    Force Workspace OSD Role

    • The OSD should subscribe to the topic: blackbox/sdkvm/deskvue/cmd/osd_sdkvm/force_workspace.

    • There are three possible scenarios:

      • The specified user is already logged in: In this case the OSD does not need to take any action regarding updating login details.
      • There is no logged in user: In this case the OSD must log the user in and update the system_level_settings file. It must also call the resrapi_cli (this will notify Boxilla that the specified user is logged into the SDKVM): 

      /usr/local/bin/restapi_cli -j user_login username &
      - If the existing user and the specified user are different: In this case the OSD must logout the existing user and login with the specified user. It must also call the resrapi_cli (this will notify Boxilla that the specified user is logged into the SDKVM): 

      /usr/local/bin/restapi_cli -j user_login username &

    • If the specified user is not logged in then the OSD must login the specified user.

    • The OSD must update the system_level_settings file to the username and the associated user id.
    • If the login of the user fails the OSD must display an appropriate error message.
    • The OSD must then launch the specified workspace.
