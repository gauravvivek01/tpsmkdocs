    1. SDKVM Architecture

    1.1 Introduction

    The Software Defined KVM product (SDKVM) is a PC based offering that allows a user connect to multiple different sources from a single same pixel multihead, multiscreen application.

    1.2 Terms and Definitions

    Term Definition
    CAS Teradici CAS (Cloud Access Software)
    Decoder See Receiver
    Encoder See Transmitter
    Multiviewer A multiviewer is similar to a video wall. But while video walls require a separate TV display for each video source, multiviewers only require one display surface. Multiviewers are able to showcase full, split, quad, and custom configurations on the same pixel surface.
    PCoIP PC over IP (PCoIP) is a remote display protocol that Teradici developed for delivering remote desktops and applications to endpoints. PCoIP works by rendering client desktops on a network or cloud server. The desktop's pixels are then compressed, encrypted and then transmitted to the client device. The client then decrypts and renders the desktop view for the user.
    RFX RemoteFX Codec Extension developed by Microsoft which uses a lossy image codec to encode screen images with efficient and effective compression. This is the lossy scheme used in Blackbox products.
    Receiver A Blackbox appliance that receives and displays a video stream. A receiver is also referred to as a client or decoder.
    Source The term Source is commonly used to describe the PC connected to a transmitter appliance.
    Shared Mode A generic term used in Blackbox to describe schemes that allow more than one receiver view the video source from a transmitter. See Municast and Multicast.
    Transmitter A Blackbox appliance that captures and transmits a video from a source PC. A transmitter is also referred to as a server or encoder.
    Work Space A collection of views containing tiles
    Video Source A source of input for a Transmitter, normally the DVI or Display Port output of a PC.
    Video Wall A video wall is a special multi-monitor setup that consists of multiple computer monitors, video projectors, or television sets tiled together contiguously or overlapped in order to form one large screen. Typical display technologies include LCD panels, Direct View LED arrays, blended projection screens, Laser Phosphor Displays, and rear projection cubes.
    Tile A instance of a connection in a given position within a workspace

    1.3.2 SDKVM OSD

    The OSD provides the main user interface for the SDKVM. The SDKVM is a multi-user system and each user can provision and manage a range of connections and workspaces.

    The OSD communicates with the workspaceserver through a REST API and MQTT messages to launch workspaces and connections. The status of these connections and workspaces are returned to the SDKVM either as MQTT messages or through polling the workspaceserver's REST API.

    1.5.1 Diagnostics Log Export using the OSD

    The OSD provides a diagnostic export button in the System Settings Tab. This will create an encoded DeskvueDiagnostics.log in an attched thumb drive or (if the no thumb drive is detected) in the /home/snuc/SDKVM/diagnostics directory.

    1.5.1.1 Creating the DeskvueDiagnostics.log file

    The OSD calls /home/snuc/SDKVM/scripts/diagnostics.sh passing the desired filename and path to the script. The script gathers relevant diagnostic files and encodes these using AES-256 with a predefined key.

    A utility script called /home/snuc/SDKVM/scripts/decode_diagnostics.sh can be used to decode the diagnostics.

    1.5.2 Diagnostics Log Export using Boxilla

    Proposal

    • We add a new get diagnostics endpoint for Boxilla. 
    https://IP:8888/control/diags/

    The SDKVM REST API endpoint calls:

     /home/snuc/SDKVM/scripts/diagnostics.sh /home/snuc/SDKVM/diagnostics/DeskvuewDiagnostics.log

    This command completes in under a second. It will create the diagnostic file in the provided path. We then embed the resultant .log file in the REST response.

    If the diagnostics directory does not exist it should be created.

    Boxilla processes the returned response and and stores the retrieved DeskvuewDiagnostics.log for retrieval by the user or a technician.

    1.5.2.1 Impacts

    • New GET REST API endpoint for diagnostics. - see POST Run Appliance Diagnostics section in the device rest api document.
    • Integrate REST API endpoint with /home/snuc/SDKVM/scripts/diagnostics.sh and retrieve the resultant file from /home/snuc/SDKVM/diagnostics/DeskvuewDiagnostics.log. This is the prefered location but any path and any filename can be passed to /home/snuc/SDKVM/scripts/diagnostics.sh.
    • Attach the contents of DeskvuewDiagnostics.log to the response and return to Boxilla.
    • Update REST API document to reflect the new interface and new behaviour for the SDKVM.

    1.5.3 Diagnostics file export and view using Boxilla

    There is a technician level feature on Boxilla that allows a technical support person to view diagnostic information through a password protected interface. The SDKVM will also support this interface.

    This uses the endpoint:

    https://IP:8888/control/diags/files

    This node returns the list of diagnostics files by first calling ~/SDKVM/scripts/diagnostic_files.sh and then creating a summary of the directory contents of ~/SDKVM/diagnostics. The technician can then request an individual file by specifying an index:

    https://IP:8888/control/diags/?file_id=2

    1.5.3.1 Impacts

    • Use existing endpoints for diagnostics files see section "GET List of Diagnostic Files" in the device rest api document.
    • Integrate existing REST API endpoint with /home/snuc/SDKVM/scripts/diagnostic_files.sh script.
    • This script will generate a list of files in /home/snuc/SDKVM/diagnostics/. The REST API should capture the list of files and prenet these back to the Boxilla request.
    • Retrieve the requested file for Boxilla using the file ID end point in response to "GET Nth Diagnostic File Content" as defined in the rest api document.

    1.5.10 Authentication of User Login

    In a SDKVM system managed by Boxilla a user login must be authenticated by Boxilla. The process described below only applies to a managed system.

    From the above diagram,

    • The OSD checks if it is in managed mode.
      • If the SDKVM is not manged the OSD uses the existing mechanism to verify a user against the credentials in user.json
      • If the SDKVM is managed then the OSD will follow the Boxilla Authentication process outlined below:

    1.5.10.1 Boxilla Authentication Process for Managed Systems

    For this process the OSD will use a utility called the STATS_CLI which is located at /usr/local/bin/stats_cli. This utility allows various commands to be sent to the Boxilla Management System which manages the SDKVM. If a user is successfully authenticated then the SDKVM sysytem will update the user details locally by writing/updating the file systems in the JSON files. The JSON files include users.json, connections.json, and workspaces JSON files in the user directory structure.

    1.5.10.1.0.1 Login Process Flow for Managed Systems
    • The User enters the login credentials on the OSD login screen.
    • The OSD will kill any existing instances of "/usr/local/bin/stats_cli". This is required because any existing instances could be as a result of previously failed login attempts/

    killall /usr/local/bin/stats_cli
    - OSD then passes the user credentials (username and password) to the Boxilla management system using the STATS_CLI application program which is available in /usr/local/bin/stats_restapi_cli. 

    /usr/local/bin/stats_cli login username password
    • The OSD should then start a 30 second timer (this should be a configurable setting in /home/snuc/SDKVM/users/system_level_preferences/system_preference.json and should default to 30 seconds) so that it can timeout if no response is received from the Boxilla Management System within that period of time.
    • The OSD will display a progress or busy indicator to indicate that the authentication process has started.
    • The STATS_CLI program launched by the OSD will return an authentication message back to OSD through MQTT interface (see details below).
    • If the authentication process ends in success then the OSD will:
      • Stop the 30 second timer.
      • Disable the progress or busy indicator.
      • Move to the OSD main screen.
      • Write a syslog entry indicating that the defined user has successfully logged in.
      • Update the system_level_preferences.json file with the logged in user and logged in user id.
    • If the authentication process ends in failure then the OSD will:
      • Stop the 30 second timer
      • Disable the progress or busy indicator.
      • Display an error message the user on the OSD. "Authentication Failed. Please try again."
      • Write a syslog entry indicating that the reason for the failure, the user name and any additional details returned in the MQTT message.
    • If the authentication process ends in failure due to a time out then the OSD will:
      • Disable the progress or busy indicator.
      • Display an error message to the user on the OSD. "Authentication Failed due to timeout. Please try again."
      • Write a syslog entry indicating that the authentication failed due to a lack of response from Boxilla.
      • Send an alert indicating that a login timeout has occurred: 
         /usr/local/bin/stats_cli timeout username
    1.5.10.1.0.2 MQTT messages required for Authentication

    The OSD should subscribe to the following MQTT endpoints:

    MQTT Topic:  topic  blackbox/sdkvm/deskvue/status/osd_sdkvm/webservice/authentication

    This endpoint will return a message indicating success or failure. The login_status provides the primary indication of success or failure.

    {"login_type" : "local", "login_status" : "success", "user_name" : " ", "details" : "Boxilla return details" }

    MQTT Topic:  topic  blackbox/sdkvm/deskvue/status/osd_sdkvm/webservice/error

    This endpoint will provide error details for a malformed request to the stats_cli.

    {"error_details" : "unknown command", "command_requested" : "" , "parameters_passed" : ""
}
    1.5.10.1.0.3 Logout Process Flow for Managed Systems
    • A user selects the logout option on the OSD.
    • The OSD will send a logout notification to Boxilla using the STATS_CLI application program which is available in /usr/local/bin/stats_restapi_cli. 
    /usr/local/bin/stats_cli logout username
    • The OSD should then start a 30 second timer (this should be a configurable setting in /home/snuc/SDKVM/users/system_level_preferences/system_preference.json and should default to 30 seconds) so that it can timeout if no response is received from the Boxilla Management System within that period of time.
    • The OSD will display a progress or busy indicator to indicate that the logout process has started.
    • The STATS_CLI program launched by the OSD will return an authentication message back to OSD through MQTT interface.
    • If the logout process ends in success then the OSD will:
      • Stop the 30 second timer.
      • Disable the progress or busy indicator.
      • Move to the OSD login screen.
      • Write a syslog entry indicating that the defined user has successfully logged out.
    • If the logout process ends in failure then the OSD will:
      • Stop the 30 second timer
      • Disable the progress or busy indicator.
      • Display an error message the user on the OSD. "Logout Failed. Please try again."
      • Write a syslog entry indicating that the reason for the failure, the user name and any additional details returned in the MQTT message.
    • If the logout process ends in failure due to a time out then the OSD will:
      • Disable the progress or busy indicator.
      • Display an error message to the user on the OSD. "Logout Failed due to timeout. Please try again."
      • Write a syslog entry indicating that the authentication failed due to a lack of response from Boxilla.
      • Move to the OSD login screen. ( I assume we would not want the failure to communicate with Boxilla from preventing us from logging out.)
      • Send an alert indicating that a login timeout has occurred: 
        /usr/local/bin/stats_cli logout timeout username
    1.5.10.1.1 MQTT messages required for Logout

    The OSD should subscribe to the following MQTT endpoints:

    MQTT Topic:  topic  blackbox/sdkvm/deskvue/status/osd_sdkvm/webservice/logout

    This endpoint will return a message on success or failure. This message can be used as confirmation that the command was processed.

    {"logout" : "local",  "details" : "Boxilla return details" }

    This endpoint will provide error details for a malformed request to the stats_cli.

    MQTT Topic:  topic  blackbox/sdkvm/deskvue/status/osd_sdkvm/webservice/error 

    {"error_details" : "unknown command", "command_requested" : "" , "parameters_passed" : ""
}

    1.5.10.2 Role of the stats_cli in the Login and Logout Process

    1. The Loginwindow will call stats_cli once user enters the username and password.
    2. The OSD starts a timer and waits for a response from the stats_cli. This response will be in the form of an mqtt message.
    3. The stats cli will call the "User Login Restapi" to authenticate the user. It will also receive an updated user json database from Boxilla for the requested user if the authentication process is successful.
    4. The Stats Cli will update the users, connection and workspaces json files in sdkvm directoy structure if an update is required using the data received from Boxilla and also update the requested username and user id in system_level_settings.json on user login success
    5. The Stats Cli calls OSD Manual Login Alert REST API based on the user login status(success/failed).
    6. The Stats Cli calls Auth Stats Restapi when the Boxilla is offline or based on the user login status(success/failed).
    7. The Stats Cli calls KVM user login status restapi to send user login status to the Boxilla after user authentication is done.
    8. The stats Cli send an mqtt message with the authentication status to the OSD.
    9. Timer sends the timeout message to Loginwindow if it doesnt receives any message from the Stats Cli within the time period.
    10. OSD will call Stats Cli along with the arguments timeout and username.
    11. Stats Cli will call OSD Manual Login alert REST API for Timeout.

    1.10 Discovery

    1.11 Discovery Process Flow

    When appliances are shipped they are configured with a default IP address. The discovery process allows a Boxilla to discover the device without knowing its actual IP address. Once discovered the user has the option of setting the IP of the unit by using the discovery protocol.

    1. When, in the Idle state, an appliance receives a discover request, the discovery appliance state machine captures Boxilla's MAC Address and IP Address from the discovery pdu and stores these values in the appliance datastore It then sets the default route for eth0 to 0.0.0.0, generates a discover response message and transitions to the Discovered state.

    2. Once Boxill has received the discovery resonse it will list it in its table of devices and provide an option for the user to set the IP address of the unit. If the user choses to set the IP address then a Set_IP protocol message is sent to configure the appliance or SDKVMs network settings e.g. IP Address, Netmask and Gateway. The Set_IP protocol message is only processed when the discover state machine is in the Discovered state. Following receipt of the Set_IP protocol message the discovery appliance binary updates the local network configuration and transitions to the Configuring state.

    3. Once the network settings have been successfully applied, the discovery appliance generates a Set_IP response message with the status set to Success. It then transitions to the Configured state.

    4. If the discovery appliance binary was unable to apply the network settings, the binary generates a Set_IP response message with the status set to Failed. It then reverts to the Discovered state.

    5. The discovery protocol does not generate explicit Manage protocol messages to transition appliances into the Managed state; this transition occurs as a result of the Audit request received from Boxilla. Appliances transition from the Discovered state to the Managed state following receipt of the first Audit request.

    6. Appliances may also become unmanaged, on the appliance this is achieved via the SSH interface whereby the unit is restored to factory defaults. This causes all discovery persistent data to be removed, forcing the unit to revert to the Idle state. On the SDKVM the system is unmanaged through the REST API.

    Note: On the SDKVM the webservice runs on bootup. This differs from the appliance where it only runs if the system is managed.

    1.11.1 Starting discovery on the SDKVM

    • The discovery binary exists in /usr/local/bin
    • The discovery service is managed by a systemd user service
    • The service is managed and started by /usr/bin/startdiscovery.sh
    • The configuration service file for restart_discovery exists in ~/.config/systemd/user/discovery.service.
    • To enable discovery as a service we use: systemctl --user enable discovery
    • On boot up the discovery service is started in the autostart.sh script in ~/.config/openbox. This script is called by the window manager.

    1.14 Modes of Operation

    1.14.1 Managed and Unmanaged Mode

    The SDKVM system can operate in either managed or unmanaged mode. In managed mode the configuration (management) of the system is controlled by our Boxilla management platform. Boxilla is responsible for creating users, administering permissions, provisioning connections and creating workspaces.

    When a SDKVM is managed then the options to locally create Users and Connections and to configure device settings are disabled. All modifications must be done on Boxilla and downloaded to the SDKVM system.

    In unmanaged mode a user with administrative access can create connections and assign them to workspaces. A user with administrative access should be able to allocate connections and workspaces to regular (local) users.

    1.14.1.1 Managing a SDKVM

    A SDKVM becomes managed through the process of discovery and management by Boxilla. When Boxilla has managed a SDKVM a two files will appear in:

        /home/snuc/SDKVM/boxilla
    These files are as follows:

    • The discovery file: /home/snuc/SDKVM/boxilla/discovery

    MGR=0:8c:10:1c:c6:d4
STATE=Discovered
DEBUG=OFF
    and the Boxilla Manager IP file: /home/snuc/SDKVM/boxilla/mgr_ip

    MGR_IP=10.211.129.3
    The OSD should use the presence of these files to determine if it is operating in managed or unmanaged mode. It should read the Boxilla mac address from the discovery file and use this as the Boxilla ID. The OSD should indicate clearly if the system is operating in managed or unmanaged mode.

    Boxilla can choose to unmanage a device. In the process on unmanaging a device it will remove the users.json, discovery and mgr_ip files and also remove any existing user data structure and associated workspaces.

    If having been previously managed the OSD detects that these files and directories have been removed it should create a new empty users directory and a new users.json file.
