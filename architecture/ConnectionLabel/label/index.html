    6. Overview

    The Connection Label feature gives users the ability to see their connection name overlaid on top of their connection. The feature uses a hardware based overlay mechanism to display the connection name in an 840x40 pixel sized box at different locations on the screen. In Release 6.3.1, the feature can only be configured via the OSD, but for future releases, this configuration will also be available via Boxilla. The feature can be enabled on all receivers bar the Emerald SE Dual-Head (No space in the FPGA for the logic). The following elements of the Connection Label can be configured on an RX:

    • Enable / disable the feature
    • Set the time for the Label to be displayed – 2 Options
    • 5 seconds
    • Always On
    • Set the position of the Label
    • Top Centre
    • Top right
    • Bottom Centre
    • Bottom Right

    Once the Feature is enabled on the RX, then a hotkey sequence (Hotkey + 'l' or 'L') can also be used to display the Label / dismiss the label.

    The basic steps for overlaying a Label on top of a connection are as follows:

    1. Enable the Connection Label feature via the OSD
    2. When a connection is being launched, add the command to write the label to the framebuffer to the 'startapp' file
    3. When the 'startapp' file is executed, the label gets written to the framebuffer
    4. When the connection is established, point the overlay logic to the framebuffer that contains the label and turn on the overlay
    5. If the Label is set to Timeout after 5s, then start a timer and turn off the Label after 5s
    6. Once the connection is terminated, turn off the overlay.

    NOTES:

    In the detail design below, all the main functions will first check that Connection Label is Enabled (read from the Sql DB) before any Label functionality is called.

    7. Design Detail

    7.1 Connection Label Configuration

    The Connection Label Configuration differs depending on whether the system is Managed or Unmanaged.

    • Managed Mode Configuration
      • The Enable / Position / Timeout for the Connection Label can only be configured via the OSD on the RX and as such is specific to that Receiver
      • The Label Colours can Only be configured via Boxilla - This value is configured on a per connection basis.

    • Unmanaged Mode Configuration
      • The Enable / Position / Timeout / Colours for the connection label are configured via the OSDon the reveiver
    sequenceDiagram participant User participant OSD participant SQL DB User->>OSD: Enable Connection Label User->>OSD: Select Label Timeout / Position User->>OSD: Select Label Colours User->>SQL DB: Write Connection Label settings (Enable/Timeout/Position/Colours)

    The following sequence diagram details how the Connection Manager gets all the required information for running the connection_label application to display the Connection Label.

    7.3 Creating a Framebuffer for the Label

    The framebuffer is created in the device tree of all receivers. The base address depends on whether it is a 2K or 4K receiver.

    The details of the framebuffers are as follows:

    • 2K

      • framebuffer@31401000{
      • compatible = "simple-framebuffer";
      • reg = \<0x31401000 0x12C00>;
      • width = \<0x000001E0>;
      • height = \<0x00000028>;
      • stride = \<0x00000780>;
      • format = "a8r8g8b8";

    • 4K

      • fbuf@7A050000{
      • compatible = "simple-framebuffer";
      • reg = \<0x00000000 0x7A050000 0x00000000 0x25800>;
      • width = \<0x000001E0>;
      • height = \<0x00000028>;
      • stride = \<0x00000780>;
      • format = "a8r8g8b8";

    The Framebuffer (fb1) is taken as an input to the application that is creating the Label – The application then writes the Label into this framebuffer. The Overlay will then display the contents of this frame buffer as required.

    7.4 Generating the Label

    The label is generated by a QT based application which writes the 'connection name' into a box of fixed size (Width 480 px by Height 40 px).

    7.4.1 Arguments:

    This application requires few mandatory arguments:

    • -nomouse: provide no mouse cursor displayed when the application is running;
    • -qws -display linuxfb:/dev/fb1: this arguments provide application will be placed in correct frame buffer;
    • -n: the argument which allow as to capture "connection name" which should follow immediately after this argument.

    The binary to create the label is called 'connection_label' and is located in /usr/bin on the Receivers.

    The code for this application is located in 'Applications/Decoder/ConnectionLabel

    7.4.2 A usage example:

    /usr/local/connection_label -nomouse -qws -display linuxfb:/dev/fb1 -n Default_TX

    7.4.3 Application flow:

    During initialization connection_label will parse the arguments searching for connection name. If connection name is not found then it will remain an empty string. Next, the mainwindow class initialization is called. The next step is to pass connection name to mainwindow class - this is accomplished by calling setConnectionName(connectionName) function from mainwindow class. The setConnectionName(connectionName) function will pass connectionName variable to label. When this is completed the mainwindow is shown to user.

    The mainwindow class ui form has predefined size of 480 px by 40 px and has only one label placed which is occupying all available space of ui form.

    Within mainwindow initialization the title bar, the close, minimize and help buttons as well as window border are removed to prevent user from moving around display or closing the connection label application.

    The colour of mainwindow background is set to RGBA (151, 253, 255,255).

    The mainwindow coordinates are then set to H0, W0 in required framebuffer.

    7.5 Writing the Label to the Framebuffer

    The application to create the label (/usr/bin/connection_label) is called at the start of the startapp.sh script. The startapp.sh script is created and executed every time a connection is being launched.

    The full sequence is as follows:

    • Connection is launched (via OSD/ Rest etc.)
    • If the connection is a TX or VM Direct connection, then Connection Labels are applicable
    • If ConnectionLabel is Enabled (check SQL DB)
      • /usr/bin/connection_label –nomouse –qws –display linuxfb:/dev/fb1 –n 'connection_name' & is written to the start of the /tmp/startapp.sh file
    • The rest of the contents of /tmp/startapp.sh are filled in as normal
    • The /tmp/startapp.sh file is executed, resulting in the connection being setup and the label being written to fb1.

    The result is that (if the connection label feature is enabled) the connection name will be written to /dev/fb1 – When the overlay is being setup, it will point to /dev/fb1 to pick up the Label.

    7.6 Configuring the Overlay to display the Label

    The details of how to configure the 2K and 4K overlays is in this section. The overlay only gets configured/re-configured when a connection is setup or when a resolution changes. The Label hotkey does not reconfigure the Label – It assumes it is correctly configured and just enables/disables it as required.

    7.6.1 4K Overlay Configuration

    The 4K overlay is done in the FPGA logic. It is configured in software using the following register set:

    • HW_OSD_SCALE_FACTOR_REG_OFFSET - 1300
      • The bottom 2 bits of this register are used to scale the overlay (The rest are unused) this allows us to scale the overlay so that it is visible at high resolutions.
        • For widths of 1920 or less, we set the value to 0x00 for No scaling.
        • For widths between 1920 and 3840, we set the value to 1, which scales the overlay by a factor of 2 (i.e. doubles the size)
        • For widths over 3840, we set the value to 2, which scales the overlay by a factor of 3 (i.e. triples the size)
    • HW_OSD_ENABLE_REG_OFFSET - 1304
      • The bottom bit is the enable/disable bit (The rest are unused). To Enable the overlay, the bottom bit is set to 1
    • HW_OSD_FLUSH_REG_OFFSET - 1308
      • This is a placeholder register (I.e. functionality not currently supported), so the contents are set to 0
    • HW_OSD_READ_BURST_SIZE_REG_OFFSET - 1314
      • As per the AFS, this should be left at the default value of 0x10
    • HW_OSD_READ_BURST_PER_FRAME_REG_OFFSET - 1318
      • This value can be calculated using the following formula: - (width * height *4) /256. In our case (label fixed at 480x40), this equates to: 0x12C
    • HW_OSD_BASE_ADDRESS_REG_OFFSET - 131C
      • This register is configured with the address of the Label – In our case, this is the address of the framebuffer(fb1) that the label is written into – The value is: 0x7A050000
    • HW_OSD_TEST_MODE_REG_OFFSET - 1320
      • This is for hardware test, so we set it to 0
    • HW_OSD_X_OFFSET_REG_OFFSET - 1324
    • HW_OSD_Y_OFFSET_REG_OFFSET - 1328
      • The X and Y offset registers are the starting coordinates where the label gets displayed. This is calculated using the following inputs:
        • The selected position of the label (top/bottom centre/right)
        • The resolution of the display
        • The size of the label (480x40 in this case)
    • HW_OSD_WIDTH_REG_OFFSET - 132C
      • This is set to the Label width – 0x1E0 (480 decimal)
    • HW_OSD_HEIGHT_REG_OFFSET - 1330
      • This is set to the label height – 0x28 (40 decimal)
    • HW_OSD_BACKGROUND_FILL_REG_OFFSET - 1334
      • This allows us to fill the rest of the screen (outside the label overlay) with black. We want to leave the background as it is, so we set this value to 0x0

    7.6.2 2K Overlay Configuration

    The 2K overlay is done in the Fpga logic. The configuration of the Label is done every time a connection is made and also every time the resolution changes (in the case of RX-TX connections). It is configured by software using the following register set. The overlay can be done for either head (on a 2K receiver), but we only do the overlay for the first head.

    Overlay base address for H0 (0x800C8380)

    Overlay base address for H1 (0x800C8390)

    Specific Registers:

    • H0_OVERLAY_CONTROL_REG - 0x800C8380 (8 bit)
    • The Control Register has the following details:
      • Bit 7 Enable bit
      • Bit 6 Dimension update bit
      • Bits 0 – 5 Test Character
    • Once the configuration of the Label is complete, Bit 6 of the Control Register will be set to update the config values.
    • Once everything is in place and displaying the label is required, Bit 7 of the Control Register is set.
    • To turn off the Label, Bit 7 of the Control Register will be set to 0
    • H0_OVERLAY_BASE_ADDR_REG - 0x800C8384 (32 bit)
    • This register is configured with the address of the Label – In our case, this is the address of the framebuffer(fb1) that the label is written into – The value is: 0x31401000
    • H0_OVERLAY_HORIZONTAL_OFFSET - 0x800c8388 (16 bit)
    • H0_OVERLAY_VERTICAL_OFFSET - 0x800C838A (16 bit)
    • The Horizontal and Vertical offset registers are the starting coordinates where the label gets displayed. This is calculated using the following inputs:
      • The selected position of the label (top/bottom centre/right)
      • The resolution of the display
      • The size of the label (480x40 in this case)
    • H0_OVERLAY_HORIZONTAL_SIZE - 0x800C838C (16 bit)
    • This is set to the Label width – 0x1E0 (480 decimal)
    • HO_OVERLAY_VERTICAL_SIZE - 0x800C838E (16 bit)
    • This is set to the label height – 0x28 (40 decimal)

    7.7 Labels for different Connection Types

    In Release 6.3.1, Connection labels are supported on RX-TX connections and VM RDP Direct connections only.

    7.8 Turning on the Label after a connection is launched

    Once an RX-TX or VM RDP connection has been setup, the Connection Label will be configured and turned on (assuming it has been enabled). This is done immediately after the connection has been setup in order to ensure that the Label appears immediately after the connection appears on the monitor. Exactly where this is done will depend on the type of connection.

    • RX-TX Connections
    • In this case, the Label is turned on in the function "bb_connection_setup_hardware()"
    • 2K VM RDP Connections
    • In this case, the Label is turned on in the function: "rfx_context_update"
    • 4K RDP Connections
    • TBD

    7.9 Timing out the Label

    There are 2 timing options available in release 6.3.1 for Connection Labels – There are:

    • Constantly On
    • Timeout after 5sec

    These options are configured via the OSD and the resultant selection stored in the Sql DB.

    The 'Constantly On' option works as the name indicates – The Label is turned on when a connection is launched and only turned off when the connection is terminated. The only thing that can change this is when the hotkey to turn on/off the Label is used.

    The 'timeout after 5sec' option is handled as follows:-

    Once the connection is launched, a timer is started. The timer is checked in the 'interval tasks' function (which is called every second) – once the timer has expired (determined by comparing the difference between the original timer and the current time), then the connection label is turned off. Turning off is done by the system call: - 'display_layer_cli –label-disable' (see section: 7.12 below)

    7.10 Turning off the Label when connection terminates

    Once a connection terminates, the Connection Label needs to be turned off (as we don't want to overlay it on top of the OSD screen) and also the application that writes the connection Label to the frame buffer needs to be stopped.

    The connection label itself is turned off by calling the display_layer_cli (see section: 7.12). This is done via the startgui.sh script.

    The application that writes the label to the frame buffer (/usr/bin/connection_label) is constantly running while the connection is up due to the fact that exiting the application also flushes the framebuffer. Once the connection terminates, this application also needs to be terminated. This is done in startgui.sh using: - 'killall connection_label'

    7.11 Enabling/disabling the Label using Hotkeys

    The extra Hotley (Hotley + 'l'/'L') has been configured in the connection_control application. These are handled in the same manner as all existing hotleys. When the Label hotkey fires, connection _control enables/disables the Label using the following command (See section: 7.12):

    display_layer_cli –label-on-off

    7.12 Reconfiguring the label after a resolution change

    In an RX-TX connection, the resolution on the target device may change while the connection is in place. In this case, a new resolution will be received by the Rx and the display will be reconfigured with this new resolution.

    If the Connection Label feature is enabled, then the Overlay needs to be reconfigured with this new resolution in order that the Label gets displayed correctly. This reconfiguration of the label is done as per section: 7.4 above. The reconfiguration is done even if the label is currently turned off such that if the Label gets turned on via the hotkey, it is properly configured.

    7.13 Additional functionality added to the Display Layer

    The following additional functionality has been added to the Display Layer to configure/enable/Disable the Connection Label:

    • Display_driver_common_configure_connection_label()
    • This function configures the Overlay for the connection label. It does this for either the 4K or 2K overlay. It configures all the required registers (see section: 7.4), but does not turn on/off the label.
    • Display_driver_common_turn_on_connection_label()
    • This function turns ON the Overlay for either 2K or 4K.
    • Display_driver_common_turn_off_connection_label()
    • This function turns OFF the overlay for either 2K or 4K
    • Display_driver_common_turn_on_off_connection_label()
    • This function turns On the Label if it is already Off and turns it Off if it is already On – It does this for 2K and 4K
    • Display_driver_common_connection_label_on()
    • This function returns TRUE if the Label is currently On – It does this for 2K and 4K

    7.14 Changes to Display Layer CLI

    2 new options added to the display layer CLI. These are as follows:

    • Label-disable (display_layer_cli –label-disable)
    • This is used to turn OFF the Overlay in order to no longer display the Connection Label. It is called in startgui.sh when a connection is exiting.
    • Label-on-off (display_layer_cli – label-on-off
    • This is used to turn ON the Overlay if it is currently OFF and turn OFF the Overlay if it is currently ON. This is used for the Label Hotkey implementation – When the Label hotkey (hotkey + 'l' or 'L') is triggered, it will turn on the label (if it is currently off) and turn off the label (if it is currently on).

    7.15 API Changes or changes to external interfaces

    There are no API changes associated with this feature for the initial version, as it is only possible to configure it on the OSD. There is a Future requirement however that requires that this feature be configurable on Boxilla – when this happens, there will need to be API changes to handle the new configuration.

    7.16 Security Considerations

    This feature has no security impacts.

    7.17 Test and Validation

    The following test cases are required:

    • Validate that the Connection Label works on 4 different connection types:
    • 2K RX-TX
    • 4K RX-TX
    • 2k RDP VM
    • 4K RDP VM
    • Validate the different configurations for all 4 connection types
    • Enable/Disable
    • Timeout
    • Label location
    • Validate that the Label times out correctly
    • Validate that a Res change (Rx-Tx only) resizes the Label and turns is on/off based on whether it was on/off before the Res change.
    • Validate that the Label Hotkey functions correctly.
    • Validate that the Label is scaled appropriately on 4K
    • Validate that the Label disappears when a connection is terminated
    • Validate that the label is visible at all resolutions

    7.18 Database and Configuration

    The Label configuration is stored in the Sql DB on the Rx. A new table has been added to store the values as follows:

    CREATE TABLE ConnectionLabel (LabelEnable INT, LabelTimeout INT, LabelPosition TEXT);

    INSERT INTO ConnectionLabel VALUES(1,0,'Bottom Centre');

    Notes:

    LabelEnable has 2 possible values – 1/0 (On/Off)

    LabelTimeout currently has 2 possible values – 0/5 (ConstantlyOn / Timeout after 5secs

    LabelPosition currently has 4 possible values – 'Bottom Centre', 'Bottom Right' 'Top Centre', 'Top Right'

    Default values:

    LabelEnable – 0 (Off)

    LabelTimeout – 5 (Timeout after 5 secs)

    LabelPosition – 'Bottom Centre'

    Database Version:

    The addition of the connectionLabel to the SQL DB requires the DB to update to version 1.6

    7.19 Performance and user experience

    The addition of the connection Label should have no noticeable impact on connection setup/switch/termination times.

    8. Connection Label Colours

    Adding the ability to Configure the Connection Label Colours is being added as an Appliance / Boxilla feature. The feature will be added in Appliance version 6.7.0 and Boxilla version 4.8 The details on how this can be configured will differ based on whether the system is Managed or Unmanaged

    8.1 Configuring the Connection Label - Enable / Timeout / Location

    There will be no changes required to how the existing aspects of the Connection Label feature are configured. Regardless of whether the Receiver is Managed or Unmanaged, the configuration will be done via the OSD. Being able to configure the Enable/Timeout/Location is an Optional requirement for Boxilla, so my assumptiuon is that it will not happen in the 4.8 releease. Adding the REST API to allow Boxilla to configure these attributes on the appliance is being considered for the 6.7.0 appliance release however (My recommendation was not to include it, as it requires input from the Boxilla team and they have limited bandwidth at the moment). From a usage point of view, the values will be written to the SQL DB on the Receiver regardless of who configures them - Based on this, there will be no changes to how these values are read/used when/if configuring via Boxilla happens.

    8.2 Generic requirements for Connection Label Colours

    The Connection Label Colour Requirements are as follows:

    • The values will be RGB values (3 * 8 bit values)
    • RGB values need to be specified for both the Background and Text colour.
    • When configuring the Colours, the administrator needs to be given a preview of the resultant Connection Label to ensure it is what is required
    • In an Unmanaged system, the Label Colours will be configured via the OSD - These colours will apply to all connections launched from this RX.
    • On a managed system, the label colours will be configured via Boxilla on a per connection basis

    8.3 Database and Configuration Updates for Connection Label Colours

    Adding Colours to the Connection Label will require additions to the ConnectionLabel Table in the SQL DB. The additions are the RGB values for the Background and text colours - These will default to the current Background and test colours (black test on a sky blue background). The changes will be appended to the existing table as follows:

    • ALTER TABLE ConnectionLabel ADD COLUMN Background_R INT;
    • update ConnectionLabel set Background_R="151";
    • ALTER TABLE ConnectionLabel ADD COLUMN Background_G INT;
    • update ConnectionLabel set Background_G="253";
    • ALTER TABLE ConnectionLabel ADD COLUMN Background_B INT;
    • update ConnectionLabel set Background_B="255";
    • ALTER TABLE ConnectionLabel ADD COLUMN Text_R INT;
    • update ConnectionLabel set Text_R="0";
    • ALTER TABLE ConnectionLabel ADD COLUMN Text_G INT;
    • update ConnectionLabel set Text_G="0";
    • ALTER TABLE ConnectionLabel ADD COLUMN Text_B INT;
    • update ConnectionLabel set Text_B="0";

    Database Version:

    The update of the connectionLabel to the SQL DB requires the DB to update to version 1.9

    8.4 Configuring Connection Label Colours in Unmanaged mode

    In unmanaged mode, the functionality will be configured as follows:-

    • The Label colours will be configured via the OSD.
    • To facilita this, a new 'Colors' tab will be added to the existing Connection Label tab on the OSD.
    • This 'Colours' tab will become active when the Connection Label Enable is selected.
    • This Colours tab will provide 'sliders' to allow the user (admin/power user) to configure the RGB values for both the Background and text colours
    • A preview of the resultant colours will be made available to the user.
    • Once the values are saved, they will be written to the Connection Label Table in the SQL DB
      • The existing values in this table will not change
      • The colours will be attached to the end of the table
      • The default for the colours are the existing colours (i.e black test on a sky blue gackground)
    • In the Unmanaged scenario, the Connection Label Colours will be on an RX basis - i.e any connection launched from this RX will use these colours.

    8.5 Configuring Connection Label Colours in Managed mode

    In managed mode, the functionality will be configured as follows:-

    • The Label colours will be configured via Boxilla.
    • The Label Colours will be configured on a Per connection basis
      • The ability to configure global 'default-colours' will be provided to the Boxilla user
      • When creating a connection, the user will have the ability to select either
        • The 'default-colours'
        • To create a customer colour combination for this connection
      • Creating a custom colours on Boxilla will be done in a similar way to how it is done on an Unmanaged RX (for consistency)
    • The RGB values for the colours will be stored in the XML file as an additional argument/qrguments to a connection
    • Any existing connections will be migrated by Boxilla to this new scheme

    NOTE: In managed mode, the facility to configure Connection Label Colours will NOT be available via the OSD

    8.6 Required changes to the XML file

    The change to the XML file needs to be as short as possible. Based on this requirement, the following addition to a connection in the XML filehas been agreed with the Boxilla team.

    labelColours='255,0,100,20,0,255'

    The first 3 values will be the RGB values for the Background colour and the second 3 values will be the RGB values for the Text colour.

    8.7 Turning On the Connection Label with different Colours

    There are 2 aspects to this work:

    8.7.1 Updates to the connection_label binary to allow the caller (Connection Manager) to pass the Colours as parameters

    The connection_label binary currently has the following parameters:

    • Framebuffere reference - This is the framebuffer that the connection Label needs to be written to
    • Connection Name - This is the text to write into the connection label

    The addition of Connection Label Colours will require the following 6 extra arguments

    • Background R value
    • Bacgground G value
    • Background B value
    • Text R value
    • Text G value
    • Text B value

    The new format for this command is:

    • /usr/bin/connection_label -nomouse -qws -display linuxfb:/dev/fb2:tty=/dev/fb2 -n "Connection Name" -bc "255" "0" "255" -tc "0" "255" "100" &

    The changes to the connection label binary are to take in the Background and Text colours and to use these as the colours when generating the Connection Label.

    8.7.2 The Logic in the Connection Manager to figure out what Colours to pass to the connection_label binary

    The changes in the Connection Manager for this feature are to do the folloiwng:-

    • Figure out what colours to use for the Connection Label
    • Pass the colour values to the connection_label binary.
    • There are no changes required in terms of when / if to call this binary.
    • The logic will need to take into consideration the following:
      • In Unmanaged mode, the Colour values will be stored int he SQL DB
      • In Managed mode, the colours will be stored in the XML file on a per connection basis
      • In managed mode, the SQL DB will contain the default colours (will never change)
      • Currently the Connection Manager is NOT aware of whether the Reveiver is managed or not - The plan is to keep it like this.

    The required changes are as follows:

    • OSD / Rest-API (anywhere that reads the copnnection string from the XML file)

      • When the connection details are being read from the XML file, we now read the 'labelColours' as well
      • If no 'labelColours' exist in the XML file (as in the Unmanaged scenario), then we set the 'labelColours' string to 'No-specified-color'
      • An extra parameter (labelColours) is added to any MQTT request that relates to Launching / switching a connection. The value of this attribure will be either:
        • A string containing the Background and test RGB values (assuming that the specified connection has Label Colours associated with it)
        • A string containing: "No-specified-colour' - This indicates the the Subscriber of the message (Connection Manager) that there is NO associated LabelColour for this connection.

    • Connection Manager changes:-

      • Connection Manager reads the 'labelColours' string from the connection requests
        • If string contains 'No-Specified-colour', then it reads the Connection Label Colours (Value associated witht he specific RX) from the SQL DB
        • Otherwise, it uses the RGB values specified in the 'labelColours' parameter associated with the connection
      • In both cases, it calls the 'connection_label' binary with the required RGB values.
