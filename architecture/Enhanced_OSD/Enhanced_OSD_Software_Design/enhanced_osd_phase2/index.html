    Enhanced OSD Phase 2

    Note: Sections in RED need further work/ review with marketing etc.

    The enhanced OSD development was split into the following Phases:

    • Phase1: This was to create a state based connection management system that would handle all aspects of connections. Ultimately, the aim was to have a reliable and efficient system for Connections that would be used as the basis of a lot of new features moving forward. In terms of functionality, this Phase would not add any extra functionality/new behaviour to the system. This is detailed the previous sections of this document.
    • Phase2: This was to add in the concept of keeping the OSD alive at all times to allow seamless switching from the OSD to a connection and ultimately back to the OSD

    What is enhanced OSD Phase2:

    • Phase2 of the Enhanced OSD project adds the ability to leave the OSD running (albeit not being displayed) while any 2k or 4k connection is running
    • Having the OSD running continuously means that when a connection exits, then the OSD can be displayed immediately
      • It will only be immediate if the OSD resolution matches the connection resolution. If not, there will be a 1 or 2 sec delay (depends on the monitor) while the monitor locks on to the new resolution This is outside our control and is consistent with our competitors products.
    • Launching a connection with Phase1 and Phase2 combined gives us the ability to continue to display the OSD until the connection has established This allows for seamless witching from OSD to connection

    • The core work here is as follows:

      • Being able to switch to/from the OSD quickly and reliably as connections are launched and terminated
      • Keeping the OSD aware of what is happening in terms of connections, so that it can make behavioural decisions (Prior to this, the OSD never ran at the same time as a connection, so this did not need to be considered) on kbd/mouse, usb devices etc

    • The OSD_Manager (in conjunction with the Connection_Manager) will decide when we need to HIDE / SHOW the OSD.

    Enhanced OSD Phase2 Requirements:

    There are 2 core requirements and they are as follows:

    • The OSD should be displayed immediately on connection termination
    • Launching a connection (from the OSD or over Rest) requires the OSD to stay displayed (with an indication that a connection is being setup) until the connection is Up Once the connection is ready, we need to HIDE the OSD and display the connection (should be seamless)

    Enhanced OSD Phase2 Display/Hide the OSD

    One of the core parts of Phase2 is to be able to display the OSD or Hide the OSD while leaving the OSD (cloudgui) process running. The following details how this will be achieved.

    Terminology:

    • Launching/Relaunching the OSD This refers to running the cloudgui binary. It needs to be done on board bootup, due to certain user actions (OSD res change, kbd selection) and also if the cloudgui binary ever crashes.
    • HIDE_OSD For the Phase1 design, this term is used to ensure than there is no OSD process running (if there is it will be killed). For Phase2, this means to no longer Display the OSD i.e leave the process running, but no longer display it on the monitor attached to Head0. Hide / Hiding are used at different places in this document, but mean the same thing.
    • SHOW_OSD For Phase1 design, this term is used to run the launch the OSD i.e run the cloudgui process. For Phase2, this term will be used to Display the OSD. Show / Showing are used at different places in this document, but mean the same thing.

    Changes Required to Buffers/ Surfaces etc to allow the OSD to co-exist with all connection types (These changes will be made to the appropriate device trees)

    The current implementation on both 2K and 4K sees the OSD sharing the same buffer/surface as some of the connection clients. This was implemented in this way due to the fact that the OSD would never be running at the same time as the clients (see the logic in the startgui.sh script when the OSD exits, the client runs and vice versa)

    In this project, the core requirement is that we can run both the OSD and the clients at the same time (to facilitate quicker switching). To be able to facilitate this requirement, we need to ensure that the OSD and the different clients never use the same block of DDR. The new buffers/surfaces will be allocated in the Devicetrees and configured via the DisplayLayer.

    Scenarios that need to be considered when deciding if/how to Display the OSD:

    • Board Bootup
      • On board bootup, the OSD should be run. This needs to be managed by the OSD Manager. The steps to run the OSD are as follows:
        • Read the OSD Resolution from the DB and setup the framebuffer as required
        • Setup the Display for the required resolution
        • Launch the OSD
        • Setup and turn ON the Overlay to display the OSD (4K only)
    • Launching a connection via the OSD.
      • In this case, the OSD is displayed until the connection is ready. Once the OSD Manager received the CONNECTED message, it HIDEs the OSD. See details in section 16.3.4.2 below.
    • Launching a connection over Rest. Basically the same as launching a connection from the OSD The difference in this case is that an MQTT message is sent to the OSD to inform it that a connection is being launched.
    • Exiting a connection (Hotkey or via Rest)
      • In this case, the OSD needs to be displayed immediately See section: 16.3.4.3 below
    • Changing the OSD resolution/ changing kbd
      • This can only happen when the OSD is being displayed (obviously enough )
      • In this case, the updated resolution is written to the SQL DB and the OSD exits (after publishing an MQTT message). The OSD is then relaunched as per Board Bootup above.

    • Handling OSD Resolution of Auto

      • If the OSD resolution is set to Auto then once a connection terminates, the OSD needs to be displayed at the resolution of the connection. This was relatively simple in the 6.3.3 release (and before), as the OSD was reloaded every time a connection exited. In this project (Enhanced OSD), the problem is much more difficult. The following 2 scenarios need to be handled:

        • If the connection is launched and the connection resolution is different to the OSD resolution, the following will be done:
          • Once the connection is successful, the display resolution will be checked and compared to the OSD resolution. If they differ, then an MQTT_EXIT_FOR_RES_CHANGE request is sent to the OSD.
          • The OSD in turn will publish an MQTT RESOLUTION_CHANGE request before exiting gracefully
          • The osd_manager will process the Resolution_change request, and launch the OSD with the updated resolution (the OSD will run but not display until the connection terminates)
        • If the connection is launched and a resolution change happens some time later which results in the OSD resolution being different to the Connection resolution (This scenario can only happen for RX-TX connections), then the client will publish an MQTT_EXIT_FOR_RES_CHANGE message. The OSD will receive it, and the resulting steps will be as above.

      • In any of the scenarios above, if the OSD resolution of Auto is read from the DB, then the following needs to happen:

        • The actual Display resolution is read from the display and the following is done:
          • The value is written to the /etc/directfbrc file (e.g:- mode=800x600)
          • The display is already at the required resolution, so NO change needed.
          • The framebuffer needs to be setup with the new resolution (fbset fb /dev/fbx xres xxx yres yyy)
          • The OSD will need to be terminated and re-loaded to pick up the correct resolution (as mentioned above)

    • Do we reload the OSD when the connection terminates

      • This would be easier I think, but NOT the option we want, as the OSD wont be ready when the connection exits..
    • OSD Login Nothing to do, as the OSD does not need to Exit/Reload.
    • Auto-Connect Any user can be configured to auto-connect to a particular connection when the user logs in. If the user is set to auto-Login, then on board bootup, the specified user is Logged In and connected to the specified connection (without any user intervention)
      • Auto-connect will be handled as if a user has launched a connection from the OSD
    • Cloudgui crashes. See Section 15 above.

    Launching/Hiding/Showing the OSD:

    Launching the OSD

    The following steps are used to do the initial launch of the cloudgui process and any subsequent re-launches:

    • Reads the configured OSD resolution from the DB
    • Does an fbset to set the framebuffer to the correct res (Note: 2K / 4K will be using different frameBuffers).
    • If the Res is set to Auto:
      • Read the resolution from the Display
      • Call fbset with the resolution of the display (i.e the resolution that the connection left the display at)
    • Call display_layer to set the display to the correct resolution
    • Run the cloudgui binary

    Hiding the OSD

    NOTE: The OSD will only be hidden if there is a connection active. Based on this, the following table identifies how hiding the OSD is done in different scenarios:

    2K Scenarios
    Hiding the OSD due to an RX-TX connection Launching In this case, switching from Primary to Secondary surface will hide the OSD and display the connection.
    Hiding the OSD due to a VM connection Launching In this case, selecting the Primary surface for the VM connection (as opposed to the OSD Primary surface) will hide the OSD. This is done writing the frame buffer address of the VM Primary surface to register: 0x800c8330
    4K Scenarios
    Any type of connection - If the h/w Overlay is enabled and points to the base address of the OSD framebuffer: The OSD will be hidden in this case by disabling the H/W Overlay.

    Showing the OSD

    2K Scenarios
    2K RX-TX connection In this case, the secondary to primary switch functionality is used (The rx-tx connection uses the secondary surface and the OSD uses a primary surface). Currently done via: display_layer_cli primary 1280 1024 (for example)
    2K RDP VM connection In this case, switching to the OSD is done by selecting a different Primary surface (The OSD Primary surface). This is done writing the frame buffer address of the OSD to the following register: 0x800c8330
    4K Scenarios
    Any connection Type Showing the OSD is done by setting up the h/w Overlay to point to the OSD framebuffer and then enabling the h/w Overlay (details below)

    Details on Displaying/Hiding the OSD for 4K Receiver.

    • A new framebuffer will be created specifically for the OSD (FB2). This is to ensure that the OSD can run at the same time as any of the clients.
    • The HW Overlay is the mechanism to be used for displaying the OSD on a 4K RX. The principle will be the same as for the connectionLabel. When a connection is running, the H/W Overlay will (if it is enabled) be displaying the connectionLabel when the connection exits, the H/W Overlay will switch to displaying the OSD
    • The register set for the H/W Overlay is as follows:
      • HW_OSD_SCALE_FACTOR_REG_OFFSET - 1300
        • The bottom 2 bits of this register are used to scale the overlay (The rest are unused) . This allows us to scale the overlay so that it is visible at high resolutions.
      • HW_OSD_ENABLE_REG_OFFSET - 1304
        • The bottom bit is the enable/disable bit (The rest are unused). To Enable the overlay, the bottom bit is set to 1
      • HW_OSD_FLUSH_REG_OFFSET - 1308
        • This is a placeholder register (I.e. functionality not currently supported), so the contents are set to 0
      • HW_OSD_READ_BURST_SIZE_REG_OFFSET - 1314
        • As per the AFS, this should be left at the default value of 0x10
      • HW_OSD_READ_BURST_PER_FRAME_REG_OFFSET - 1318
        • This value can be calculated using the following formula: - (width * height *4) /256. In our case (label fixed at 480x40), this equates to: 0x12C
      • HW_OSD_BASE_ADDRESS_REG_OFFSET - 131C
      • HW_OSD_TEST_MODE_REG_OFFSET - 1320
        • This is for hardware test, so we set it to 0
      • HW_OSD_X_OFFSET_REG_OFFSET - 1324
      • HW_OSD_Y_OFFSET_REG_OFFSET - 1328
        • The X and Y offset registers are the starting coordinates where the label gets displayed. This is calculated using the following inputs:
        • The selected position of the label (top/bottom centre/right)
        • The resolution of the display
        • The size of the label (480x40 in this case)
      • HW_OSD_WIDTH_REG_OFFSET - 132C
      • HW_OSD_HEIGHT_REG_OFFSET - 1330
      • HW_OSD_BACKGROUND_FILL_REG_OFFSET - 1334
        • This allows us to fill the rest of the screen (outside the label overlay) with black. We want to leave the background as it is, so we set this value to 0x0

    Proposed Algorithm:

    The mechanism being used is the Overlay mechanism. This allows us to overlay any size image on top of the existing display. The overlay image can be positioned anywhere and can be scaled as required. The ability to blacken the rest of the display also exists and will be used as required. Using the Overlay mechanism means that the Display can be left at the connection resolution and the OSD will overlay on top The ability to scale the overlay will help to provide a OSD than is easy to read.

    • Overlay Scaling (Reg 1300): This will only be used when the OSD resolution is set to Auto In this case, the requirement is that the OSD needs to Display at the same resolution as the connection. If the 4K RX-TX connections resolution is greater than 1920, then the scaling factor will be set as follows:

      • For widths of 1920 or less, we set the value to 0x00 for No scaling.
      • For widths between 1920 and 3840, we set the value to 1, which scales the overlay by a factor of 2 (i.e. doubles the size)
      • For widths over 3840, we set the value to 2, which scales the overlay by a factor of 3 (i.e. triples the size)
      • Overlay Base Address (Reg: 131C)
      • This register is configured with the address of the Label In our case, this is the address of the framebuffer(fb1) that the label is written into The value is: 0x7A050000
      • Overlay Positioning (Regs: 1324, 1328)
      • These registers will be setup based on: 1) The display resolution and 2) the OSD resolution. The aim will be to display the OSD att he centre of the screen.
      • Overlay Width/Height (Regs: 132C, 1334
      • These registers will be programmed with the width and height of the OSD

    Hiding the OSD can be done by either turning Off the Overlay or pointing the Overlay to something else (e.g the connectionLabel)

    TBD :- Handling different OSD resolutions (+ Auto option) needs to be considered in the context of using the H/W overlay. Im not sure that taking the existing behaviour (as in the following 2 points) and bringing it forward is the best thing to do. This will need to be discussed further.

    1. If the OSD is set to Auto, then the requirement is that we overlay the OSD on a display size of the connection. The scaling will be used to present a useable OSD.
    2. If the OSD is set to a specific Resolution, then before switching to the OSD, we need to set the display to the OSD resolution and then overlay the OSD on top.

    Details on Displaying/Hiding the OSD for 2K Receiver.

    • A new Primary surface will be added for the OSD on 2K (New FrameBuffer) This is to ensure that it can run at the same time as a 2K VM connection
    • Switching to displaying the OSD (from displaying the connection) depends on the type of connection:
      • 2K RX-TX connection
        • In this case, the secondary to primary switch functionality is used (The rx-tx connection uses the secondary surface and the OSD uses a primary surface)
      • 2K RDP VM connection
        • In this case, switching to the OSD is done by selecting a different Primary surface (The OSD Primary surface). This is done writing the frame buffer (either the VM or OSD) to the following register: 0x800c8330
    • Switching from displaying the OSD to displaying the connection will be done as follows:
      • 2K RX-TX connection:- In this case the switch will be done by switching from Primary to Secondary mode
      • 2K RDP VM connection:- In this case, the switch will be done by selecting the Primary surface for the VM connection

    Proposed Algorithm

    Displaying the OSD required that 1) Primary mode is configured and 2) That the OSD Primary surface is selected (rather than the VM primary surface).

    • Switching between secondary and primary modes has already been implemented in the display_layer CLI, so the sequence of register writes already exists for this The plan is to use this functionality (Not the CLI) to switch between secondary and primary modes.
    • Switching from displaying one Primary surface to another (as in the case where we are going from displaying a VM connection to displaying the OSD and vice versa) can be done by a single register write (Register: 0x800c8330) The value written to this register is the base address of the primary surface to be displayed.

    Enhanced OSD Phase 2 cloudgui design

    In phase 2 following aspects was taken into consideration:

    • Connection launching process;
    • In connecting state we want to block user from accessing the USB functionality and this remains until the connection is terminated. This prevents someone inadvertently sending events to the OSD when it is in the background. User access to USB related functionality needs to be blocked when connection initialization started, but not completed for managed, unmanaged devices and based on user privilege.
    • When the connection is running the user should not be able to send USB events to the OSD. All keyboard and mouse events to be blocked when connection is running.
    • In terminating state we want to block user from accessing the USB functionality and this remains until the connection is terminated. This prevents someone inadvertently sending events to the OSD when it is in the background. User access to USB related functionality needs to be blocked when connection termination started, but not completed for managed, unmanaged devices and based on user privilege.
    • Allow user access to OSD when connection completed and OSD is visible for managed, unmanaged devices and based on user privilege.
    • Auto login with auto connect option.
    • OSD inactivity timer. If we disable the keyboard and mouse then there is a danger we will invoke these timeouts so this must be disabled when connection is running and cloudgui stay in the background.
    • Last connection name selection in the list when user returns from connection. Not an issue if the OSD stays active, but if there is a crash then we will need to recover and save this file (see below: Decision required).
    • MQTT msg processing when OSD is running long lasting processes (concurrent processing) which we dont want to interrupt (such as upgrade, diagnostics, etc). If current user is running a process like diagnostics (which takes time) we dont want to interrupt this process because of any of the forced events like login, logout or connect.

    Decision required:

    What will be the process in the unlikely event of OSD failure when connection is in progress (CM states are Connecting, Connected or Terminating). OSD will require to launch and set currently running connection name in the list (using of /tmp/WAS_CONNECTED file). OSD as well need to limit user access obtaining the state of CM how to achieve this.

    Solutions might include:

    • Add CM current state to /tmp/WAS_CONNECTED file;
    • Pass CM current state as an arg when cloudgui relaunched;
    • Request CM current state as MQTT msg when cloudgui relaunched;

    Cloudgui and external interfaces communication

    According to requirements of this feature Cloudgui will need to provide mechanism to receive and publish MQTT messages from different external interfaces. Following diagram is describing cloudgui incoming and outgoing MQTT messages.

    Connection Manager changes required

    This section highlights the additional messages required to support a concurrent OSD. Additional messages are sent to and from the OSD.

    Current CM state: Idle of document Enhanced OSD detailed Design v0.7 will require changes (changes highlighted in orange colour).

    In IDLE state, the CM is performing the following operations:

    1. Validate received message. If received message is not valid then CM adds error log into syslog file and wait for next message.
    2. valid requests in the IDLE state are:
      • Force Connect from Boxilla
      • Make Connect from Boxilla
      • Connect from OSD
      • Favourite switch (hotkey)
      • Force Logout from Boxilla.
      • Force Login from Boxilla.
    3. In force logout case, the CM publishes CM_LOGOUT_RQ_TOPIC message to cloudgui and then waits for next message.
    4. In force logout case, the CM publishes CM_LOGIN_RQ_TOPIC message providing user name to be logged in to cloudgui and then waits for next message.
    5. In case of valid message, CM sends connection details to Connection Launcher process to launch the connection.
    6. CM publishes SET_CONNECTING _STATE_TOPIC message to cloudgui
    7. CM updates its state to CONNECTING.
    8. The CM waits until a message is received.

    Current CM state: CONNECTING STATE of document Enhanced_OSD detailed Design v0.7 will require changes (changes highlighted in orange colour).

    In CONNECTING state, the CM is performing the following operations:

    1. Validate received message. If received message is not valid then CM adds log into syslog file and waits for next message.
    2. valid requests in the CONNECTING state are:
      • Exit code from Launcher process.
      • Connection established notification from client.
    3. Following operations are performed in below valid message scenarios.
      • Exit code from Launcher process: If connection (client) failed then the CM receives exit status (exit code) of client from Connection Launcher. Then CM perform post termination operation, sends exit code to osd manager, SET_IDLE_STATE_TOPIC message to cloudgui and updates CM state to IDLE.
      • Connection established notification from client: CM informs connection status to Boxilla, sends exit code to osd manager, SET_CONNECTED_STATE_TOPIC message to cloudgui and then updates the CM state to CONNECTED.
    4. The CM waits until a message is received.

    Current CM state: CONNECTED STATE of document Enhanced OSD detailed Design v0.7 will require changes (changes highlighted in orange colour).

    In CONNECTED state, CM is doing following operations:

    1. The CM checks connection status and if connection (client) has already terminated abnormally then there is no need to send exit request to client so the CM publishes SET_DISCONNECTING _STATE_TOPIC message to cloudgui and enters into TERMINATING state directly. And if connection is running then it performs following operations
    2. Validate received message. If received message is invalid then CM adds log in syslog file and waits for next message.
    3. In case of abnormal termination of connection, CM gets exit code from Launcher then CM perform post termination operation and sends exit code to osd manager. Then CM publishes SET_IDLE _STATE_TOPIC message to cloudgui and updates CM state to IDLE and wait for next message.
    4. In case of Force logout, CM updates respective data structure.
    5. Valid request in CONNECTED state are:
      1. Terminate request from Connection Control/Webservice.
      2. Exit Code from connection launcher process.
      3. Force Connect from Boxilla.
      4. Favourite Connect from Connection Control.
      5. Switch to Previous from Connection Control.
      6. Force Logout from Boxilla.
      7. Force Login from Boxilla.
    6. In case of valid message CM sends an exit connection request to client.
    7. In force logout case, the CM publishes CM_LOGOUT_RQ_TOPIC message to cloudgui and then waits for next message.
    8. In force login case, the CM publishes CM_LOGIN_RQ_TOPIC message providing user name to be logged in to cloudgui and then waits for next message.
    9. CM publishes SET_DISCONNECTING _STATE_TOPIC message to cloudgui
    10. Updates CM state to TERMINATING
    11. The CM waits until a message is received.

    Current CM state: TERMINATING STATE of document Enhanced OSD detailed Design v0.7 will require changes (changes highlighted in orange colour).

    1. Validate received message. If received message is not valid then CM adds log into syslog file and waits for next message.
    2. Valid request in TERMINATING state:
      • Exit code from connection launcher process.
      • Force logout.
    3. In case of Force logout, CM updates respective flag in show osd topics data structure and then wait for exit code from launcher process. CM publishes CM_LOGOUT_RQ_TOPIC message to cloudgui and then wait for exit code from launcher process.
    4. In case of Force logout, CM updates respective flag in show osd topics data structure and then wait for exit code from launcher process. CM publishes CM_LOGIN_RQ_TOPIC providing user name to be logged in message to cloudgui and then wait for exit code from launcher process.
    5. In case of Exit code from connection launcher process, CM Tear down the connection.
    6. The CM notify connection status to Boxilla and the OSD Manager.
    7. The CM publish show osd message to osd manager. CM publishes SET_IDLE _STATE_TOPIC message to cloudgui. Updates CM state to IDLE. In case of Force/fav/switch/persistence connection, the CM publishes SET_CONNECTING _STATE_TOPIC message to cloudgui sends connection details to connection launcher process to launch new connection and updates CM state to CONNECTING.
    8. The CM waits until a message is received.

    OSD Manager changes required

    This section highlights the additional changes required to support a concurrent OSD (changes highlighted in orange colour).

    Remove Force login/ force logout MQTT messages processing from OSD manager. In phase 2 CM_LOGIN_RQ_TOPIC and CM_LOGOUT_RQ_TOPIC will be processed in cloudgui.

    Exit message will require changes. In case of abnormal connection termination, Connection Manager publish exit code to OSD Manager on mqtt OSD_SHOW_TOPIC. OSD manager get the exit message using exit code from bbox_error_strings module. OSD Manager update osd state to SHOW_OSD. The exit message displayed by publishing error text SET_DISCONNECT_ERROR_TOPIC to cloudgui on mqtt and it blocks OSD Manager in the SHOW_OSD switch case of the main loop of the OSD Manager.

    Cloudgui changes required

    In order to provide communication mechanism to receive and publish MQTT messages new MQTT_Communicator class (name to be agreed) implementation is required within cloudgui application. In order to provide constantly running non-blocking process for listening of incoming MQTT messages MQTT_Communicator class will be subthreaded from other cloudgui classes and run constantly while cloudgui is executed. Following diagram is showing communication required between cloudgui classes (external MQTT messages simplified).

    MQTT_Communicator Class

    MQTT_Communicator class is responsible for receiving and interpreting incoming MQTT messages and performing relevant functional calls which will be implemented within existing cloudgui classes. This class also will provide centralized platform to format and publish MQTT messages.

    To handle incoming messages a separate thread will run asynchronously, and incoming mqtt message process is blocking. Outgoing messages will be sent asynchronously within main cloudgui thread so that any transmit failure is immediate.
