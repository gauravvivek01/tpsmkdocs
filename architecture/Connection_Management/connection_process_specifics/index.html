    1. Connection Management - More Details on specific aspects

    1.1 Bonded Connections.

    Bonded connections are very different to standard connections. As such, they warrant special attention in this document.

    1.1.1 Bonded Connection Overview:

    A bonded connection is a group of connections that get launched simultaneously on a group of Receivers. Creating a bonded connection can only be done in a Manager environment. The basic concept is to create a ‘bonded group’ which is made up of a number of Emerald Receivers. A bonded connection is then created by assigning a specific connection to each of the receivers in the bonded group.

    1.1.2 Bonded Connection Design

    A bonded connection can be launched by selecting the bonded connection on any of the receivers in the Bonded group and hitting the ‘Connect’ button. This will result in all the individual connections being established (subject to the required checking on Boxilla). The basic scheme for launching a bonded connection works as follows:

    • A Bonded connection is selected on a receiver and the Connect button is hit.
    • A Call is then made from the Cloudgui to the RestAPI to launch the bonded connection
      • This is done by the RestAPI sending a message to Boxilla to request that the Bonded Connection be Launched
    • Boxilla then checks to see if it is possible to make the connections.
      • If it is not possible (one transmitter not free would be one reason), then an Error is returned to the RestAPI. The RestAPI in turn published the SET_DISCONNECT_ERROR Topic which is subscribed to by the Cloudgui. The Cloudgui will tell the user that the connection failed and the reason why.
      • If the Boxilla checks pass, the connections are launched by Boxilla using the Force Connect Rest API on all the receivers. This is detailed in the ‘Webservice’ section of this document.

    The basic scheme for terminating a bonded connection works as follows:

    • A user on one of the receivers that is part of the bonded connection hits the Exit Hotkey
    • The connection is terminated as normal (See how Hotkey Exit is handled in Connection Control part of this document)
    • A Notification is then send to Boxilla (via restapi) that this connection has exited.
    • Boxilla realises that this is part of a bonded connection and sends Terminate Requests (Rest API) to all the other receivers in the bonded group – This results in the full bonded connection being terminated.

    1.2 Changes Required to Buffers/ Surfaces etc. to allow the OSD to run constantly.

    The previous implementation on both 2K and 4K sees the OSD sharing the same buffer/surface assome of the connection clients. This was implemented in this way because the OSD would never be running at the same time as the clients In this project, the core requirement is that we can run both the OSD and the clients at the same time (to facilitate quicker switching). To be able to facilitate this requirement, we need to ensure that the OSD and the different clients never use the same block of DDR. The new buffers/surfaces will be allocated in the Devicetrees and configured via the DisplayLayer.

    1.3 Overlay Mechanisms

    In all, there are three different Overlay mechanisms for different types of Emerald Receivers. The 4K overlay mechanism allows scaling (max scale factor 3) – This is useful to ensure that an 800x600 OSD can be used on a 4K display resolution. The 2K overlay mechanism is available on all 2K receivers bar the SE Dual Head devise – This is due to the Overlay logic not having enough space in those devices Fpga. A Primary mode Overlay mechanism is available for SE Dual Head boards. The basics of the three schemes are the same – Point the Overlay to the image being displayed, configure the Overlay and finally turn on the overlay.

    1.3.1 4K Overlay

    1.3.1.1 Overview

    A block of Fpga DDR has been allocated to hold the OSD – A FrameBuffer has been allocated in the 4K RX Devicetree pointing to this DDR Block. The reason for this separate OSD buffer is to ensure that the OSD can run at the same time as any of the clients. The HW Overlay is the mechanism to be used for displaying the OSD on a 4K RX. The principle will be the same as for the connectionLabel. When a connection is running, the H/W Overlay will (if it is enabled) be displaying the connectionLabel – when the connection exits, the H/W Overlay will switch to displaying the OSD. Moving forward, the Overlay mechanism will have more uses.

    1.3.1.2 Overlay Registers

    • HW_OSD_SCALE_FACTOR_REG_OFFSET – 1300
      • The bottom 2 bits of this register are used to scale the overlay (The rest are unused). This allows us to scale the overlay so that it is visible at high resolutions.
    • HW_OSD_ENABLE_REG_OFFSET – 1304
      • The bottom bit is the enable/disable bit (The rest are unused). To Enable the overlay, the bottom bit is set to 1
    • HW_OSD_FLUSH_REG_OFFSET – 1308
      • This is a placeholder register (I.e. functionality not currently supported), so the contents are set to 0
    • HW_OSD_READ_BURST_SIZE_REG_OFFSET – 1314
      • As per the AFS, this should be left at the default value of 0x10
    • HW_OSD_READ_BURST_PER_FRAME_REG_OFFSET – 1318
      • This value can be calculated using the following formula: - (width * height *4) /256. In our case (label fixed at 480x40), this equates to: 0x12C
    • HW_OSD_BASE_ADDRESS_REG_OFFSET - 131C
    • HW_OSD_TEST_MODE_REG_OFFSET – 1320
      • This is for hardware test, so we set it to 0
    • HW_OSD_X_OFFSET_REG_OFFSET – 1324
    • HW_OSD_Y_OFFSET_REG_OFFSET – 1328
      • The X and Y offset registers are the starting coordinates where the label is displayed. This is calculated using the following inputs:
        • The selected position of the label (top/bottom centre/right)
        • The resolution of the display
        • The size of the label (480x40 in this case)
    • HW_OSD_WIDTH_REG_OFFSET - 132C
    • HW_OSD_HEIGHT_REG_OFFSET – 1330
    • HW_OSD_BACKGROUND_FILL_REG_OFFSET – 1334
      • This allows us to fill the rest of the screen (outside the label overlay) with black. We want to leave the background as it is, so we set this value to 0x0

    1.3.1.3 Overlay Algorithm:

    The OSD is written to the FrameBuffer at a fixed size of 800x600. The Overlay algorithm is based on what resolution the display is set to. The configuration for the OSD Resolution determines the display resolution (Can be set to a fixed value or left at the resolution of the connection if set to ‘auto’ mode). The overlay image can be positioned anywhere and can be scaled as required. The ability to blacken the rest of the display also exists and will be used to ensure that the areas of the display outside the OSD are black. The ability to scale the overlay will help to provide a OSD than is easy to read at all Display Resolutions The following is the algorithm (basically a sequence of register writes)

    • Configure the 4K Overlay to display the OSD
      • Setup Scaling (Reg 1300): This will only be used when the OSD resolution is set to ‘Auto’ In this case, the requirement is that the OSD needs to Display at the same resolution as the connection. If the 4K RX-TX connections resolution is greater than 1920, then the scaling factor will be set as follows:
        • For widths of 1920 or less, we set the value to 0x00 for No scaling.
        • For widths between 1920 and 3840, we set the value to 1, which scales the overlay by a factor of 2 (i.e. doubles the size)
        • For widths over 3840, we set the value to 2, which scales the overlay by a factor of 3 (i.e. triples the size)
      • Overlay Base Address (Reg: 131C)
        • This register is configured with the address of the OSD – In our case, this is the address of the framebuffer that contains the OSD – The value is: 0x7B18A000
      • Overlay Positioning (Regs: 1324, 1328)
        • These registers will be setup based on the display resolution. The aim will be to display the OSD at the centre of the screen.
      • Overlay Width/Height (Regs: 132C, 1334
        • These registers will be programmed with the width and height of the OSD (800x600)
    • Turn on the Overlay.
      • Overlay Enable (Reg: 1304) – Set the bottom bit to 1.
    • Turn off the Overlay
      • Overlay Enable (Reg: 1304) – Set the bottom bit to 0

    1.3.2 2K Overlay (Bar SE Dual-Head)

    1.3.2.1 Overview

    A block of Fpga DDR has been allocated to hold the OSD – A Framebuffer has been allocated in the 2K RX Device tree pointing to this DDR Block. The reason for this separate OSD buffer is to ensure that the OSD can run at the same time as any of the clients. The Overlay principle will be the same as for the connectionLabel. When a connection is running, the Overlay will (if it is enabled) display the connectionLabel – when the connection exits, the Overlay will switch to displaying the OSD. Moving forward, the Overlay mechanism will have more uses.

    1.3.2.2 Overlay Registers

    • H0_OVERLAY_BASE_ADDR_REG_OFFSET (0x8384) -This is used to contain the base address of the image being overlaid.
    • HO_OVERLAY_CONTROL_REG_OFFSET (0x8380)
      • Enable Bit is Bit 7 (1 = Enabled)
      • Update/Apply bit is Bit 6 (1 to apply the value)
    • H0_OVERLAY_HORIZONTAL_OFFSET_REG (0x8388)
    • HO_OVERLAY_VERTICAL_OFFSET_REG (0x838A)
      • Horizontal and Vertical position to start the OSD at
    • H0_OVERLAY_HORIZONTAL_SIZE_REG (0x838C)
    • H0_OVERLAY_VERTICAL_SIZE_REG (0x838E)
      • Horizontal and vertical size of OSD – 800x600 in our case

    1.3.2.3 Overlay Algorithm

    The algorithm is very simple. The Display resolution is read first and the horiz / vertical offsets are calculated based on this in conjunction with the OSD size *800x600). The offsets are then written to the two offset registers. The size (800x600) is written to the size registers. The Address of the Framebuffer is written to the base address register. The next thing to ensure that the secondary surface is pushed off screen – Writing 0x780 to the SECONDARY_STREAM_DISPLAY_POS_H_0_ADDR register will achieve this. The last step is to turn ON the overlay and apply the change – This is done in the Control Register. Turning off the Overlay will ‘hide’ the OSD – This can be done with the Enable/Update bit in the Control register

    1.3.3 2K Overlay – SE Dual-Head

    1.3.3.1 Overview

    The basic principle is the same as per the rest of the 2K receivers - A block of Fpga DDR has been allocated to hold the OSD – A Framebuffer has been allocated in the 2K RX Device tree pointing to this DDR Block. The overlay mechanism is a primary mode Overlay, so it will not function when the Primary mode is being used (e.g. in an RDP VM connection). This works for Overlaying the OSD (as the connection needs to exit first), but won’t work for the connectionLabel, as this is over laid on top of a connection (+ it will have other limitations moving)

    1.3.3.2 Overlay Registers

    • DISPLAY_OUTPUT_FORMAT_CONTROL_0_ADDR (0x8303)
      • Blank the Display
    • PRIMARY_STREAM_BASE_ADDR_0_ADDR (0x8330)
      • Set the Base address of the OSD framebuffer (0x31400000)
    • PRIMARY_STREAM_DISPLAY_SIZE_H_0_ADDR (0x8320)
    • PRIMARY_STREAM_DISPLAY_SIZE_V_0_ADDR (0x8322)
      • Set the Size of the Overlay being displayed – 800x600 in this case
    • PRIMARY_STREAM_DISPLAY_POS_H_0_ADDR (0x8324)
    • PRIMARY_STREAM_DISPLAY_POS_V_0_ADDR (0x8326)
      • Set the co-ordinates where the OSD will start from
    • PRIMARY_SURFACE_STRIDE_REGISTER (0x833C)
      • Set the Primary OSD width to 800
    • DISPLAY_FETCH_CONTROL_0_ADDR (0x8302)
      • Set primary fetches only
    • DISPLAY_TEST_PATTERN_0_ADDR (0x8304)
      • Turn OFF test modes
    • H0_OVERLAY_CONTROL_REG_OFFSET (0x8380)
      • Turn on the Update bit.

    1.3.3.3 Overlay Algorithm

    The algorithm is very simple. The Display resolution is read first and the horiz / vertical offsets are calculated based on this in conjunction with the OSD size (800x600). The offsets are then written to the two offset registers. The size (800x600) is written to the size registers. The Address of the Framebuffer is written to the base address register. The last step is to turn ON the overlay and apply the change – This is done in the Control Register. Turning off the Overlay will ‘hide’ the OSD – This can be done with the Enable/Update bit in the Control register

    1.5 Persistent Connections

    Any connection can have persistence enabled. This means that if the connection fails for any reason, then the connection will be re-launched. The following are important points to note:

    • If the connection fails on its first attempt, then it will not be re attempted
    • If it is a controlled connection exit (Hotkey / Rest API terminate ...), then it will not be re- attempted
    • Once persistence kicks in, the connection will continuously be re-attempted while it continues to fail
    • The only way to break out of the continuous re launching of the connection is with a Hotkey Exit

    The Connection Manager handles connection persistence as follows:

    • Once a connection exits (CM receives the SEND_EXIT_CODE Topic), the CM checks it the connection is persistent or not.
    • If it is persistent, then it checks the exit code
      • If the exit code reflects a controlled exit, then nothing further is done and the OSD is displayed
      • If the exit code reflects a failure, then a re-launch of the connection is attempted
