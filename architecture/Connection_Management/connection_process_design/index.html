    1. Connection Management - Detailed Design.

    The following gives a detailed description of all the individual processes involved in Connection Management on the Emerald Receivers. All the processes involved are individual applications that run on the Emerald Receivers and communicate via Mosquitto messages. In general, the behaviour is the same on all types of receivers - any differences will be identified in the specific sections.

    1.1 Connection Manager (CM)

    1.1.1 CM Overview

    The Connection Manager (CM) is a state machine based standalone process that handles ALL Connection based requests. It communicates with the other applications that are part of the overall connection process over Mosquitto. The state machine has four states that change as connections are made/switched and terminated. As states are changed, the CM publishes a mosquito message with the current state – This allows any other application to be aware of the Connection state on the receiver. The CM states are as follows:

    • Idle
    • Connecting
    • Connected
    • Terminating

    1.1.2 CM structure

    The CM is made up of the following basic parts:–

    • A mosquito callback function that is called when any of the messages of interest to the CM are Published
    • A message_queue that the Callback function posts to whenever a mosquito message arrives
    • A state Machine that gets driver through its states as mosquito messages get processed

    1.1.3 CM Initialization

    The CM initialization is as follows:

    • A new Mosquitto Client is created for the CM
    • The Mosquitto callback function is created for the CM
    • The CM Mosquitto client is connected to the Mosquitto Broker
    • A CM Mosquitto message processing thread is created

    The CM subscribes to the Mosquitto messages that it is interested in.
    A message queue is created (for communicating between the Callback and the CM Thread).
    The CM thread starts waiting for messages to arrive in its message queue.

    1.1.4 CM State Machine states and valid messages

    The CM state machine is designed to accept any Message in any state. That said, not all messages are valid in every state. If the CM receives a message that is not valid for a particular state, then it logs it to syslog and drops it. This is core to the CM design and ensures that we can function in a deterministic manner. The reason behind this approach is due to the fact that connections can be launched/switched/terminated via the REST API and also from the OSD (In theory, both of these can be active at the same time).
    The following lists the valid messages in each CM state. Any other message that arrives is flagged as Invalid in syslog and no further action is taken.

    • State: IDLE
      • MAKE_OSD_CONNECTION_TOPIC
      • FORCE_CONNECTION_TOPIC
      • FORCE_LOGOUT_TOPIC
      • GET_CURRENT_CM_STATE_TOPIC
    • State: CONNECTING
      • CONNECTION_LAUNCHED_TOPIC
      • SEND_EXIT_CODE_TOPIC
      • TERMINATE_CONNECTION_CC_TOPIC
      • GET_CURRENT_CM_STATE_TOPIC
    • State: CONNECTED
      • FORCE_LOGOUT_TOPIC
      • TERMINATE_CONNECTION_CC_TOPIC
      • TERMINATE_CONNECTION_WEBSERVICE_TOPIC
      • TERMINATE_WITH_NO_OSD_TOPIC
      • PREVIOUS_CONNECTION_TOPIC
      • FORCE_CONNECTION_TOPIC
      • SEND_EXIT_CODE_TOPIC
      • GET_CURRENT_CM_STATE_TOPIC
    • State: TERMINATING
      • TERMINATE_CONNECTION_CC_TOPIC
      • SEND_EXIT_CODE_TOPIC
      • GET_CURRENT_CM_STATE_TOPIC

    1.1.6 CM States – Detailed Description:

    1.1.6.1 CM state: IDLE

    The following details how the CM processes incoming messages in the IDLE state. The required processing depends on the message topic the CM receives in the IDLE state. The basic flow is as per the flow chart above – The CM state machine is waiting for messages (Topics) to arrive in its message queue. Once a message arrives, it processes it based on the current State Machine state.
    In this case, the following is how messages are processed in the IDLE state

    • Incoming Message:- Launch a connection Topic (Make_Osd_Connection or Force_Connection)

      • The Launch Connection Topic is published
        • The Connection Launcher Process subscribes to this message. When it receives it, it launched the connection client (see details in the Connection Launcher section of this document)
      • The State is updated to CONNECTING
      • The CM State Changed Topic is published with the current state
      • Go back to the start of the loop and check for new messages

    • Incoming Message:- Force Logout Topic (Force_Logout)

      • The CM Logout Request Topic is published
        • The Cloudgui Process subscribes to this message. When it receives it, it logs out the user (see details in the Cloudgui section of this document)
      • The State is NOT changed.
      • Go back to the start of the loop and check for new messages

    • Incoming Message:- Get CM State Topic (Get_Current_CM_State)

      • The CM State Changed Topic is published with the current state
        • Anybody that is interested in the state of the CM will have subscribed to this message and will now receive the current CM state.
      • The State is NOT changed
      • Go back to the start of the loop and check for new messages

    • __Incoming Message:- Any other Topic __

      • Write an Error to Syslog indicating that an Invalid message was received in the IDLE state.

    1.1.6.2 CM state: CONNECTING

    The following details how the CM processes incoming messages in the CONNECTING state. The required processing depends on the message topic the CM receives in this state. The basic flow is as per the flow chart above – The CM state machine is waiting for messages (Topics) to arrive in its message queue. Once a message arrives, it is processed based on the current State Machine state.

    • Incoming Message:- Connection has successfully launched Topic (Connection Launched)

      • The OSD_HIDE topic is published
        • The OSD Manager subscribes to this message – When it receives it, it no longer displays the OSD – See the OSD Manager section for more details
      • The State is changed to CONNECTED.
      • The CM State Changed Topic is published with the current state
      • Go back to the start of the loop and check for new messages

    • Incoming Message:- Connection has failed to establish Topic (send exit code)

      • The OSD_SHOW topic is published
        • The OSD Manager subscribes to this – When it receives it, it displays the OSD – See the OSD Manager Section for more details.
      • The State is changed to IDLE.
      • The CM State Changed Topic is published with the current state
      • Go back to the start of the loop and check for new messages

    • Incoming Message:- Terminate Connection (Hotkey) Topic (terminate connection CC) (Note: This doesn’t fit in this state – It has only being included to ensure that we can always Hotkey exit from a connection from all states)

      • The OSD_SHOW topic is published
        • The OSD Manager subscribes to this – When it receives it, it displays the OSD – See the OSD Manager Section for more details.
      • The State is changed to IDLE.
      • The CM State Changed Topic is published with the current state
      • Go back to the start of the loop and check for new messages

    • Incoming Message:- Get CM State Topic (Get Current CM state)

      • Handled same as in IDLE state

    1.1.6.3 CM state: CONNECTED

    The following details how the CM processes incoming messages in the CONNECTED state. The required processing depends on the message topic the CM receives in this state. The basic flow is as per the flow chart above – The CM state machine is waiting for messages (Topics) to arrive in its message queue. Once a message arrives, it is processed based on the current State Machine state.

    • Incoming Message:- Force Logout / terminate Topic (Force_Logout, Terminate_CC, Terminate_webservice, Terminate_no_osd)

      • The OSD_SHOW topic is published (In all cases bar ‘terminate_no_osd’)
        • The OSD Manager subscribes to this message – When it receives it, it no longer displays the OSD – See the OSD Manager section for more details
      • For KVM and RDP connections, The EXIT_CONNECTION topic is published.
        • The Client (bbfreerdp, xfreerdp, dfreerdp) subscribe to this message and exit cleanly when received.
      • For other clients, they are signalled to exit.
      • The State is changed to TERMINATING
      • The CM State Changed Topic is published with the current state.
      • Go back to the start of the loop and check for new messages

    • Incoming Message:- Switch to Previous / Force Connect Topics (Previous_connection , Force_Connection)

      • Connection clients are told to exit ( see details in CONNECTED state)
      • The state is set to TERMINATING
      • Flags are set to indicate we are switching to previous or switching to a new connection. The connection details for the new connection are also set.
        • The rest of this process is handled in the ‘TERMINATING’ state when the Clients exit and the Connection Launcher published the ‘send_exit_code’ topic
      • Go back to the start of the loop and check for new messages

    • Incoming Message:- Send exit code Topic (send_exit_code)

      • There are 2 scenarios here that need to be considered:
        • Connection is NOT Persistent
      • The OSD_SHOW Topic is published
      • The CM state is changed to IDLE
      • The CM State Changed Topic is published with the current state
        • Connection IS Persistent
      • In this case, the Exit Code is checked to see if it is a Controlled Exit or if the connection has exited due to an error.
      • Controlled Exit:- The behaviour is as per the ‘non persistent’ case and the OSD gets displayed
      • Non-Controlled Exit:- In this case, we do not go back to the OSD. Instead, we attempt to re-launch the connection (and continue to do so while it exits with a non-controlled exit code). This will continue until either:
        • The connection succeeds (typically doe to the cause of failure getting resolved)
        • A Hotkey exit is triggered by the user.

    • Incoming Message:- Get CM State Topic (Get Current CM state)

      • Handled same as in IDLE state

    1.1.6.4 CM state: TERMINATING

    The following details how the CM processes incoming messages in the TERMINATING state. The required processing depends on the message topic the CM receives in this state. The basic flow is as per the flow chart above – The CM state machine is waiting for messages (Topics) to arrive in its message queue. Once a message arrives, it is processed based on the current State Machine state.

    • Incoming Message:- Client has exited Topic (send_exit_code)

      • If previous_flag or force_flag are set (I.e. if we are in the middle of a switch)
        • The Launch Connection Topic is published (with the new connection)
          • The Connection Launcher Process subscribes to this message. When it receives it, it launched the connection client (see details in the Connection Launcher section of this document)
        • The State is updated to CONNECTING
      • Otherwise
        • The state is set to IDLE
      • The CM State Changed Topic is published with the current state

    • Incoming Message:- Get CM State Topic (Get Current CM state)

      • Handled same as in IDLE state

    1.2 OSD Manager

    1.2.1 OSD Manager Overview

    The OSD Manager (OM) handles all aspects of displaying / hiding the OSD. The terminology we use moving forward is ‘show’ and ‘hide’. The OM also handles controlled restarts of the Cloudgui process and non-controlled restarts in the case of the Cloudgui process exiting unexpectedly. The OM is totally driven by Mosquitto messages. The OM has the following basic functionality:

    • Initialization
    • Show
    • Hide
    • Relaunch

    1.2.2 OSD Manager Structure

    The OM is made up of the following basic parts:–

    • A mosquito callback function that is called when any of the messages of interest to the OM are Published
    • A message queue that the Callback function posts to whenever a mosquito message arrives
    • A thread sitting on the message queue and processing any incoming requests

    1.2.3 OM Detailed Description

    The following flowchart details how the OSD Manager works.




    1.2.3.1 Initialization

    OM Initialization has two basic parts. The first one is to do the Mosquitto configuration. The second is to setup the display and display the OSD. The following details the steps involved

    • A new Mosquitto Client is created for the OM
    • The Mosquitto callback function is created for the OM
    • The OM Mosquitto client is connected to the Mosquitto Broker
    • An OM Mosquitto message processing thread is created
    • The OM subscribes to the Mosquitto messages that it is interested in
    • A message queue is created (for communicating between the Callback and the CM Thread)
    • The CM thread starts waiting for messages to arrive in its message queue
    • Setup the display
    • Run the ‘cloudgui’ binary
    • Overlay the OSD on the display.

    1.2.3.2 Show OSD

    The ‘show’ functionality is what is used to configure and display the OSD. This is done under the direction of the Connection Manager. The connection manager requires the OM to ‘show’ the OSD when a connection has exited (or more precisely when a connection is exiting).
    The basic sequence is as follows:

    • OM callback is called with Show message (published by the CM).
    • Callback adds a message to the OM Message Queue
    • OM thread detects the message and does the following:
      • Setup the display
      • Overlay the OSD on the display
    1.2.3.2.1 Setup the Display
    1.2.3.2.2 Overlay the OSD

    1.2.3.3 Hide OSD

    The ‘hide’ functionality is what is used to remove the OSD from the display. This is done under the direction of the Connection Manager. The connection manager requires the OM to ‘hide’ the OSD when a connection has successfully established.
    The basic sequence is as follows:

    • OM callback is called with Hide message (published by the CM).
    • Callback adds a message to the OM Message Queue
    • OM thread detects the message and does the following:
      • Turns off the OSD Overlay mechanism

    1.2.3.4 Relaunch OSD

    Relaunching the Osd process (Cloudgui) is required in the following two cases:

    • Where the OSD exits due to user changes that require it restart with the new settings
      • This occurs when either the OSD Resolution of the Keyboard Language is changed. In both these cases, the Cloudgui publishes a Mosquitto Message indicating what has changed and then exits cleanly. The OM subscribes to these messages – When it receives either of them, it goes through the process of re-launching it.
    • Where the OSD exits unexpectedly (i.e. Crashes / segments ...)
      • In this case, the OM is subscribing to a ONE_SECOND Mosquitto topic (published by ConnectionControl). Every time it gets this message, it checks to see if the ‘cloudgui’ process is still running. If it is not, it increments a count – If the count hits ten, then it concludes that the cloudgui has exited un-expectedly and goes through the process of relaunching it.

    Relaunching the OSD is achieved in the following three basic steps:

    • Setup the display
    • Run the ‘cloudgui’ binary
    • Overlay the OSD on the display.

    1.3 Connection Launcher

    1.3.1 Connection Launcher Overview

    The Connection Launcher (CL) is an application whose only function is to launch the connection application (bbfreerdp, dfreerdp ...) and publish an Exit Code when the application exits. A dedicated application is required because we need to run the connection application in the foreground in order to know when it has exited – this means that the CL application will not be able to do anything else while a connection is running.

    The CL has two basic inputs. The first is a Mosquitto message to ‘Launch’ a connection. The second is when the connection application exits with an exit code.

    1.3.2 Connection Launcher structure

    The CL is made up of the following basic parts:–

    • A mosquito callback function that is called when any of the messages of interest to the CL are Published
    • A message queue that the Callback function posts to whenever a mosquito message arrives
    • A thread sitting on the message queue and processing any incoming requests

    1.3.3 CL Detailed Description

    The following flowchart details how the Connection Launcher works.




    1.3.3.1 Initialization

    CL Initialization is based on setting up the Mosquitto configuration. The following details the steps involved (Note: In this case, we could have used a simpler scheme as the CM only subscribes to one message – We have stuck with this however to keep a consistent approach in all applications)

    • A new Mosquitto Client is created for the CL
    • The Mosquitto callback function is created for the CL
    • The CL Mosquitto client is connected to the Mosquitto Broker
    • An CL Mosquitto message processing thread is created
    • The CL subscribes to the Mosquitto messages that it is interested in
    • A message queue is created (for communicating between the Callback and the CL Thread)
    • The CL thread starts waiting for messages to arrive in its message queue

    1.3.3.2 Operation

    The basic operation is as follows:

    • The CL message processing thread is sitting on the message queue, waiting for messages to arrive.
    • The CM published the ‘launch_connection_topic’ (The CL has subscribed to this)
    • The CL callback fires and posts a message to the message queue with details of the connection to be launched
    • The CL then launches the connection application (bbfreerdp, dfreerdp ...) with the parameters supplied
    • < At this point, the CL is effectively locked up while the connection is running >
    • The connection will exit at some point with an Exit Code
    • The CL then publishes the exit_code via the ‘send_exit_code’ topic.
    • The CL then goes back to waiting for a new message to arrive in its message queue

    1.4 Connection control

    1.4.1 Connection Control Overview

    Connection control is the standalone application that handles the User Hotkeys. The current list of Hot keys are as follows:

    • Hotkey o (Exit Connection)
    • Hotkey p (Previous Connection)
    • Hotkey 0 to 9 (Launch connection associated with particular favourite key)
    • Hotkey l (Toggle ConnectionLabel)

    It also is used to publish a ONE_SECOND topic, as it has a thread running at a 1sec (roughly) interval. This message can be subscribed to by any application that needs to do something on a timed basis.

    1.4.2 Connection Control Structure

    The basic structure of the connection Control Application is that is has a main thread that is waiting for events to arrive from Libinput. These events are parsed to see if any of they are the particular Hotkey sequences that we are interested in. If they are of interest, then either the CC application handles them itself or more typically tells another application to handle them. Telling another application to process a Hotkey is done via Mosquitto Topics.

    1.4.3 Connection Control Mosquitto setup

    The Mosquitto setup for the Connection control application is different to the others described above. This application currently does not subscribe to any Mosquitto Topics. The result of this is that we do not need a Mosquitto callback or a thread to sit waiting for incoming Mosquitto messages. A Mosquitto Client is however required, as this application needs to publish Mosquitto topics because of different Hotkeys being detected.

    1.4.4 Connection Control detailed description.

    The following details how each of the Hotkeys are handled by the Connection Control application.

    1.4.4.1 Connection Exit Hotkey (o)

    Once this hotkey is detected by the CC application, it is handled by publishing a Mosquitto message. The topic name is: TERMINATE_CONNECTION_CC_TOPIC. The Connection Manager subscribes to this message. See details of how it is handled in the Connection Manager section of this document.

    1.4.4.2 Previous Connection Hotkey (p)

    Once this hotkey is detected by the CC application, it is handled by publishing a Mosquitto message. The topic name is: PREVIOUS_CONNECTION_TOPIC. The Connection Manager subscribes to this message. See details of how it is handled in the Connection Manager section of this document.

    1.4.4.3 Favourite Hotkey (0-9)

    Once this hotkey is detected by the CC application, it is handled by publishing a Mosquitto message. The topic name is: FORCE_CONNECT_TOPIC. The Connection Manager subscribes to this message. See details of how it is handled in the Connection Manager section of this document.

    1.4.4.4 Connection Label Toggle Hotkey (l)

    Once this hotkey is detected by the CC application, it is handled in the CC application. It checks if the ConnectionLabel is on or off (checks the Overlay mechanism to see if it is on/off and also checks to see what application if the overlay mechanism setup to display). If the Label is off, it turns it on and if it is off it turns it on – This is only done if the Overlay mechanism is configured for the ConnectionLabel.

    1.5 Cloudgui

    1.5.1 Aspects taken into consideration within Cloudgui design:

    • Connection launching process;
    • In connecting state we want to block user from accessing the USB functionality and this remains until the connection is terminated. This prevents someone inadvertently sending events to the OSD when it is in the background. User access to USB related functionality needs to be blocked when connection initialization started, but not completed for managed, unmanaged devices and based on user privilege.
    • When the connection is running the user should not be able to send USB events to the OSD. All keyboard and mouse events to be blocked when connection is running.
    • In terminating state we want to block user from accessing the USB functionality and this remains until the connection is terminated. This prevents someone inadvertently sending events to the OSD when it is in the background. User access to USB related functionality needs to be blocked when connection termination started, but not completed for managed, unmanaged devices and based on user privilege.
    • Allow user access to OSD when connection completed and OSD is visible for managed, unmanaged devices and based on user privilege.
    • Auto login with auto connect option.
    • OSD inactivity timer. If we disable the keyboard and mouse then there is a danger we will invoke these timeouts so this must be disabled when connection is running and cloudgui stay in the background.
    • Last connection name selection in the list when user returns from connection. Not an issue if the OSD stays active, but if there is a crash then we will need to recover and save this file (see below: Decision required).
    • MQTT msg processing when OSD is running long lasting processes (concurrent processing) which we don’t want to interrupt (such as upgrade, diagnostics, etc). If current user is running a process like diagnostics (which takes time) we don’t want to interrupt this process because of any of the forced events like login, logout or connect.

    1.6 Webservice / Http_Enable

    1.6.1 Note on Http Enable Application:

    The Webservice / Http Enable applications are very similar. The ‘http_enable’ application only runs when http is enabled on Boxilla and it only supports a subset of the Rest APIs over Http (Webservice uses https). It is used to allow Control Bridge to make / break connections. The APIs it supports are Force Connect, Make Connection and Terminate Connection. The structure of the code / how it functions is the same as the Webservice application.

    1.6.2 Webservice Overview

    Webservice is the application that is used to handle the rest API. The rest API is the interface to the Emerald appliances that we provide to the outside world – It is typically used by Boxilla, but can be used by other applications that have been given access. The Webservice application only runs on units that are managed. In the context of this document, we are only interested in the section of the rest API that relates to connections. The following is the list of Connection APIs:-

    • Force Connect
    • Make Connection
    • Terminate Connection
    • Terminate Connection with No OSD
    • Force Login
    • Force Logout.

    1.6.3 Webservice Mosquitto Setup

    The Webservice application uses Mosquitto to publish connection related topics for other applications (typically the Connection Manager) to process. Based on this, it needs to create a Mosquitto client and connect to the Mosquitto broker. It does not subscribe to any Mosquitto topics and as such does not need a callback / thread to process incoming Mosquitto messages.

    1.6.4 Webservice Detail Description

    This description only relates to how the Connection based APIs are handled.

    1.6.4.1 Force Connection REST API

    This is the API used by the Boxilla manager to launch a connection. If there is an existing connection on the receiver unit, this is terminated and then the new connection is launched. This request is implemented as follows:

    • The details of the connection are validated. The Force Connect Rest API specifies a User and a connection name. The XML file is checked to see if the user and connection name exist and also to ensure that the specified user has access to the specified connection.
    • The connection parameters are read from the XML file are formatted into json format
    • A Mosquitto topic is then published – the topic is FORCE_CONNECT and the details of the connection is the data associated with this message
    • The Connection Manager (It subscribes to this topic) processes the request from here. See details in the CM section of this document.

    1.6.4.2 Make Connection Rest API triggered

    This is similar to Force Connection, but in this case, the rest API parameters are the TX IP only. A hardcoded KVM connection to this TX IP is then created. As in the Force Connect case, any existing connection is terminated before launching this connection.

    • The connection details are built up from default values into a json format data block.
    • A Mosquitto topic is then published – the topic is FORCE_CONNECT and the details of the connection is the data associated with this message
    • The Connection Manager (It subscribes to this topic) processes the request from here. See details in the CM section of this document.

    1.6.4.3 Terminate Connection Rest API triggered

    This is the main api used by Boxilla to terminate a connection. There is no data associated with this API – ultimately it means terminate whatever connection is running. This is handled in Webservice as follows: - The Webservice application published a Mosquitto topic called:- TERMINATE_CONNECTION_WEBSERVICE - The Connection Manager (It subscribes to this topic) processes the request from here.

    1.6.4.4 Terminate Connection No OSD Rest API triggered

    This is the Rest API used by Boxilla when it needs to Terminate a connection but does not want the OSD to appear – This is typically used in Boxilla Presets. The reason it does not want the OSD to appear is because it knows it is planning to launch a different connection on this receiver once the current one is terminated. This is handled in Webservice as follows:

    • The Webservice application published a Mosquitto topic called:- TERMINATE_WITH_NO_OSD
    • The Connection Manager (It subscribes to this topic) processes the request from here.

    1.6.4.5 Force Login over Rest

    This is the Rest API user by Boxilla to Login a user into the OSD. It is processed as follows:

    • The Webservice application publishes a topic called:- FORCE_LOGIN
      • The data associated with the Mosquitto contains the username to login
    • The Cloudgui (It subscribes to this topic) processes the request from here.

    1.6.4.6 Force Logout over Rest

    This is the Rest API user by Boxilla to Logout a user from the OSD. It is processed as follows: - The Webservice application publishes a topic called:- FORCE_LOGOUT - The Connection Manager (It subscribes to this topic) processes the request from here.
