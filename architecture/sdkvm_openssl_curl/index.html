    SDKVM - Open SSL and Curl Issues

    Introduction

    This Technote will illustrates an openssl issue discovered on the SDKVM and its resolution.

    To get the curl version on the SDKVM

    curl -V

    curl 7.81.0 (x86_64-pc-linux-gnu) libcurl/7.81.0 OpenSSL/3.0.2 zlib/1.2.11 brotli/1.0.9 zstd/1.4.8 libidn2/2.3.2 libpsl/0.21.0 (+libidn2/2.3.2) libssh/0.9.6/openssl/zlib nghttp2/1.43.0 librtmp/2.3 OpenLDAP/2.5.14
Release-Date: 2022-01-05
Protocols: dict file ftp ftps gopher gophers http https imap imaps ldap ldaps mqtt pop3 pop3s rtmp rtsp scp sftp smb smbs smtp smtps telnet tftp 
Features: alt-svc AsynchDNS brotli GSS-API HSTS HTTP2 HTTPS-proxy IDN IPv6 Kerberos Largefile libz NTLM NTLM_WB PSL SPNEGO SSL TLS-SRP UnixSockets zstd

    Openssl Issue

    Running the following command:

    curl -X PUT https://10.231.132.250/bxa-api/devices/kvm/ipaddr -H "Authorization: Basic UkVTVF9CYkFkbWluVXNlcjpCb3hpbGxAMjAyMA==" -H "Content-Type: application/json" -H "Accept: application/json" -d '{"mac":"48:21:0B:35:32:74","ip":"10.231.132.55","gateway":"10.211.128.1"}'

    Provides the following response

    curl: (60) SSL certificate problem: unable to get local issuer certificate

    curl -k -X PUT https://10.231.132.250/bxa-api/devices/kvm/ipaddr -H "Authorization: Basic UkVTVF9CYkFkbWluVXNlcjpCb3hpbGxAMjAyMA==" -H "Content-Type: application/json" -H "Accept: application/json" -d '{"mac":"48:21:0B:35:32:74","ip":"10.231.132.55","gateway":"10.211.128.1"}'

curl: (56) OpenSSL SSL_read: error:0A000126:SSL routines::unexpected eof while reading, errno 0
{"status":"200","message":"Successfully updated IP for 48:21^CB:35:32:74 appliance."}

    The application error observed is:

    ulfius_send_http_streaming_request: libcurl error: 56 message: Failure when receiving data from the peer

    In snuc device we are using a ulfius library which uses curl to communicate to Boxilla (the server in this instance).

    Multiple attempts were made to handle ssl certificate issue OpenSSL Read error , below are the details of the various strategies used:

    1. Try setting flag to handle the error by implementing new method in ulfius library
    2. Try passing ca path option from ulfius library.
    3. Curl and openssl version downgrading.
    4. Interoperability of curl latest version with openssl current version.
    5. Different methodology to handle openssl issue

    These are discussed in detail

    Setting the SSL_OP_IGNORE_UNEXPECTED_EOF flag

    We tried to set the SSL_OP_IGNORE_UNEXPECTED_EOF flag to handle the error.

    This flag was added in openssl 3.0 to handle the EOF error.

    This flag was set in ulfius library as follows:

    file : Open_Source_SDKVM_LIB/ulfius/src/u_send_request.c

    static CURLcode sslctx_function(CURL *curl, void *sslctx, void *parm)
{
SSL_CTX_set_options(sslctx, SSL_OP_IGNORE_UNEXPECTED_EOF);
 corrib_syslog(LOG_NOTICE, " inside ctx_function");
  return CURLE_OK;
}
// to invoke above method , curl_easy_setopt() function must be used in below way :
curl_easy_setopt(curl_handle, CURLOPT_SSL_CTX_FUNCTION, sslctx_function);

    This approach did not work; it produced the error: curl(4) error which states :

     A feature or option that was needed to perform the desired request was not enabled or was explicitly disabled at build-time. To make curl able to do this, you probably need another build of libcurl.

Curl version at device : 7.81
OpenSSL version at device : 3.0.2

    With further research it was found that SSL_OP_IGNORE_UNEXPECTED_EOF has not been implemented for the GnuTLS backend.

    This option only works for libcurl powered by OpenSSL, wolfSSL or mbedTLS. If libcurl was not built against these libraries then this functionality is absent.

    Passing ca path option from ulfius library.

    This option involved setting the CURLOPT_CAPATH variable which is a variable that points to the directory holding CA certificates

    curl_easy_setopt(CURL *handle, CURLOPT_CAPATH, char *capath);

    We pass a char * to a null-terminated string naming a directory holding multiple CA certificates to verify the peer with. If libcurl is built against OpenSSL, the certificate directory must be prepared using the OpenSSL c_rehash utility.

    While this method worked to resolve certificate issue but the EOF error still causing problem. When the application tries to wait for receiving response from boxilla we get an error as the application doesn’t wait for more inputs from server once the error is received.

    When manually running the curl commands it still throws EOF error.

    This methodology helped to resolve 1 part of issue but the issue still remains with the error:

    ulfius_send_http_streaming_request: libcurl error: 56 message: Failure when receiving data from the peer
This methodology gave us new direction of investigation towards interoperability of curl and openssl different version.

    Downgrade Curl and openssl version

    We downgraded curl and openssl , this resolves the issue.

    curl Version: curl -V
curl 7.54.0 (arm-buildroot-linux-gnueabihf) libcurl/7.54.0 OpenSSL/1.0.2k zlib/1.2.11 libssh2/1.9.0
Protocols: dict file ftp ftps gopher http https imap imaps pop3 pop3s rtsp scp sftp smb smbs smtp smtps telnet tftp
Features: IPv6 Largefile NTLM SSL libz TLS-SRP UnixSockets HTTPS-proxy

    curl -X PUT https://10.231.132.250/bxa-api/devices/kvm/ipaddr -H "Authorization: Basic UkVTVF9CYkFkbWluVXNlcjpCb3hpbGxAMjAyMA==" -H "Content-Type: application/json" -H "Accept: application/json" -d '{"mac":"48:21:0B:35:32:74","ip":"10.231.132.55","gateway":"10.211.128.1"}'

{"status":"200","message":"Successfully updated IP for 48:21:0B:35:32:74 appliance."}

the openssl certificate issue and EOF error  was due to strict checking which is introduced in openssl 3.0

    While downgrading resolved the issue, we cannot afford to revert to earlier versions so we looked for another approach.

    Using the latest version of curl with openssl current version.

    We decided to evaluate the interoperability of different versions of curl and openssl

    • Downgraded curl : curl 7.70.0 (x86_64-pc-linux-gnu) libcurl/7.70.0 OpenSSL/3.0.2
    • Downgraded openssl : curl 7.81.0 (x86_64-pc-linux-gnu) libcurl/7.81.0 OpenSSL/3.1.0
    • Upgrade curl to latest version 8.0.1: curl 8.0.1 (x86_64-pc-linux-gnu) libcurl/8.0.1 OpenSSL/3.0.2

    The 3rd option worked for us and resolved the issue.

    To Upgrade curl

    To upgrade curl please follow below link : https://yannmjl.medium.com/how-to-manually-update-curl-on-ubuntu-server-899476062ad6

    this will remove the existing curl and install the 8.0.1 curl version.

    How to update a certificate with standard end date

    Run the command:

    openssl ca -config /home/snuc/SDKVM/security/ssl/openssl.cnf -policy signing_policy -out $SSL_LOCATION/ssl/certs/device.crt -startdate 150815080000Z -enddate 150815090000Z -cert /home/snuc/SDKVM/security/ssl/private/rootCA.crt -keyfile /home/snuc/SDKVM/security/ssl/private/rootCA.key -infiles $SSL_LOCATION/ssl/certs/device.csr

    Once this command is executed it will display a prompt for (y/n) to sign certificate , if one enters (y) it will proceed and create certificate with the specified start and end date 

    snuc@snuc-NUC12WSKi7:~/experiment/black$ openssl ca -config openssl.cnf -policy signing_policy -out clientcert.pem -startdate 120815080000Z -enddate 120815090000Z -cert /home/snuc/SDKVM/security/ssl/private/rootCA.crt -keyfile /home/snuc/SDKVM/security/ssl/private/rootCA.key -infiles device.csr
