    1.6 Starting the webservice on SDKVM

    1.6.1 On the appliance on 4k

    The webservice is started by calling the /etc/init.d/S68Discovery script calling ->opt/blackbox/restart_discovery.sh

    1.6.2 On the appliance on 2k

    The webservice is started by calling the /etc/init.d/S69Webservice script calling ->opt/blackbox/restart_webservice.sh

    1.6.3 Starting webservice On the SDKVM

    • The webservice binary exists in /usr/local/bin
    • The restart_webservice script is managed by a systemd user service
    • The service is managed and started by /home/snuc/SDKVM/scripts/restart_webservice.sh. This also operates a systemd user service.
    • The configuration service file for restart_webservice exists in ~/.config/systemd/user/restart_webservice.service.
    • To enable restart_webservice as a service we use: systemctl --user enable restart_webserivice
    • The webservice itself is not run as a service because it is stopped and started by the restart_webservice script.

    1.15 Capability Endpoints for SDKVM

    Based on the current set of capabilities returned from an appliance I think we should return the following:

    Response Body:{"http_enable":"0"}
    or perhaps

    Response Body:{}

    2.4 Current Apppliance Login Implementation

    2.4.1 User log in workflow on appliance

    • OSD->on_logonButton_clicked: username and password
    • OSD-->on_logonButton_clicked: a) if device is in managed state, copies CloudDataA.xml to secondary file CloudDataB.xml
    • OSD-->on_logonButton_clicked: b) check the checksum of appliance and boxilla xml using ssh
    • OSD-->on_logonButton_clicked: c) get new clouddata xml file from Boxilla if checksum not matched using ssh
    • on_logonButton_clicked->authenticateviaxml: credentials validation
    • authenticateviaxml-->check login:Get the username, password, privilege and AD from xml based on keys
    • authenticateviaxml-->check login:Send manual login success alert and authentication stats, if device is safemode,and username,password is matched in xml file along with privilege is Administrator
    • authenticateviaxml-->check login:Send manual login failure alert and authentication stats,if device is safemode,and either of username and password is not matched in xml or privilege is not Administrator
    • authenticateviaxml-->check login:Send manual login success alert and authentication stats,if device is not safemode and username,password are matched in xml along with ad user is false
    • authenticateviaxml-->check login:Send manual login failure alert and authentication stats,if device is not safemode,and either of username and password is not matched in xml along with ad user is false
    • authenticateviaxml-->check login:check authenticationViaAD,if device is not safemode and username is matched in xml along with ad user is true
    • on_logonButton_clicked->authenticationViaAD:ad user validation
    • on_logonButton_clicked-->authenticationViaAD:send ad login request to boxilla
    • on_logonButton_clicked-->authenticationViaAD:get ad user auth_status request until auth status is OK from boxila if ad login request is OK
    • on_logonButton_clicked-->authenticationViaAD:copies CloudDataA.xml to secondary file CloudDataB.xml once ad auth status is OK
    • on_logonButton_clicked-->authenticationViaAD:check the checksum of appliance and boxilla xml using ssh
    • on_logonButton_clicked-->authenticationViaAD:get new clouddata xml file from Boxill if checksum not matched using ssh
    • on_logonButton_clicked-->authenticationViaAD:send ad auth complete to boxilla once xml validation done
    • on_logonButton_clicked-->authenticationViaAD:Send manual login success alert and authentication stats after ad authentication sucess
    • on_logonButton_clicked-->authenticationViaAD:logs the error if ad login request or ad userauth_status or ad auth complete or xml validation failed

    2.5 Proposal for login process on the SDKVM

    2.5.1 Static User Scenario

    1. OSD->flutter:User enters username and password on the OSD
    2. OSD-->on_logonButton_clicked:
      • (a) if device is managed state the existing users.json file is copied to users_backup.json. The OSD presents an indication that it is checking credentials. Once the file has been downloaded the webservice logic will send an MQTT message to the OSD indicating that the users.json is now available for validation of the user's credentials.
      • (b) the SDKVM then checks the checksum of this file (users.json)against the checksum of the Boxilla version of the file (a new Boxilla endpoint will be provided for this)
      • (c) If the checksums differ then the SDKVM will download the current Boxilla version and overwrite the local users.json version.
    3. on_logonButton_clicked->authenticateviajson: After the file has been downloaded the credentials are verified.
    4. authenticatevijson-->check login: If sucessful then the OSD sends a manual login success alert using the webservice CLI
    5. authenticateviajson-->check login: If unsucessful then the OSD will send a manual login failure alert and authentication stats using the webservice CLI
