    1.4 RESTAPI INDIVIDUAL INACTIVITY TIMERS REQUIREMENTS

    • Boxilla will default to have the Receiver timers set to ‘use global timer’;
    • Boxilla will default to having the global timers set to disabled
    • Boxilla will send the configuration of both timers to the specific receiver in one message;
    • Boxilla will query the receiver to ensure that it supports individual Receiver timers;
    • Boxilla will query the receiver to get the timer configuration.

    1.6 SHOULD WE MOVE GLOBAL TIMERS TO SQLITE IN THE FUTURE?

    This is the consideration we should take into account when designing this feature. Accessing one values from xml file and another from SQLite database is making codes long, messy and difficult to understand. Also, in general, xml file is putting limitations on our system (xml file size is affecting system performance) and therefore it is good step to migrate from xml file usage to SQLite database in the future. But this big movement should be done for all data stored in xml file in one go.

    For ‘Local’ timers implementation I suggest avoid moving ‘Global’ timers configurations from xml file to SQLite database. The reason for that is maintenance (code changes) of other functionality not directly connected to timers configuration (it will make codes even longer, messier and difficult to understand). However, detailed information presented below in the case such decision is made.

    Decision from 01/02/2023: Do not move “global timers” from XML – that would change things from Boxilla viewpoint, so not part of this plan.

    1.6.1 Moving Global inactivity times on managed domain to SQLite database in the future

    Considering the case if Glogal inactivity times on managed system will be stored in SQLite in the future the flowing scheme can be implemented: Create additional table entries for Global Inactivity timers: GlobalOSDInactivity and GlobalConnectionInactivity. Example of additional entries shown below in red colour:

    CREATE TABLE Timers (TimeoutConfigurations TEXT, LocalOSDInactivity INT, LocalConnectionInactivity INT, GlobalOSDInactivity INT, GlobalConnectionInactivity INT);

INSERT INTO Timers VALUES(Global,0,0,0,0);

    New values can be accessed on OSD only for read with one exclusion (see description below of initial migration process) on managed domain and read/write in unmanaged domain;

    New values can be accessed on Boxilla for read/write for managed receivers;

    OSD will first read timers configurations and if it is ‘global’ will try to read GlobalOSDInactivity and GlobalConnectionInactivity from SQLite database. In the case table entries for Global Inactivity timers not found in SQLite database OSD will read these values from xml file.

    Boxilla will need to take responsibility to update Global Inactivity timers in SQLite database and xml files as well for backward compatibility for managed receivers.

    Note

    Values modified in SQLite database will be applied immediately after modification, where values modified in xml file will require user re-login.

    Export/Import: need to implement inactivity timers comparison within SQLite database and xml file before exporting and after importing.

    With new implementation of enhanced OSD it is possible to notify OSD that xml database update is required

    1.6.2 Moving inactivity times on unmanaged domain to SQLite database in the future

    Considering the case if Glogal inactivity times on unmanaged system will be stored in SQLite in the future the flowing scheme can be implemented:

    Create additional table entries for Global Inactivity timers: GlobalOSDInactivity and GlobalConnectionInactivity. Example of additional entries shown below in red colour:

    CREATE TABLE Timers (TimeoutConfigurations TEXT, LocalOSDInactivity INT, LocalConnectionInactivity INT, GlobalOSDInactivity INT, GlobalConnectionInactivity INT);

INSERT INTO Timers VALUES(Global,0,0,0,0);

    Unmanaged OSD: only Global Timers are displayed and accessed for read/write (Local timers settings are hidden)

    OSD in unmanaged system: timers configurations will be always set as ‘global’. OSD will try to access GlobalOSDInactivity and GlobalConnectionInactivity in SQLite database. In the case table entries for Global Inactivity timers not found in SQLite database OSD will read these values from xml file. Should user modify inactivity timers settings write will be applied to SQLite database and to xml file (xml file write must be performed to maintain export/import functionality) or within export copy current inactivity timers settings from SQLite database to xml file and within import copy inactivity timers settings from xml file to SQLite database.

    1.6.3 Initial migration process

    The functionality of initial setup of Global Inactivity timers might be considered in following way:

    • Add new entries with default values (GlobalOSDInactivity default = 0, GlobalConnectionInactivity default = 0) to SQLite database using patch script (version also need to be updated);

    • Within OSD load (after xml file pull from Boxilla for managed domain) implement functionality to compare Global Inactivity timers in xml file and SQLite database. If values are different set Global Inactivity timers values in SQLite database to values stored in xml file.
