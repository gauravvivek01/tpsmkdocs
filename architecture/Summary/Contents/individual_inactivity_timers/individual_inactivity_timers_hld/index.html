    1.2 Terms and Definitions used in Blackbox

    Notes to the Authors – (Delete this explanatory text from final approved version)

    This section will include Black Box terms and Definitions; more general technology definitions need not be included here. They can, if required for a specific technology impact study, be added but one should exercise common sense and only define the technology terms that are relevant to the feature under consideration. The table of terms included in this section of the template are for reference only and should be replaced as appropriate in each specific instance.

    Term Definition
    CLU Image A optimized encrypted file format used for 2K appliance software images.
    Decoder See Receiver
    Encoder See Transmitter
    FIT Image A optimized encrypted file format used for 4K appliance software images.
    Municast The use of multiple individual TCP streams to share a video from a single transmitter to multiple receivers
    Multicast The use of an individual Multicast UDP stream to share a video from a single transmitter to multiple receivers. Blackbox uses reliable multicast so that any dropped frames are resent and no data loss occurs.
    Pixel Perfect Mode The original scheme used by 4K products where the entire frame of the source video was captured continuously and sent with compression to the decoder.
    RFX RemoteFX Codec Extension developed by Microsoft which uses a lossy image codec to encode screen images with efficient and effective compression. This is the lossy scheme used in Blackbox products.
    Reciever A Blackbox appliance that receives and displays a video stream. A receiver is also referred to as a client or decoder.
    RPU Real Time Processing Unit. The Zynq Ultrascale+ MPSoC features two A5 RPUs. In our 4K products one of these is used for Display Port functionality using IP provided by Xilinx. The second on the encoder side acts a video processing feed between FPGA and TOE
    Shared Mode A generic term used in Blackbox to describe schemes that allow more than one receiver view the video source from a transmitter. See Municast and Multicast.
    Tile Differencing Mode A lossless scheme used to reduce bandwidth; only tiles (portions of the screen) that change are sent from encoder to decoder.
    TOE TCP Offload Engine: A hardware based TCP stack used to accelerate the transmission of data. This is essential in high bandwidth, low latency transmission of video and audio.
    Technical Debt Technical Debt is a concept in software development that reflects the implied cost of additional rework caused by choosing an easy (limited) solution now instead of using a better approach that would take longer.
    Transmitter A Blackbox appliance that captures and transmits a video from a source. A transmitter is also referred to as a server or encoder.
    Video Source A source of input for a Transmitter, normally the DVI or Display Port output of a PC.

    This feature provide the ability to configure the OSD inactivity and connection inactivity timers applicable to individual Receiver on managed domain via the OSD and later from Boxilla. To provide OSD inactivity and connection inactivity timers applicable to individual Receiver on managed domain new “Local” timers configuration introduced. The user will be able to select which timer configurations to apply on managed Receiver – Global or Local. The feature will not affect:

    • Existing (Global – configurable only from Boxilla) OSD inactivity and Connection inactivity timers on managed Receivers;

    • Timer settings on unmanaged Receivers.

    This document describe only design changes applicable on Receiver

    • Design changes for Boxilla are out of scope of this document;

    • The design changes made on assumption that OSD inactivity and Connection inactivity timer configuration changes applicable together.

    1.3.1 CURRENT INACTIVITY TIMERS OVERVIEW

    Our system has 2 inactivity timers:

    • OSD Inactivity Timer;

    • Connection Inactivity Timer.

    Currently Inactivity timer settings are stored in XML database file.

    1.3.1.1 OSD INACTIVITY TIMER DESCRIPTION

    The purpose of OSD Inactivity Timer in the case when configured to be ON (check box is ticked) is to monitor user activity within OSD application. If the system does not capture user activity within the specified amount of time, the system proceeds to log out user automatically. User activity within the system is defined as mouse and keyboard event detection. OSD Inactivity Timer valid settings are 2 to 60 minutes. In the case if OSD Inactivity Timer configured to be OFF (check box is unticked), user will never logout automatically no matter the period of inactivity. OSD Inactivity Timer value read from XML database file and applied in following cases:

    • User open ‘Timers’ page;

    • user login success;

    • user change timer settings;

    • timer is activated and user activity is detected;

    • user return to OSD after disconnection (enhanced osd only).

    1.3.1.2 CONNECTION INACTIVITY TIMER DESCRIPTION

    The purpose of Connection Inactivity Timer in the case when configured to be ON (check box is ticked) is to monitor user activity within RDP client application. If the system does not capture user activity within the specified amount of time, the system proceeds to RDP client termination and user returned to OSD automatically. User activity within the system is defined as mouse and keyboard event detection. Connection Inactivity Timer valid settings are 2 to 60 minutes. In the case if Connection Inactivity Timer configured to be OFF (check box is unticked), user not returned to OSD automatically no matter the period of inactivity. Connection Inactivity Timer value read from XML database file and applied in following cases:

    • User open ‘Timers’ page;

    • Before connection initialization, Connection Inactivity Timer value passed as an argument to relevant client on connection initialization. The client processing of Connection Inactivity Timer value is out of scope of this document.

    1.3.1.3 CURRENT SYSTEM DESIGN

    In managed system current design only Boxilla can configure Inactivity timers. Timer configurations implemented on Global level. Timer reconfigurations on individual receiver is not possible. Boxilla can configure:

    • Inactivity timers with same timeouts for all receivers within domain;

    • Inactivity timers disabled for all receivers within domain;

    • OSD Inactivity timer disabled and Connection Inactivity timer enabled for 2 – 60min for all receivers within domain;

    • Connection Inactivity timer disabled and OSD Inactivity timer enabled for 2 – 60min for all receivers within domain; OSD on managed Receivers restricting Inactivity timer settings.     1.4 Design Detail

    1.5 INDIVIDUAL TIMER’S DESIGN:

    To configure the OSD inactivity and Connection inactivity timers that are applicable to individual Receiver on managed domain the new concept of Local timers introduction is needed.

    The information about Timer settings and Local timer values should be stored on individual Receivers and be accessible from OSD and Boxilla (as per requirements of this feature). In our current system we have SQLite database on each individual Receiver which is storing particular Receiver information and is not updated globally for domain. This is where new configurations and settings will be stored. New timer settings/values accessed by read/write requests from OSD (and Boxilla).

    • Diagnostics process: No changes needed. SQLite database extracted within Diagnostics process.

    1.5.2.1 SQLITE DATABASE CHANGES

    Current SQLite database example is in the Appendix.

    Timer settings and Local timer values will be stored in SQLite database. The changes include 1 new record – “Timers”. “Timers” have 3 entries “TimeoutConfiguration” in Text format “LocalOSDInactivity” and “LocalConnectionInactivity” for value storage in int format (see table below).

    1.5.2.1.1 New values addition to SQLite database:

    Table 1 SQLite Timers table and values description

    New table name New value name New value format Valid options Value description
    Timers TimeoutConfiguration Text “global” or “local” “Timeout configuration” value is “global” the inactivity timers values to be read and applied from XML database file; “Timeout configuration” value is “local” the inactivity timers values to be read and applied from SQLite database;
    Timers LocalOSDInactivity int 0, 2-60 “Local OSD inactivity timer” value is 0 - OSD inactivity timer is disabled; “Local OSD inactivity timer” value is between 2 and 60 - OSD inactivity timer is enabled for specified number of minutes;
    Timers LocalConnectionInactivity int 0, 2-60 “Local Connection inactivity timer” value is 0 - OSD inactivity timer is disabled; “Local Connection inactivity timer” value is between 2 and 60 - OSD inactivity timer enabled for specified number of minutes;

    Default value settings

    Timeout configuration - “global”;

    Local OSD inactivity timer – 0;

    Local Connection inactivity timer – 0;

    The new default entries will be following:

    CREATE TABLE Timers (TimeoutConfigurations TEXT, LocalOSDInactivity INT, LocalConnectionInactivity INT);

INSERT INTO Timers VALUES('Global’0,0);
    1.5.2.1.2 SQLite database versioning handling

    To add new tables and entries to SQLite database the patching process is used. SQLite database patches are stored in one designated place within our system and applied within system boot up process sequentially. SQLite database version checked on bootup and only patches with a higher version are applied. New patch rx_database_patch_1_8.sql addition required to the system within implementation. This patch will do following:

    • SQLite database version will be updated to 1.8;
    • Add Timers table (as described in section above);
    • Set default values for Timers Table.
    1.5.2.1.3 SQLite database restore to factory default

    Restore process: No changes needed. SQLite database removed on restore. On boot-up after restore default SQLite database created and patched.

    1.5.2.2.1 UI Timers Page Default Settings on Managed domain

    OSD UI Timers page have default settings from initial timers development. The existing default values on managed domain are following:

    • 'OSD Inactivity Timer' chechbox = unchecked, Disabled
    • 'OSD Inactivity Timer' spinbox = 15, Disabled
    • 'Connection Inactivity Timer' chechbox = unchecked, Disabled
    • 'Connection Inactivity Timer' spinbox = 15, Disabled
    • 'Apply' button = Disabled

    Note

    Inactivity timer values of spinboxes set to 15, Disabled does not correspond to value stored in database (database holds value ‘0’). This value based on a user questionnaire from past, when timers were initially introduced into our design. Value ‘15’ the most often set choice by users in case when the timer is active.

    To provide user presentation of new timer settings OSD UI Timers page default settings require modification. New default values are following:

    (New) 'Timers settings' label = enabled (New) Radio button 'Global' = checked (New) Radio button 'Local' = unchecked (Changed) 'Apply' button = Enabled

    “Timers” page is initially set to default values on cloudgui load within user login process and has dependency on user privilege. Only user with ‘Administrator’ privilege can reconfigure timers. User with ‘Power user’ privilege can only view Timers settings. For ordinary user ‘Timers’ page is not accessible. The actual timer settings read and set on UI form when user request to open ‘Timers’ page.

    1.5.2.2.2 Timer configuration access within Cloudgui:

    Additional timers read logic required when user change selection of Global/Local radio button to reflect currently saved OSD and Connection inactivity timer values from corresponding database.

    OSD inactivity timer values read require modifications to provide value corresponding to newly introduced settings. This include additional logic for Global/Local timer detection and corresponding value read. The following cases affected by this change:

    • cloudgui load;
    • User open ‘Timers’ page;
    • user login success;
    • user change timers settings;
    • timer is activated and user activity is detected;
    • user returned to OSD after connection (enhanced osd only).

    Connection inactivity timer value read require modifications to provide value corresponding to newly introduced settings. This include additional logic for Global/Local timer detection and corresponding value read. The following cases affected by this change:

    • User open ‘Timers’ page;
    • Just before connection initialized. Connection Inactivity Timer value is passed as an argument to relevant client on connection initialization.

    1.5.4 RESTAPI CHANGES

    Decision from 16/02/2023: Agreed with Boxilla team.

    To provide functionality for local Inactivity timers values read/write on requests from Boxilla RestAPI require following additions: RestApi on RX add following GET calls:

    • GET_Timers_Config (add new)

      • Parameter: timers_config, osd_inactivity, connection_inactivity

      • Format: char array, int, int

    RestApi on RX modify following GET calls:

    • GET_System_Capabilities (modification of existing call)

      • Parameters: timers_configuration, osd_inactivity, connection_inactivity

      • Formats; "global, local", "0,2-60", "0,2-60".

    • GET_Version_info (modification of existing call)

      • Parameter: api_version;

      • Format: char array

        • Current version according document 1.0.10 to be changed to 1.0.11

    RestApi on RX add following PUT calls:

    • PUT_Timers_Config

      • Parameter: timers_config, osd_inactivity, connection_inactivity

      • Format: char array, int, int

        • Note: if timers_config ==”global” only timers_config will be modified in SQLite database

    Full changes list will be added to the Appliance Blackbox KVM REST API ENG-0005-020 document (currently may be found in Appendix).
